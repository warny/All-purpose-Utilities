# Utils.Mathematics — Fourth audit pass (2026-07-11)

Follow-up review focused on vector operators, barycenters, affine-subspace contracts, Fourier metadata helpers, symbolic convenience extensions, and validation consistency. This file complements the previous audit files; no production code is changed.

## Critical and high-priority findings

### 43. `Vector<T>.operator ==` considers every non-null vector equal to `null`

The implementation is:

```csharp
(vector1 is not null && vector1.Equals(vector2)) || vector2 is null
```

When `vector1` is non-null and `vector2` is null, the first term is false but the second term is true. Therefore `someVector == null` returns true. The `!=` operator delegates to this result and is wrong as well.

**Risk:** null guards, conditional logic, assertions, collections, and any consumer using vector equality operators can behave catastrophically. This is distinct from `Equals`, which correctly rejects null.

**Fix:** use the conventional pattern:

```csharp
ReferenceEquals(vector1, vector2)
    || (vector1 is not null && vector1.Equals(vector2))
```

Add exhaustive null/operator truth-table tests.

**Priority: P0 language/operator correctness.**

**Already fixed.** `Vector<T>.operator ==` (`Vector.Operators.cs`) already uses the conventional
`ReferenceEquals(vector1, vector2) || (vector1 is not null && vector1.Equals(vector2))` pattern this
finding recommends — a previous audit pass (`fix numerical-correctness and quality audit findings #442`)
corrected it before this pass4 review ran, and this file was not updated to reflect that. The full
truth-table coverage this finding asks for also already exists: `VectorTests.EqualityOperator_BothNull_ReturnsTrue`,
`EqualityOperator_LeftNullRightNonNull_ReturnsFalse`, `EqualityOperator_LeftNonNullRightNull_ReturnsFalse`
(the exact regression this finding describes), `EqualityOperator_SameReference_ReturnsTrue`,
`EqualityOperator_EqualValues_ReturnsTrue`, `EqualityOperator_UnequalValues_ReturnsFalse`. No production
or test code changed for this item; verified by re-reading `Vector.Operators.cs` and running the existing
suite.

### 44. Weighted barycenter divides by zero when total weight is zero

`ComputeBarycenter` accumulates weighted coordinates and then divides every component by `totalWeight` without checking it. Opposite weights, all-zero weights, cancellation through rounding, or non-finite weights can therefore produce infinities/NaNs or a type-specific failure.

**Fix:** validate finite weights according to policy and reject a zero/near-zero total weight before division. Consider returning the unnormalized weighted sum separately when zero-total affine combinations are useful.

**Priority: P1 mathematical correctness.**

### 45. Barycenter selector/input validation is incomplete

The generic overload dereferences `weightedPoints`, `getWeight`, `getVector`, and selected vectors without explicit null checks. The first selected point determines the dimension before any null/dimension validation. Later dimension mismatch throws `InvalidOperationException`, while comparable public vector APIs use argument exceptions.

**Fix:** validate delegates, enumerable, elements, selected vectors, dimensions, and weights before or during one clearly specified enumeration. Use consistent argument exception types and parameter names.

**Priority: P1 API robustness.**

### 46. Non-generic symbolic convenience methods silently force `double`

`Derivate()`, `Derivate(string)`, `Gradient()`, and `Integrate()` route to their generic variants with `T = double`, regardless of the parameter and return types of the supplied lambda.

A lambda over `float`, `decimal`, or another `IFloatingPoint<T>` can consequently receive double constants/method assumptions and fail expression construction or compilation. The API surface does not signal that these overloads are double-only.

**Fix:** either restrict and validate the non-generic overloads to lambdas whose mathematical scalar type is exactly `double`, or infer the scalar type and dispatch to a closed generic transformer through a validated factory. Keep explicit generic overloads for callers who choose the scalar domain.

**Priority: P1 symbolic API correctness.**

### 47. Symbolic convenience methods still select variables by nullable names

The convenience methods pass `ParameterExpression.Name` into the transformer. Parameter names are optional in expression trees, and different parameters may share one name. `Gradient()` suppresses nullability with `!` rather than validating identity or ambiguity.

**Fix:** add overloads taking the exact `ParameterExpression` and make them the internal path. Name-based APIs should resolve exactly one parameter or fail with a contextual diagnostic.

**Priority: P1 symbolic correctness.**

### 48. `AffineSubspace.Contains` accepts invalid tolerances

`Contains(point, tolerance)` directly compares distance to the supplied value. A negative tolerance makes even an exact member return false; NaN produces false; positive infinity makes every finite-distance point a member.

**Fix:** require a finite non-negative tolerance, or explicitly document infinity/NaN behavior. Prefer a shared tolerance/options type rather than raw unchecked scalars.

**Priority: P1 API correctness.**

### 49. Affine-subspace factories and operations lack null-member validation

`FromSpan`, `FromNormals`, `Project`, `IntersectWith`, and related paths assume non-null anchors, arrays, direction/normal vectors, lines, and subspaces. Null values fail through incidental dereferences, often before the documented dimension validation.

**Fix:** add explicit public-boundary guards and reject null members with precise parameter/index diagnostics.

**Priority: P1 API robustness.**

### 50. Affine-subspace numerical claims exceed the implemented algorithm

The class remarks state that the Gram–Schmidt representation makes “all geometric operations numerically stable.” The implementation uses a fixed absolute epsilon and classical/modified projection-style operations without scale-aware tolerance, re-orthogonalization, residual reporting, or conditioning diagnostics.

**Risk:** callers may rely on a stronger stability guarantee than the implementation provides, especially with nearly dependent directions/normals or large coordinate scales.

**Fix:** weaken the documentation and introduce explicit numerical options. Use stable QR/SVD-style basis construction where robust rank decisions are required.

**Priority: P1 contract accuracy.**

## Medium-priority findings

### 51. Affine-subspace equality is approximate but object hashing is intentionally coarse and undocumented

`Equals` uses geometric tests under a fixed epsilon, while `GetHashCode` includes only dimension and ambient dimension. This preserves the minimum equal-implies-same-hash requirement, but causes every subspace of the same dimensions to collide.

**Risk:** severe degradation in hashed collections containing many subspaces and ambiguity about whether object equality is exact representation equality or tolerance-dependent geometric equivalence.

**Fix:** keep object equality exact/representation-based and provide a geometric comparer, or document the collision-heavy design and expose a canonical tolerance-aware hashing strategy where feasible.

**Priority: P2 collection performance and API semantics.**

### 52. `AffineSubspace<T>.IFormattable` ignores formatting inputs

The format and provider parameters are documented as unused. This repeats the misleading `IFormattable` contract already present on `Line<T>` and matrix formatting inconsistencies.

**Fix:** implement scalar/vector format propagation or remove `IFormattable` until meaningful formatting is supported.

**Priority: P2 API quality.**

### 53. Fourier metadata extensions do not validate null, empty, or non-finite inputs

All methods dereference `transform` without null guards. `GetFrequencies` accepts `NaN` and positive infinity because only `sampleRate <= 0` is checked. For an empty transform, it computes `sampleRate / 0`; the resulting infinity is unused because the output is empty, silently accepting an invalid transform length.

**Fix:** validate non-null transform, supported non-zero length, and finite positive sample rate. Align transform-length validation with FFT requirements or document that metadata helpers support arbitrary lengths.

**Priority: P2 API robustness.**

### 54. One-sided frequency-bin semantics are incomplete for real signals

`GetFrequencies` returns exactly `N/2` bins. For an even-length real-signal FFT, the conventional non-negative spectrum contains `N/2 + 1` bins including the Nyquist bin. The method omits it while `GetAmplitudes` and `GetPhases` return all `N` bins.

**Risk:** consumers zip arrays with incompatible spectral conventions or unknowingly discard the Nyquist component.

**Fix:** expose explicit APIs such as `GetAllBinFrequencies`, `GetOneSidedFrequencies(includeNyquist: true)`, and matching one-sided amplitudes/phases. Define odd-length behavior.

**Priority: P2 signal-processing contract.**

### 55. “Amplitude” helpers return raw magnitudes without FFT normalization policy

`GetAmplitudes` returns `Complex.Magnitude` directly. Depending on forward-transform normalization, signal length, one-sided conversion, and window coherent gain, these are raw spectral magnitudes rather than physical sinusoidal amplitudes.

**Fix:** rename to `GetMagnitudes` or add explicit normalization modes, one-sided doubling rules, and optional window-gain correction.

**Priority: P2 terminology and correctness.**

### 56. Scalar vector/matrix division has no zero/non-finite policy

Vector and matrix `/ scalar` operators perform direct division without checking zero, near-zero, NaN, or infinity. This is sometimes desirable at the primitive operator level, but higher-level methods inconsistently assume such operations are safe.

**Fix:** keep raw operators only with documented IEEE behavior and ensure higher-level algorithms validate denominators. Alternatively provide checked division helpers.

**Priority: P2 consistency.**

## Additional duplicated intent to reduce

- Equality/null semantics should use shared conventional operator templates across `Vector`, `Matrix`, `Polynomial`, `Line`, and `AffineSubspace`.
- Barycenter, normalization, projection, line distance, homogeneous conversion, and matrix division all need one denominator-validation policy.
- Symbolic extension methods should share one parameter-identity and scalar-type inference/resolution path.
- Geometric object equality should separate exact representation equality from tolerance-aware equivalence through comparers.
- Fourier frequency, magnitude, phase, normalization, and one-sided selection should be represented by one coherent spectrum abstraction.
- Public numerical tolerances should be validated through one common options/value type.

## Required tests

- Full vector equality truth table: null/null, value/null, null/value, same reference, equal values, unequal values; verify both `==` and `!=`.
- Weighted barycenters with all-zero, cancelling, near-zero, negative, NaN, and infinite weights.
- Null delegates, null enumerable, null selected vector, and mismatched dimensions in every barycenter overload.
- Non-generic derivation/integration over `double`, `float`, and `decimal` lambdas, with explicit rejection or inferred dispatch.
- Unnamed and duplicate-named expression parameters through derivative/gradient helpers.
- Affine `Contains` with negative, zero, finite positive, NaN, and infinite tolerances.
- Null anchors/directions/normals/points/lines/subspaces at all public boundaries.
- Nearly dependent affine bases over multiple scales and scalar types; measure orthogonality and residuals.
- Hash-set performance/correctness for many affine subspaces of equal dimensions.
- Fourier helpers with null/empty/odd/even transforms and NaN/infinite sample rates.
- Nyquist-bin inclusion and matched one-sided frequency/magnitude/phase array lengths.
- Raw magnitude versus normalized sinusoidal amplitude under known FFT/window inputs.

## Deployment warning

Until item 43 is fixed, never use `Vector<T> == null` or `!= null`; the current operator reports a non-null vector as equal to null. Until items 44–50 are addressed, weighted barycenters can divide by zero, symbolic convenience methods may use the wrong scalar type/parameter, and affine operations accept invalid tolerances and overstate their numerical stability.
