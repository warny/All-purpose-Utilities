# Utils.Mathematics — Sixth audit pass (2026-07-11)

Follow-up review focused on README/API drift, angle computation, decomposition contracts, matrix metadata, and remaining documentation/testability issues. This file complements the previous audit files; no production code is changed.

## High-priority findings

### 70. `VectorExtensions.AngleWith` can return a wrong angle after norm-product overflow

The method computes `self.Norm * other.Norm` before division. Two finite norms can overflow even when the cosine ratio itself is representable. The resulting infinite denominator drives the cosine toward zero and can return approximately π/2 for vectors whose actual angle is very different.

**Fix:** normalize through scale-safe norms/dot products, or compute the cosine using rescaled vectors. Validate finite intermediate values and share the stable norm policy recommended in earlier passes.

**Priority: P1 numerical correctness.**

### 71. `AngleWith` has only exact zero-vector detection

A near-zero vector passes `normProduct == 0` and may produce a meaningless angle dominated by rounding. Null operands also fail through incidental dereferences.

**Fix:** validate nulls explicitly and use the shared scale-aware zero/tolerance policy. Consider a `TryAngleWith` result for degenerate vectors.

**Priority: P1 API and numerical correctness.**

### 72. LU and inverse APIs return matrices with unreliable structural metadata

`DiagonalizeLU` marks both returned matrices as triangular but explicitly marks them non-diagonal and non-identity even when the actual result has those properties. `Invert` similarly hard-codes every structural flag to false.

Because non-null cached flags suppress recomputation, these are semantic results, not mere missed optimizations.

**Fix:** provide only mathematically guaranteed metadata and leave all other cached flags unknown (`null`), or centralize verified metadata construction.

**Priority: P1 object-state correctness.**

### 73. The README publishes multiple examples that do not compile against the current API

Examples refer to:

- namespace `Utils.Mathematics.Fourrier` instead of `Fourier`;
- `FastFourrierTransform` instead of static `FastFourierTransform`;
- constructing the FFT class with `new`;
- a nonexistent `Transform(buffer, start, end)` overload;
- calling `a.ComputeBarycenter(...)` as an instance extension;
- a nonexistent `IsDiagonalized` property.

**Risk:** the package README is shipped with the NuGet package and presents unusable onboarding code.

**Fix:** compile all README examples in CI from extracted snippets or sample projects, and derive API names from one maintained source.

**Priority: P1 public documentation correctness.**

### 74. README transformation examples advertise behavior known to be incorrect

The README demonstrates translation and composition with column-vector multiplication, but the current translation factory places coefficients in the wrong row. It also documents a shear example based on the broken `Skew` parameter mapping.

**Risk:** consumers receive plausible-looking but mathematically incorrect results from the primary usage guide.

**Fix:** correct production behavior first, then add executable examples asserting transformed coordinates rather than only constructing matrices.

**Priority: P1 deployment/documentation safety.**

### 75. README LU identity assertion is false for pivoted decompositions and the current implementation

The guide states `L * U == m`. With partial pivoting, the proper contract normally includes a permutation (`P * A = L * U`), and the current implementation does not construct a valid `L` anyway.

**Fix:** replace the tuple API with a decomposition result exposing `P`, `L`, `U`, pivot indices, rank/status, and residual validation. Update examples only after the decomposition is corrected.

**Priority: P1 mathematical contract.**

## Medium-priority findings

### 76. Angle computation clamps non-finite cosine values without an explicit policy

`T.Clamp` protects ordinary drift beyond `[-1,1]`, but NaN/infinite inputs or intermediate values are not diagnosed. Depending on `T`, NaN may propagate to `Acos`, while infinity may be silently clamped to an endpoint.

**Fix:** validate finiteness before clamping and reserve clamping for small documented floating-point drift.

**Priority: P2 diagnostics.**

### 77. Package metadata advertises SI conversions and number-to-text features not present in this project

The README and package description mention SI conversions and number-to-text conversion utilities, while the project contents are focused on numerical methods, symbolic expressions, Fourier, polynomials, and linear algebra.

**Risk:** package discovery and consumer expectations are misleading, and responsibilities between modules are blurred.

**Fix:** align package description/features with actual public namespaces or add explicit references to separate packages where those capabilities live.

**Priority: P2 packaging quality.**

### 78. README says vector/matrix support “any `IFloatingPoint<T>`” while actual constraints are stronger

`Vector<T>` and `Matrix<T>` additionally require `IRootFunctions<T>`, and some APIs require trigonometric or power interfaces. The simplified claim hides capability boundaries that matter for generic consumers.

**Fix:** document base type constraints and per-operation additional capabilities precisely.

**Priority: P2 generic API documentation.**

### 79. README examples rely on comments instead of executable result assertions

Many mathematically sensitive examples state expected results only in comments. This allowed API drift and incorrect transformation/LU claims to remain undetected.

**Fix:** maintain examples as executable tests with numeric assertions and include their source in documentation generation.

**Priority: P2 test/documentation architecture.**

## Additional duplicated intent to reduce

- Stable vector norm, angle, normalization, projection, and line-distance logic should share one rescaling/degeneracy implementation.
- Matrix result factories should centralize structural metadata rather than passing hand-written booleans.
- README, XML documentation, tests, and package metadata should be generated or validated from executable examples.
- LU, solve, inverse, and determinant should share one decomposition result and pivot contract.

## Required tests

- Angle between large parallel, orthogonal, and opposite vectors whose individual norms are finite but whose product overflows.
- Near-zero, zero, null, NaN, and infinite vectors in `AngleWith`.
- Structural predicates on identity/diagonal/triangular matrices returned by inverse and decomposition APIs.
- Compile every README code block against the packaged public API.
- Execute translation, rotation, scaling, shear, and composition examples and assert coordinates.
- Validate LU examples through `P·A`, `L·U`, and residual norms.
- Check package description/features against exported namespaces.
