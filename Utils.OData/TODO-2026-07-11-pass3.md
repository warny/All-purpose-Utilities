# Utils.OData — Third audit pass (2026-07-11)

Third review of `Utils.OData`, focused on the streaming `IDataReader`, cancellation and disposal semantics, metadata-cache identity, schema resolution, source-generator determinism, generated identifiers, and EDM nullability. This pass does not repeat findings from the first two audit files and does not change production code.

## High-priority findings

### 39. Streaming cancellation is silently converted into successful end-of-data

`StreamBatchesAsync` catches `OperationCanceledException` and completes the channel without an error. `ODataStreamingDataReader.Read` also catches `OperationCanceledException` and returns `false`.

**Risk:** callers cannot distinguish a complete result set from a partially consumed query cancelled by the caller, timeout, reader disposal, or linked-token cancellation. A cancelled import can therefore be committed as if all rows were read.

**Fix:** preserve cancellation as cancellation. Complete the channel with the `OperationCanceledException`, and let `Read` rethrow it unless cancellation was initiated specifically by `Dispose` after the consumer has abandoned the reader. Track disposal cancellation separately from request cancellation.

**Priority: P1 data-completeness correctness.**

### 40. Reader disposal blocks synchronously and suppresses background failures

`ODataStreamingDataReader.Dispose` cancels the linked token and then calls `_backgroundTask.Wait()`. It catches and discards any single-inner `AggregateException`, not only cancellation.

**Risk:** disposal can block a request thread, deadlock under a synchronization-sensitive implementation, and hide HTTP, conversion, or structural failures that occurred in the producer. The comment says only cancellation is expected, but the filter accepts every single exception.

**Fix:** implement `IAsyncDisposable` for asynchronous completion, avoid synchronous waiting where possible, and suppress only cancellation caused by reader disposal. Preserve or expose non-cancellation producer failures through `Read`, `DisposeAsync`, or an explicit completion/error property.

**Priority: P1 lifecycle and error integrity.**

### 41. The reader finalizer invokes managed cancellation but cannot complete managed cleanup

The finalizer calls `Dispose(false)`, which sets `_isClosed` and calls `_cancellationSource.Cancel()`, but deliberately does not wait for the task or dispose the cancellation source.

**Risk:** finalizer-thread callbacks can execute arbitrary registered cancellation code; meanwhile the background task, channel, and cancellation source can remain alive until their own references disappear. The type owns no unmanaged resource that requires a finalizer.

**Fix:** remove the finalizer. Rely on deterministic `Dispose`/`DisposeAsync`, and make ownership explicit in documentation and analyzers/tests.

**Priority: P1 resource lifecycle.**

### 42. `GetBytes` and `GetChars` do not validate offsets, lengths, or destination bounds

Both methods cast `long` field offsets directly to `int`, accept negative offsets and lengths, and delegate destination bounds to incidental array exceptions. Values above `int.MaxValue` wrap during the cast.

**Risk:** incorrect counts, overflow, and inconsistent exceptions relative to `IDataRecord` expectations.

**Fix:** validate field offset, destination offset, length, and destination capacity before copying. Reject values outside the supported `int` range with precise argument exceptions. When `buffer` is null, return the full source length according to the documented reader contract rather than a count derived from an arbitrary requested length.

**Priority: P1 `IDataReader` contract correctness.**

### 43. Duplicate column names silently overwrite ordinal mappings

The ordinal dictionary uses case-insensitive assignment (`_ordinals[column.Name] = column.Ordinal`). Duplicate names, including names differing only by case, silently leave only the last ordinal addressable by name while both columns remain accessible by index.

**Risk:** projections, aliases, expanded properties, or malformed payloads can produce ambiguous schemas whose name lookup returns a different column than expected.

**Fix:** detect duplicate normalized names while constructing the schema. Either reject them with a descriptive schema error or generate deterministic qualified/unique names and expose original names separately.

**Priority: P1 schema integrity.**

### 44. Non-object entries are silently emitted as rows containing only `DBNull`

`WriteBatchAsync` converts every array entry that is not a `JsonObject` into an empty row of the configured width.

**Risk:** scalar values, null entries, nested arrays, or malformed service responses become apparently valid database rows with all-null fields, hiding protocol/schema errors and corrupting downstream imports.

**Fix:** reject unsupported row shapes with page and row-index diagnostics. Support scalar projections explicitly only when the schema represents a single scalar column.

**Priority: P1 data integrity.**

### 45. Property resolution can borrow a same-named property from an unrelated entity type

When the inferred entity type is missing or does not contain a column, `ResolveProperty` scans every entity type in every schema and returns the first property with a case-insensitive matching name.

**Risk:** a common name such as `Id`, `Name`, `Date`, or `Amount` can acquire the EDM type of an unrelated entity. The reader then reports the wrong `FieldType` and applies the wrong converter.

**Fix:** never use a global name-only fallback. Resolve through the entity set/type declared by metadata and `@odata.context`; if that cannot be done, use an explicitly untyped converter or return a schema-resolution error.

**Priority: P1 type correctness.**

### 46. Metadata cache identity ignores the requested metadata source

`GetMetadataFromBaseAsync`, `GetMetadataFromUrlAsync`, and `GetMetadataFromJsonAsync` all share one `_metadataCache`. Once any source succeeds, every later call returns that cached EDMX without evaluating its URL provider.

**Risk:** requesting metadata for a different endpoint or payload on the same `QueryOData` instance silently returns the first service's schema. This can mis-type columns and generate invalid queries.

**Fix:** key the cache by canonical metadata URI, or restrict the instance contract to one immutable service root and reject metadata URLs outside that identity. Add explicit refresh/invalidation support if schemas may change.

**Priority: P1 cache correctness.**

### 47. The cached mutable EDMX object is returned directly to every caller

The same `_metadataCache` reference is exposed on every successful metadata request. The deserialized metadata graph uses mutable arrays/properties.

**Risk:** one caller can mutate the cached schema and affect concurrent or later readers. There is no versioning, defensive copy, or immutability contract.

**Fix:** make metadata model objects immutable, expose a read-only normalized model, or clone before returning. At minimum document that callers must not mutate and keep the mutable graph internal.

**Priority: P1 shared-state correctness.**

### 48. Runtime metadata downloads have no response-size limit

`QueryOData.FetchMetadataAsync` reads and deserializes the response stream without enforcing the 10 MiB limit used by `ODataContext`.

**Risk:** a malicious or misconfigured endpoint can cause excessive memory and XML parsing work. The two metadata-loading paths have inconsistent security policies.

**Fix:** centralize metadata downloading/deserialization behind one bounded implementation with timeout, cancellation, content-type policy, decompression limits, and consistent diagnostics.

**Priority: P1 resource-exhaustion protection.**

### 49. The source generator performs synchronous network I/O during compilation

`ODataEntityGenerator` creates a new `HttpClient`, calls `GetAsync(...).GetAwaiter().GetResult()`, and downloads a remote EDMX as part of source generation. It supplies no timeout, cancellation token, stable cache, or offline policy.

**Risk:** builds become network-dependent, non-hermetic, slow or indefinitely blocked, and can produce different generated source from the same commit. IDE design-time builds may repeatedly contact the endpoint. Credentials/proxy behavior is also uncontrolled.

**Fix:** prohibit network access in the generator. Require EDMX as an `AdditionalFile` or pre-restored build artifact. A separate explicit tool/MSBuild target may download metadata before compilation with caching and locked checksums.

**Priority: P1 build reliability and supply-chain determinism.**

### 50. Generated context declarations lose important source type shape

The generator emits `public partial class <Name>` using only the simple symbol name. It does not preserve accessibility, generic parameters/constraints, record/class shape, containing types, or nested-type hierarchy.

**Risk:** generation fails or produces a different type for internal, generic, nested, or specially declared contexts. Hint names based only on `classSymbol.Name` can also collide across namespaces or containing types.

**Fix:** generate from fully qualified symbol structure, reproduce containing partial declarations and type parameters/constraints, preserve accessibility and declaration kind, and derive collision-resistant hint names from metadata names.

**Priority: P1 generator correctness.**

### 51. EDM and service names are emitted as raw C# identifiers

Entity names and property names are appended directly into class/property declarations. No keyword escaping, Unicode/identifier validation, collision resolution, or normalization is performed.

**Risk:** legal EDM names such as `namespace`, names containing punctuation/spaces, leading digits, duplicate normalized names, or names conflicting with generated members make the consumer compilation fail.

**Fix:** centralize identifier sanitization using Roslyn identifier rules, prefix/escape keywords, preserve the original EDM name through an attribute or mapping table, and diagnose irreconcilable collisions.

**Priority: P1 generator correctness.**

### 52. Generated nullability is inferred from key membership instead of the EDM `Nullable` facet

Every non-key value type is generated nullable and every key value type non-nullable. Reference properties are also made nullable unless they are keys. The metadata property's actual nullability constraint is ignored.

**Risk:** generated contracts disagree with the service schema: required non-key properties appear optional, while nullable key-like/custom scenarios may be forced non-null. Serialization, validation, and consumer assumptions become inaccurate.

**Fix:** parse and honor the EDM `Nullable` facet, using the protocol default when absent. Treat key constraints as an additional validation rule, not as the source of nullability.

**Priority: P1 generated model fidelity.**

## Medium-priority findings

### 53. Case-insensitive JSON property lookup can hide ambiguous payloads

Column materialization resolves property names ignoring case. A JSON object containing both `Name` and `name` can map nondeterministically according to enumeration order, while OData property names are case-sensitive identifiers.

**Fix:** use ordinal matching by default. If a compatibility fallback is necessary, first require no case-insensitive ambiguity and emit a diagnostic when fallback is used.

**Priority: P2 protocol fidelity.**

### 54. Reader schema claims all columns allow `DBNull` regardless of EDM metadata

`GetSchemaTable` hardcodes `AllowDBNull = true` for every field, even when the metadata marks a property non-nullable.

**Fix:** carry EDM nullability into `ColumnDefinition` and expose it through the schema table. Keep transport-level missing-value handling separate from declared schema nullability.

**Priority: P2 schema fidelity.**

### 55. Metadata source failures are cached only by omission and can trigger repeated expensive retries

Successful metadata is cached, but deterministic failures such as an invalid document or permanent 404 are retried on every call under the semaphore.

**Fix:** define a refresh policy and optionally cache failures for a short duration with their canonical URL. Do not cache cancellation or transient failures indefinitely.

**Priority: P2 resilience and load control.**

### 56. Generator diagnostics expose full exception strings

On any metadata-loading exception, the generator reports `ex.ToString()` in `ODATA003`.

**Risk:** diagnostics can leak local filesystem paths, proxy/host details, stack traces, or other environment-specific information into build logs. They are also noisy and unstable for tests.

**Fix:** emit a concise stable message and optionally a separate diagnostic property/detail under an explicit verbose mode.

**Priority: P2 diagnostics and information exposure.**

## Duplications of intent to reduce

- Runtime context loading, runtime query metadata loading, and generator metadata loading each implement different HTTP, size, timeout, and error policies.
- EDM-to-CLR mapping is duplicated between `EdmFieldConverterRegistry` and the source generator and can drift.
- Entity/type/property resolution should be one metadata-model service shared by streaming conversion and code generation.
- Cancellation, producer completion, and reader disposal are currently represented through overlapping channel, task, token, and exception paths without one lifecycle state machine.
- Identifier normalization and original-name mapping should be a single generator utility used for contexts, entities, properties, hint names, and diagnostics.

## Required tests

- Cancel before the first page, between pages, while the channel is full, and during `Read`; verify cancellation is never reported as normal EOF.
- Dispose a reader with a successful, faulted, blocked, and cancelled producer; verify no deadlock and no suppression of unrelated errors.
- Exercise `GetBytes`/`GetChars` with negative, overflowing, null-buffer, zero-length, and insufficient-destination arguments.
- Return duplicate column names and case-only collisions; verify deterministic rejection or qualification.
- Return scalar, null, array, and malformed entries in `value`; verify they cannot become silent all-null rows.
- Use multiple entity types with same-named properties of different EDM types; verify no cross-entity type borrowing.
- Request metadata from two different URLs on one client and verify cache identity.
- Mutate returned metadata in a test and verify callers cannot corrupt shared cached state.
- Serve oversized/chunked/compressed metadata to every runtime loading path.
- Run generator tests for offline remote URLs, cancellation/timeouts, nested/generic/internal contexts, duplicate simple names, keywords, invalid identifiers, and EDM nullability.
- Assert runtime converter mappings and generator property mappings come from the same canonical table.

## Recommended order

| Priority | Action |
|---|---|
| P1 | Preserve cancellation and producer errors through the streaming reader |
| P1 | Correct disposal/finalization and validate binary/text reader offsets |
| P1 | Make schema resolution entity-scoped and reject malformed row shapes |
| P1 | Key and protect the metadata cache; unify bounded metadata loading |
| P1 | Remove network I/O from the source generator |
| P1 | Preserve context type shape, sanitize identifiers, and honor EDM nullability |
| P2 | Tighten schema casing/nullability and define failure-cache policy |

## Deployment warning

Until items 39–52 are addressed, treat streamed reads as potentially incomplete without an observable cancellation signal, do not reuse one `QueryOData` instance across different metadata sources, do not expose cached metadata to mutation, and avoid remote EDMX URLs in source-generator inputs. Generated code is reliable only for simple top-level non-generic contexts and EDM names that already happen to be valid unique C# identifiers.
