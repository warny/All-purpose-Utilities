# Utils.OData — Second audit pass (2026-07-11)

Follow-up review focused on EDM conversion fidelity, LINQ-provider contracts, URI compilation, metadata lifetime, error representation, streaming behavior, and failure diagnostics. This file complements `TODO.md`; no production code is changed.

## High-priority findings

### 21. **[FIXED]** EDM conversion failures are silently converted into unrelated CLR values

`EdmFieldConverterRegistry.ConvertNode` catches every deserialization exception. After several fallbacks, it may return the raw string even when the declared converter type is `int`, `Guid`, `DateTimeOffset`, or another non-string CLR type.

The `EdmFieldConverter` still advertises its original `ClrType`, so an `IDataReader` column can report `Int32` while returning a `string` value.

**Risk:** runtime type inconsistency, invalid casts in consumers, hidden malformed server data, and errors surfacing far away from the conversion point.

**Fix:** return either the declared CLR type, `DBNull`, or a structured conversion error containing entity, property, EDM type, raw value, and target CLR type. Never return an arbitrary fallback type under a typed column contract.

**Priority: P1 data correctness.**

### 22. **[FIXED]** All converter exceptions are swallowed without diagnostics

The registry uses broad `catch` blocks for JSON conversion, Base64 decoding, and `Convert.ChangeType`. No original exception or property context is retained.

**Risk:** malformed payloads, unsupported EDM values, culture issues, overflow, and implementation bugs are indistinguishable from ordinary null values or strings.

**Fix:** catch only expected format/overflow exceptions, preserve the inner exception in a structured OData conversion exception, and make lenient fallback an explicit option rather than the default.

**Priority: P1 observability and correctness.**

### 23. **[FIXED]** `Edm.Date` is mapped to `DateTime` instead of a date-only CLR type

OData `Edm.Date` has no time-of-day or timezone. Mapping it to `DateTime` introduces an artificial midnight and `Kind` ambiguity.

**Fix:** use `DateOnly` on supported target frameworks, or expose a configurable mapping. Keep `DateTimeOffset` for `Edm.DateTimeOffset`.

**Priority: P1 semantic fidelity.**

### 24. **[FIXED]** `Edm.TimeOfDay` and `Edm.Duration` share one `TimeSpan` parser despite different lexical grammars

`Edm.TimeOfDay` is a clock time, while `Edm.Duration` uses the OData/ISO duration representation. Generic `TimeSpan.TryParse` does not reliably represent both wire formats and may accept values outside the EDM domain.

**Fix:** implement dedicated parsers and validation for each EDM primitive, with round-trip tests against the OData lexical forms.

**Priority: P1 protocol correctness.**

### 25. **[FIXED]** Unknown and collection EDM types silently degrade to `string`

Any unsupported type, missing type, complex type, enum type, spatial type, or `Collection(...)` resolves to `DefaultConverter`, whose advertised CLR type is `string`.

**Risk:** arrays/objects can be stringified, schema information is lost, and unsupported metadata appears successfully handled.

**Fix:** distinguish unknown, complex, enum, collection, spatial, and primitive types. Either materialize appropriate JSON/CLR shapes or return an explicit unsupported-type diagnostic.

**Priority: P1 metadata/materialization correctness.**

### 26. **[FIXED (partial)]** LINQ URI compilation does not encode entity names, filters, or expansions

`ODataQueryCompilation.ToUriString()` directly concatenates `EntitySetName`, expansion names, and filter fragments into a URI-like string.

**Risk:** spaces, quotes, `&`, `#`, `+`, Unicode, and reserved characters can corrupt the URI or alter query-option boundaries. This also duplicates URI construction already present in `ODataQueryBuilder`.

**Fix:** keep compilation as a structured model and delegate final URI generation to one encoder-aware builder. Do not encode entire OData expressions blindly; encode query parameter values at the correct URI layer.

**Priority: P1 request correctness and intent duplication.**

### 27. The LINQ provider implements `IQueryProvider.Execute` but cannot execute normal LINQ terminal operators

`Execute<TResult>` only succeeds when `TResult` is `ODataQueryCompilation` or `object`. Standard operations such as `Count()`, `Any()`, `First()`, enumeration, projections, and scalar selection invoke provider execution expecting other result types and receive `NotSupportedException`.

**Risk:** the object presents as a general `IQueryable<T>` even though it is primarily a query-compilation DSL. Users discover unsupported behavior only at runtime.

**Fix:** either implement a real execution pipeline, or expose a dedicated compile-only query type/API instead of a broadly compatible-looking LINQ provider. Clearly enumerate supported LINQ operators.

**Priority: P1 API contract accuracy.**

### 28. **[FIXED]** Non-generic `CreateQuery` resolves element types by taking the first generic argument

`ResolveElementType` assumes any generic expression type's first argument is the sequence element. This is not valid for arbitrary generic types and can produce an unrelated element type.

**Fix:** resolve `IQueryable<T>`/`IEnumerable<T>` interfaces explicitly and reject non-sequence expression types with a contextual exception.

**Priority: P1 LINQ correctness.**

### 29. Metadata download in constructors has no cancellation or dependency-injection path

`ODataContext(string)` performs synchronous network I/O during construction. The caller cannot supply a cancellation token, handler, configured `HttpClient`, authentication policy, retry policy, or test double.

**Risk:** blocked threads, difficult testing, hidden network side effects, authentication inconsistency with `QueryOData`, and inability to cancel application shutdown.

**Fix:** separate construction from asynchronous initialization, or provide an async factory accepting `HttpClient` and `CancellationToken`. Reuse the same transport policy as query execution.

**Priority: P1 lifecycle and operability.**

### 30. Remote metadata redirects are followed without an explicit destination policy

The metadata client uses the default redirect behavior. A trusted metadata URL may redirect to another host or scheme, forwarding implicit ambient behavior and fetching untrusted content.

**Fix:** define whether cross-host and HTTPS-to-HTTP redirects are allowed, validate the final URI, and expose redirect policy through transport configuration.

**Priority: P1 security boundary.**

## Medium-priority findings

### 31. `ReturnValue<T>` mixes HTTP status codes and application error codes without a typed category

`ErrorReturnValue` contains only an integer and a message. Code `1` is used for “No data returned,” while HTTP status codes may also occupy the same field.

**Risk:** callers cannot reliably distinguish transport, protocol, deserialization, validation, cancellation, empty-result, and application errors.

**Fix:** add an error kind/category, optional HTTP status, OData error code, inner exception, request URI, and response body excerpt. Treat empty collections as success.

**Priority: P2 error-model quality.**

### 32. **[FIXED (partial)]** `ErrorReturnValue` allows null or empty messages and arbitrary integer codes

The record has no validation or invariant. Invalid error objects can be created and propagated as if meaningful.

**Fix:** validate message and model codes with enums/value objects where possible.

**Priority: P2 diagnostics.**

### 33. Metadata stream size policy is inconsistent across entry points

Remote streams are capped at 10 MB, while caller-provided streams and local files are copied without the same bound. All paths then copy the complete EDMX into another `MemoryStream` before deserialization.

**Fix:** use one configurable maximum-size policy for remote, file, and supplied streams, and deserialize directly when seekability/lifetime permits.

**Priority: P2 resource management.**

### 34. Synchronous stream APIs are used throughout metadata loading

`CopyTo`, `Read`, and `Write` are synchronous even for network-origin streams. The public construction path therefore blocks threads during both download and copying.

**Fix:** add asynchronous stream loading with cancellation. Keep a clearly named synchronous path only when required.

**Priority: P2 scalability.**

### 35. URI generation is duplicated across two incompatible representations

`ODataQueryBuilder` builds a complete URL from `IQuery`, while `ODataQueryCompilation.ToUriString()` independently serializes a subset of options. They differ in supported features, encoding, validation, and output shape.

**Fix:** introduce one immutable OData request model and one final serializer. LINQ translation should populate that model rather than creating another URI grammar.

**Priority: P2 architecture and duplication.**

### 36. LINQ compilation captures only filters and expansions

The provider shape suggests LINQ query support, but the compilation model has no order, projection, skip, top, count, search, aggregation, or terminal-operation representation.

**Fix:** either narrow the public promise to filter/expand compilation or extend the intermediate model systematically before advertising broader LINQ support.

**Priority: P2 feature-contract clarity.**

### 37. Filter accumulation always joins separate `Where` calls with `and` without preserving an explicit query tree

The translator stores filters as strings and later joins them. This works for simple chained `Where` calls but loses structured operator information and makes later normalization, parameterization, diagnostics, or protocol-version handling difficult.

**Fix:** retain an AST/intermediate expression model until final serialization.

**Priority: P2 maintainability.**

### 38. Converter registry is closed and global

The static dictionary cannot be extended per context/service for custom EDM types, enums, aliases, vendor-specific values, or generated strong types.

**Fix:** support an immutable default registry plus per-context overrides injected into materialization.

**Priority: P2 extensibility.**

## Duplications of intent to reduce

- Final URI serialization must exist in one component shared by manual and LINQ queries.
- EDM primitive formatting/parsing should use one protocol-aware codec for filters, JSON materialization, generated entities, and data readers.
- Error creation should pass through one structured factory preserving transport/protocol/conversion context.
- Metadata transport should reuse the same configurable HTTP infrastructure as data queries.
- Strict versus lenient conversion should be one explicit option, not implicit broad catches.

## Required tests

- For every supported EDM primitive, verify valid boundary values, nullability, malformed values, overflow, and exact CLR return type.
- Verify `IDataReader.GetFieldType(i)` always matches the non-`DBNull` runtime type returned by `GetValue(i)`.
- Cover `Edm.Date`, `TimeOfDay`, `Duration`, enum, collection, complex, binary, and unknown types.
- Ensure malformed numeric/GUID/date/Base64 payloads produce structured errors rather than strings.
- URI tests with spaces, apostrophes, ampersands, plus, hash, Unicode, slashes, and already encoded values.
- Compare manual query-builder and LINQ-compiled URI output for equivalent structured requests.
- Exercise `Count`, `Any`, `First`, projection, ordering, skip/take, enumeration, and unsupported methods; verify explicit supported/unsupported contracts.
- Test non-generic `CreateQuery` with valid and invalid sequence expression types.
- Metadata loading cancellation, timeout, oversized remote/local/supplied streams, redirects, authentication, and custom handlers.
- Distinguish empty success from HTTP, OData protocol, conversion, cancellation, and validation failures.

## Deployment warning

Until items 21–30 are addressed, typed data-reader columns may return values of the wrong CLR type, malformed server values may be silently hidden, LINQ queryables can fail on ordinary terminal operations, URI fragments can be corrupted by reserved characters, and metadata construction performs uncancellable synchronous network I/O.
