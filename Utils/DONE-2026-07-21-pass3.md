# Utils — Third quality and security audit pass (2026-07-19)

Follow-up static audit of the current core `Utils` package. This pass avoids repeating the completed findings in `DONE-2026-07-11.md` and the open items in `TODO-2026-07-19-pass2.md`. It focuses on TOTP state and parameter invariants, transaction failure semantics, external resource trust boundaries, lazy-value dictionary contracts, concurrency, and bounded I/O. No production code is changed.

## High-priority findings

### ✅ 29. `Authenticator` does not validate digits, interval length, key length, or verification range

The constructors accept any `digits` and `intervalLength`. A zero interval causes division by zero in `CurrentMessage`; a negative interval reverses the time counter; negative digits fail incidentally in string allocation; large digit counts overflow the `(int)Math.Pow(10, Digits)` modulus. `VerifyAuthenticator` also accepts a negative range and silently checks no codes.

**Fix:** validate a positive interval, a supported digit range, a non-empty key of an algorithm-appropriate minimum length, and `range >= 0`. Use an integer power-of-ten table or checked arithmetic rather than converting `Math.Pow` to `int`.

**Priority: P1 cryptographic configuration correctness.**

### ✅ 30. The TOTP key and HMAC instance remain externally mutable after construction

The constructor retains the caller-owned key array directly. Both `Key` and `Algorithm` are public properties exposing mutable objects. Callers can alter key bytes, replace `Algorithm.Key`, call `Initialize`, or otherwise change the cryptographic state after construction.

**Risk:** authentication results can change unexpectedly, configuration integrity cannot be guaranteed, and secrets remain exposed through a public mutable array.

**Fix:** copy the key into private storage, do not expose the live key, and expose only immutable algorithm metadata. If key export is required, return a defensive copy through an explicitly named method.

**Priority: P1 secret and state integrity.**

### ✅ 31. `Authenticator` is not safe for concurrent code generation or verification

Every calculation assigns `Algorithm.Key` and calls `ComputeHash` on the same `HMAC` instance. `HMAC` implementations are stateful and are not safe for concurrent use. Parallel calls can race while changing or consuming internal state.

**Fix:** create a fresh keyed HMAC per calculation from an algorithm factory, or protect the complete keyed hash operation with a private lock. Prefer factory-owned instances so the authenticator controls lifetime and mutation.

**Priority: P1 authentication correctness under concurrency.**

### ✅ 32. Cryptographic resource ownership and disposal are undefined

`Authenticator` stores a caller-provided `HMAC`, never disposes it, and does not implement `IDisposable`. If the class were changed to dispose it, that would surprise callers that retain ownership; in the current form, internally created algorithms from the string constructor can leak disposable cryptographic resources.

**Fix:** stop accepting a live `HMAC` as long-lived state, accept an algorithm identifier/factory instead, and create/dispose an instance per operation. If retaining the overload, document ownership explicitly and implement a clear leave-open/owned-resource policy.

**Priority: P1 resource and API ownership.**

### ✅ 33. Algorithm-name construction can fail through incidental cast/null behavior

The string constructor casts the result of `CryptoConfig.CreateFromName` directly to `HMAC`. Unknown names can return null and known non-HMAC names can produce another algorithm type, resulting in an incidental null or cast failure rather than a documented configuration exception.

**Fix:** validate the name, inspect the returned object, dispose unsupported instances, and throw `ArgumentException` or `CryptographicException` with the supplied algorithm name. Prefer an explicit supported algorithm enum/factory.

**Priority: P1 predictable cryptographic API behavior.**

### ✅ 34. The `GoogleAuthenticator` factory's compatibility claim is too strong

The named factory selects HMAC-SHA-256. The common/default TOTP profile used by Google Authenticator provisioning is SHA-1 unless the provisioning metadata explicitly selects another algorithm. A method named `GoogleAuthenticator` therefore risks producing codes that do not match a normally provisioned account.

**Fix:** either use the standard default profile, rename the factory to describe its exact SHA-256 parameters, or accept/provide an `otpauth://` provisioning model that carries algorithm, digits, period, issuer, and account name explicitly.

**Priority: P1 interoperability.**

### ✅ 35. Verification accepts any digit characters only indirectly and does not define counter overflow behavior

`VerifyAuthenticator` checks only string length before comparing. Non-digit strings are processed through repeated HMAC calculations even though they can never match a numeric OTP. Counter arithmetic (`baseMessage + i` and `baseMessage - i - 1`) is unchecked and can wrap for extreme ranges or custom timestamps.

**Fix:** reject non-ASCII digits before hashing, cap the accepted window, and use checked counter arithmetic. Expose an overload accepting an explicit timestamp/counter for deterministic testing and clock-control policies.

**Priority: P1 validation and denial-of-service resistance.**

### ✅ 36. A commit failure triggers rollback of actions that may already have committed

`TransactionExecutor` commits actions sequentially inside the same `try` that covers execution. If a later `Commit` throws, the catch block invokes `Rollback` on every executed action, including actions whose commit already succeeded. The interface does not establish that rollback-after-commit is valid or reversible.

**Risk:** external state can become partially committed while rollback code reports success or performs an invalid compensating action.

**Fix:** define an explicit transaction state model. Either require prepare/commit semantics where all commits are guaranteed not to fail after preparation, or track committed actions separately and use a documented compensation operation rather than ordinary rollback.

**Priority: P1 transaction consistency.**

### ✅ 37. A rollback exception masks the original failure and prevents remaining rollbacks

The rollback loop is not protected per action. The first rollback exception exits the loop, skips earlier actions, and replaces the original execution/commit exception.

**Fix:** attempt every rollback, collect failures, preserve the original exception, and throw a structured aggregate/transaction exception that distinguishes the primary failure from rollback failures.

**Priority: P1 failure recovery.**

### ✅ 38. Transaction execution has no protection against re-enumeration side effects or mutable action sequences

Actions are consumed lazily from the caller's enumerable while side effects are already occurring. Enumeration can block, throw, yield null, or observe concurrent mutation after earlier actions have executed.

**Fix:** materialize and validate the complete action list before starting any action, unless streaming execution is an intentional documented feature. Reject null entries before the first side effect.

**Priority: P1 deterministic transaction setup.**

### ✅ 39. No asynchronous or cancellation-aware transaction contract exists

The executor supports only synchronous methods. Long-running I/O actions cannot compose naturally with async code, and there is no cancellation policy defining whether cancellation before/after an action requires rollback.

**Fix:** add a separate asynchronous contract using `ValueTask`/`Task` and `CancellationToken`, with explicit rules for cancellation during execute, prepare, commit, and rollback. Do not wrap synchronous actions in `Task.Run` as the primary design.

**Priority: P1 scalability and lifecycle semantics.**

### ✅ 40. External resource path containment is platform-inconsistent and does not address links

The containment check always uses `OrdinalIgnoreCase`. On case-sensitive file systems, paths that differ only by case are distinct and the comparison can approve a path outside the intended root. Lexical `GetFullPath` containment also does not prevent an in-root symbolic link from resolving to a file outside the root.

**Fix:** use platform-appropriate path comparison and define whether symbolic links/reparse points are allowed. For untrusted resource files, resolve and verify the final target or disallow external references entirely.

**Priority: P1 file-system security boundary.**

### ✅ 41. A resource file can request construction of arbitrary loaded types

For an unrecognized `ResXFileRef` type, the reader calls `Type.GetType` and later `Activator.CreateInstance(type, byte[])`. A resource file can therefore select any resolvable type with a matching constructor and execute its constructor when the value is accessed.

**Risk:** loading an untrusted or replaceable `.resx` file becomes a code-execution primitive within the application's trust boundary.

**Fix:** remove generic reflective construction or require an explicit caller-provided allowlist/factory mapping. Treat unknown types as unsupported data, not executable object descriptions.

**Priority: P1 code-execution surface.**

### ✅ 42. `ExternalResource` violates its `IReadOnlyDictionary<string, object>` value contract

The indexer unwraps `IResXValue`, but `TryGetValue`, `Values`, and enumeration return the private lazy wrapper objects. The same key therefore has a different observable value depending on which dictionary API is used.

**Fix:** resolve values consistently in every dictionary member, or expose a distinct raw-entry API. `TryGetValue` must return the same logical value as the indexer for the same key.

**Priority: P1 public collection correctness.**

### ✅ 43. External file resources are read without size limits

Lazy text, binary, and custom-object paths call `File.ReadAllText` or `File.ReadAllBytes`. A small resource file can reference an arbitrarily large local file and force complete allocation when accessed.

**Fix:** add configurable per-file and total resource limits, inspect file length before allocation, stream where possible, and fail with a resource-specific exception before materializing excessive content.

**Priority: P1 memory safety.**

### ✅ 44. The shared encoding cache is mutated without synchronization

`GetEncodingFromTypeName` reads and adds to a static `Dictionary`. Concurrent construction of resources containing the same or different unknown encoding names can race, throw, or corrupt expected cache behavior.

**Fix:** use an immutable predefined map plus `ConcurrentDictionary` for validated extensions, or avoid caching dynamically created encodings. Never cache arbitrary reflection-created encoding objects without a trust policy.

**Priority: P1 concurrency correctness.**

## Medium-priority findings

### ✅ 45. Unknown encodings silently fall back to the process-dependent default encoding

Failures during type resolution or instantiation are swallowed and `Encoding.Default` is returned. The same resource can therefore decode differently across operating systems and invalid metadata is hidden.

**Fix:** support a fixed allowlist of encoding names/code pages and throw a contextual resource-format exception for unknown values. If fallback is required, make it explicit in options and prefer UTF-8 rather than the process default.

**Priority: P2 deterministic decoding.**

### ✅ 46. Lazy binary values expose their mutable cached byte array

The binary wrapper returns the same cached `byte[]` instance on every access. A consumer can modify it and thereby change the resource value observed by all later callers.

**Fix:** return an immutable representation (`ReadOnlyMemory<byte>` with controlled backing storage) or a defensive copy. Document the allocation/lifetime contract.

**Priority: P2 resource immutability.**

### ✅ 47. Resource parsing silently skips malformed entries and unsupported values

Missing names/values, malformed file references, traversal attempts, and type/encoding failures are commonly skipped or converted to plain strings without diagnostics. This makes configuration mistakes and tampering difficult to distinguish from genuinely absent resources.

**Fix:** provide strict parsing by default with file, key, and XML location in exceptions; optionally offer an explicit permissive mode with a diagnostics collection.

**Priority: P2 observability and configuration safety.**

## Required regression tests

- Reject invalid TOTP digits, periods, empty/short keys, negative windows, and unsupported algorithms with documented exceptions.
- Mutate the original key array after construction and prove generated codes do not change.
- Run thousands of concurrent TOTP calculations/verifications and compare them with isolated algorithm instances.
- Verify cryptographic resources are deterministically disposed under the chosen ownership model.
- Check the standard SHA-1 TOTP vectors and separately test any SHA-256 profile/factory.
- Force execute, commit, and multiple rollback failures; prove all compensations are attempted and the original failure remains identifiable.
- Use an enumerable that throws or mutates while enumerated and verify validation occurs before the first action executes.
- Test path containment on case-sensitive and case-insensitive systems, including symlink/reparse-point escapes.
- Reject arbitrary custom resource types unless explicitly allowlisted.
- Assert indexer, `TryGetValue`, `Values`, and enumeration return equivalent logical resource values.
- Enforce text/binary resource size limits before full allocation.
- Load resources concurrently with unknown encoding names and verify deterministic cache behavior.
- Prove callers cannot mutate cached binary resource state through a returned array.

## Deployment warning

Until these items are addressed, do not share one `Authenticator` instance across threads, do not expose its `Key` or `Algorithm` properties to untrusted code, and do not assume the named Google factory matches a default Google Authenticator profile. Treat `TransactionExecutor` as safe only when commit and rollback operations cannot fail. Load `ExternalResource` only from trusted, access-controlled resource files and directories; do not permit arbitrary custom `ResXFileRef` types or unbounded external files.
