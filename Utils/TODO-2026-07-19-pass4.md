# Utils — Fourth quality and correctness audit pass (2026-07-19)

Follow-up static audit of the current core `Utils` package after `DONE-2026-07-11.md` and audit passes 2 and 3. This pass avoids repeating previously documented findings and focuses on multidimensional array views, date-period arithmetic, command-line parsing, nested-list splitting, validation helpers, API mutability, and missing regression coverage. No production code is changed.

## Critical and high-priority findings

### 48. `ArrayAccessor<T>` validates the backing array before the derived offset is initialized

The base constructor calls the virtual `CheckSize()` method. For `ArrayAccessor<T>`, this dispatches to the derived override before the derived constructor assigns `Offset`. The check therefore always uses the default offset `0`; the real offset is assigned only afterwards.

**Risk:** `new ArrayAccessor<T>(array, nonZeroOffset, dimensions)` can pass construction even when `offset + product(dimensions)` exceeds the backing array. Later indexing or `AsSpan` then fails outside the constructor contract.

**Fix:** never call a virtual member from the base constructor. Validate the complete `(array, offset, dimensions)` tuple in the derived constructor before publishing the instance, or pass the offset into a non-virtual base validation routine.

**Priority: P0 structural correctness.**

### 49. Dimension multiplication and offset arithmetic can overflow

`CheckSize` and `CreateDimensionCaches` multiply dimensions using `int`, and cached offsets also use `int`. Large valid-looking dimensions can wrap to a small or negative value. `requiredSize + Offset` can overflow as well.

**Risk:** invalid layouts may pass validation, cache offsets may alias unrelated elements, and later indexing can access the wrong position or fail incidentally.

**Fix:** validate every dimension as non-negative, calculate products and offset sums with checked `long` arithmetic, reject values above `int.MaxValue`, and only then build integer caches.

**Priority: P1 memory-layout integrity.**

### 50. `Sizes` exposes the caller-owned mutable array while caches remain frozen

The constructor stores the supplied `sizes` array directly in the public `Sizes` property. Callers can modify it after construction, but `dimensionCaches` retains values calculated from the original dimensions.

**Risk:** bounds checking can use new dimensions while position calculation uses old strides, producing incorrect element access or out-of-range operations.

**Fix:** defensively copy dimensions and expose an immutable/read-only view. Keep one immutable layout object as the single source of truth for sizes, strides, and total length.

**Priority: P1 invariant preservation.**

### 51. Zero-dimensional and invalid-dimensional accessors fail through incidental exceptions

An empty dimension array is accepted far enough for `AsSpan()` to access `dimensionCaches[0]`; element indexing also uses `references[^1]`. Negative dimensions reach `new int[size]`, and zero-sized dimensions are not given an explicit contract.

**Fix:** define whether rank zero and zero-length dimensions are supported. Reject unsupported layouts in the constructor with contextual `ArgumentOutOfRangeException`, and test every boundary explicitly.

**Priority: P1 predictable construction contract.**

### 52. Enumeration ignores the accessor's logical offset and dimensions

`GetEnumerator()` returns the backing array's enumerator directly. An accessor representing a slice therefore enumerates elements before its offset and after its logical extent.

**Risk:** consumers receive a different data set through enumeration than through indexing or `AsSpan`, violating the type's view abstraction.

**Fix:** enumerate exactly the logical contiguous region represented by the accessor, or explicitly expose the backing collection separately and document enumeration as backing-storage enumeration.

**Priority: P1 collection semantics.**

### 53. `DateUtils.EndOf(Week)` calculates an incorrect week end

The week branch computes `(7 - (currentDay - firstDay)) % 7`, adds that value to the current date, and then adds another day before subtracting one tick. When the date is already the first day of the week, the difference is zero and the result is the end of that same day, not the end of the seventh day.

**Fix:** derive the start-of-week offset once and return `startOfWeek.AddDays(7).AddTicks(-1)`, preserving the selected calendar and date kind policy.

**Priority: P0 date correctness.**

### 54. Unix timestamp conversion is asymmetric and environment-dependent

`ToUnixTimeStamp` converts to UTC, while `FromUnixTimeStamp` converts the result to local time. A timestamp round-trip therefore changes `Kind` and displayed clock time according to the host time zone.

**Risk:** server, desktop, and test environments observe different results from the same timestamp.

**Fix:** return UTC by default, preferably through `DateTimeOffset.FromUnixTimeSeconds`, and provide an explicitly named local-time conversion only when requested.

**Priority: P1 temporal contract.**

### 55. Working-day loops have no progress guarantee

`AddWorkingDays` repeatedly asks the provider for a non-working-day count and assigns that count to the next increment. A provider that reports every day as non-working, reports a value outside the queried range, or otherwise violates assumptions can keep the loop running indefinitely. `PreviousWorkingDay` likewise has no horizon when no prior working day exists.

**Fix:** formalize the provider contract, validate returned counts against the queried inclusive range, add checked progress guards or a configurable search horizon, and throw a domain-specific exception when no working day can be found.

**Priority: P1 termination safety.**

### 56. Date period helpers do not define or consistently preserve `DateTime.Kind`

Some branches use `.Date`, while month, quarter, and year branches reconstruct values through `Calendar.ToDateTime`, which commonly produces `DateTimeKind.Unspecified`. The same input can therefore preserve or lose `Kind` depending on the selected period.

**Fix:** define the intended `Kind` policy and apply `DateTime.SpecifyKind` consistently, or migrate period-boundary APIs to `DateOnly` where time-zone semantics are not intended.

**Priority: P1 temporal consistency.**

### 57. `ParseCommandLine` silently accepts malformed quoting and cannot represent empty arguments

The parser never rejects an unterminated quote. It also filters all null/empty/whitespace results, so `""` and quoted whitespace cannot be represented as legitimate arguments. Consecutive spaces and explicit empty arguments become indistinguishable.

**Fix:** implement an explicit tokenizer with a documented escaping grammar, reject unterminated quotes with a source index, and preserve explicitly quoted empty values while optionally ignoring unquoted separator runs.

**Priority: P1 parsing correctness.**

### 58. Doubled quotes can corrupt parser state before a following space

Inside a quoted argument, every quote immediately returns to normal state. For a doubled quote escape, the first quote closes the string and the second reopens it. Depending on following characters, spaces can be treated under the wrong state and token boundaries can differ from the intended escaped-quote grammar.

**Fix:** recognize escaped quote pairs atomically while in quoted state, or adopt a well-defined platform command-line grammar and test it through a table of edge cases.

**Priority: P1 parsing correctness.**

### 59. `SplitCommaSeparatedList` can pop an empty stack and silently accepts unclosed markers

Any closing marker invokes `depth.Pop()` without first verifying that a matching opener exists. Conversely, the method reaches the end and yields the final value even when one or more opening markers remain unmatched.

**Fix:** reject unmatched closing and opening markers with `FormatException` including the source index and expected marker. Do not expose incidental `InvalidOperationException` from the stack implementation.

**Priority: P1 parser error contract.**

### 60. Nested marker matching is based only on the first character and can close the wrong marker

The API accepts `Parenthesis` values whose start and end members are strings, but the implementation compares only `[0]`. For equal start/end markers it pops whenever any marker is active, without checking that the active top marker is the same type.

**Risk:** multi-character delimiters are misrepresented, and mixed quote/parenthesis nesting can be closed by the wrong delimiter.

**Fix:** either constrain the API to single-character markers or implement full-token matching. Always compare a closing marker against the current stack top before popping.

**Priority: P1 grammar integrity.**

### 61. `Brackets(string)` and the global bracket collection are insufficiently protected

The string constructor indexes positions 0 and 1 without validating null or exact length. `Brackets.All` returns a mutable array that any caller can alter globally, changing default trimming behavior for the process.

**Fix:** validate a two-character input, use an immutable value type or record, and expose `All` as an immutable collection or a defensive copy.

**Priority: P1 global-state integrity.**

## Medium-priority findings

### 62. Validation helpers mix predicate-return and throw-immediately contracts

Methods such as `MustBeOfSize` return an `Exception?` for fluent collection, while `MustBeOfRank` is declared similarly but throws a generic `Exception` directly on failure. This prevents consistent composition through `CheckBase.Must` and contradicts the XML documentation.

**Fix:** choose one model: pure validators returning `Exception?`, or guard methods throwing typed exceptions. Rename and type the two families distinctly.

**Priority: P2 API coherence.**

### 63. Fluent validation cannot reliably accumulate independent errors

`CheckBase.Must` invokes each predicate with `Value`; the `Value` getter first throws any previously collected errors. Once one validation has recorded an error, later calls may throw before additional checks can run. Null delegates and a null checks array are also not validated.

**Fix:** evaluate checks against the private stored value, validate the delegate collection, and defer throwing until explicit value extraction or `ThrowErrors`.

**Priority: P2 diagnostic completeness.**

### 64. Aggregated validation discards original exception identity and stack information

`CheckBase.ThrowErrors` creates a generic `Exception` containing only messages; `Arg<T>` similarly wraps all failures in one `ArgumentException`. Callers cannot inspect individual causes or preserve original stacks.

**Fix:** use `AggregateException`, a dedicated validation exception carrying a read-only cause list, or attach individual failures as structured data while retaining the appropriate argument name.

**Priority: P2 diagnostics.**

## Required regression tests

- Construct an accessor whose dimensions fit at offset zero but exceed the array at a non-zero offset; construction must fail.
- Reject negative dimensions, multiplication overflow, offset overflow, and unsupported rank-zero layouts.
- Mutate the caller's original dimensions array and prove the accessor layout cannot change.
- Verify enumeration returns exactly the logical slice.
- Test week endings for every weekday and every configured first day of week.
- Round-trip Unix timestamps under multiple local time zones without changing the UTC instant.
- Use hostile calendar providers that report no working days, invalid counts, and non-progressing ranges.
- Verify a consistent `DateTime.Kind` policy across day, week, month, quarter, and year.
- Test empty quoted arguments, quoted whitespace, doubled quotes, repeated spaces, and unterminated quotes.
- Reject unmatched and mismatched nested delimiters with stable contextual exceptions.
- Prove mutation of a returned global bracket collection cannot alter later parsing.
- Verify fluent validators collect all independent errors and preserve their individual causes.

## Recommended order

| Priority | Action |
|---|---|
| P0 | Remove virtual construction-time size validation and correct week-end arithmetic |
| P1 | Make array layouts immutable and overflow-safe |
| P1 | Harden date conversion and working-day termination contracts |
| P1 | Replace ad-hoc command-line and nested-list parsing with explicit grammars |
| P2 | Normalize validation-helper contracts and structured diagnostics |

## Deployment warning

Until these items are addressed, do not use a non-zero-offset `ArrayAccessor<T>` without independently checking the complete slice bounds, and do not rely on `EndOf(Week)` for calendaring or billing boundaries. Treat command-line and nested-list parsers as suitable only for trusted, simple input; malformed delimiters can be accepted silently or fail through unrelated exceptions.
