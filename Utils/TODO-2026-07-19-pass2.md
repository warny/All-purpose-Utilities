# Utils — Second quality and security audit pass (2026-07-19)

Follow-up static audit of the current core `Utils` package after the findings consolidated in `DONE-2026-07-11.md`. This pass avoids repeating the thirteen completed items and focuses on remaining collection contracts, range invariants, random-distribution validation, interpolation parsing, MIME serialization, packaging, and missing regression coverage. No production code is changed.

## High-priority findings

### ✅ 14. `LRUCache<K,V>` accepts zero and negative capacities

The constructor stores `capacity` and immediately passes it to `Dictionary` without defining the cache contract. A negative value fails incidentally in `Dictionary`; a zero value is accepted, but the first `Add` calls `RemoveFirst()` on an empty list and then inserts an item, leaving a cache whose `Count` exceeds its declared capacity.

**Risk:** invalid configurations are either rejected through an unrelated implementation exception or silently violate the primary capacity invariant.

**Fix:** reject `capacity <= 0` with `ArgumentOutOfRangeException`, or explicitly support a zero-capacity cache by making every insertion a no-op. Prefer the former unless a zero-capacity sentinel is a documented requirement.

**Priority: P1 collection invariant.**

### ✅ 15. The `LRUCache<K,V>` indexer getter violates `IDictionary` semantics

The getter calls `TryGetValueCore` and returns its output unconditionally. When the key is absent, it returns `default(V)` rather than throwing `KeyNotFoundException`, making a missing key indistinguishable from a stored default value.

**Risk:** callers relying on normal dictionary semantics can silently consume a fabricated value and continue with corrupted application state.

**Fix:** throw `KeyNotFoundException` when `TryGetValueCore` returns false. Keep `TryGetValue` as the non-throwing API.

**Priority: P1 public contract correctness.**

### ✅ 16. `LRUCache<K,V>.Remove(KeyValuePair<K,V>)` ignores the supplied value

The `ICollection<KeyValuePair<K,V>>` contract requires removal only when both the key and value match. The current implementation delegates directly to `Remove(item.Key)`, so `Remove(new(key, wrongValue))` still deletes the cached entry.

**Risk:** generic collection algorithms can remove data they did not match, and compare-and-remove patterns are not safe.

**Fix:** under the same lock, locate the node, compare its value with `EqualityComparer<V>.Default`, and remove only on a full pair match.

**Priority: P1 collection contract correctness.**

### ✅ 17. `LRUCache<K,V>.CopyTo` methods rely on incidental array exceptions

The main collection and the `Keys`/`Values` views do not validate null arrays, negative indices, or insufficient remaining capacity before copying. Partial writes can occur before an `IndexOutOfRangeException`, unlike normal BCL collection behavior.

**Fix:** validate all arguments up front and throw `ArgumentNullException`, `ArgumentOutOfRangeException`, or `ArgumentException` consistently before copying any item.

**Priority: P1 predictable API behavior.**

### ✅ 18. `DistributedRandom` does not validate the normalization interval

The constructor evaluates the supplied function at 0 and 1 and later divides by `maxValue - minValue`. A constant function produces division by zero and `NaN`; a function returning infinities or `NaN` poisons every result. A decreasing function reverses the output distribution without an explicit contract.

**Fix:** require finite endpoint values and a non-zero interval. Define whether decreasing mappings are supported; reject them if the API intends a cumulative/inverse-distribution function. Validate every generated mapped value as finite and within the declared endpoint interval, or document that arbitrary transformations may return values outside `[0,1]`.

**Priority: P1 numeric correctness.**

### ✅ 19. `DistributedRandom.NextInt` can return the exclusive upper bound or overflow

`NextInt` casts the floating result directly to `int`. If the transformation produces exactly `1`, a value above `1`, `NaN`, or infinity, the result no longer follows the documented `[min,max)` contract. Large ranges also perform the subtraction in `int` before conversion, so `max - min` can overflow.

**Fix:** compute integer sampling with checked/wider arithmetic, validate the normalized sample, and clamp only according to a documented policy. Prefer a dedicated integer algorithm rather than casting the continuous API.

**Priority: P1 range correctness.**

### ✅ 20. `CyclicRange<T>` does not validate boundaries against its cycle

The constructor verifies only `minValue <= maxValue`; it accepts `start` or `end` outside those bounds. `Contains`, splitting, overlap, and intersection then operate on a logically invalid cycle without any failure.

**Fix:** require both boundaries to lie in `[minValue,maxValue]`. Also null-check reference-type values or add an appropriate non-null generic constraint.

**Priority: P1 structural invariant.**

### ✅ 21. Equal cyclic boundaries are interpreted as one point with no way to represent a full cycle

`Start == End` currently means a singleton. For cyclic domains, callers frequently also need a full-cycle range, and the two meanings cannot be inferred from the same endpoints.

**Risk:** APIs may silently represent “all day” as one instant when constructed with identical start/end values.

**Fix:** make the semantic choice explicit through a constructor option, a named factory such as `FullCycle`, or a separate flag. Preserve the singleton interpretation only if it is clearly documented and tested.

**Priority: P1 domain semantics.**

### ✅ 22. Cyclic compatibility uses object equality while range operations use ordering equality

`Contains` and boundary logic rely on `CompareTo`, but compatibility and intersection require `EqualityComparer<T>.Default` equality for `MinValue` and `MaxValue`. Types whose comparer considers two representations equal while `Equals` does not can therefore behave as the same ordered cycle but be rejected as incompatible.

**Fix:** use one explicit comparer consistently across construction, containment, compatibility, equality, and intersection. Prefer accepting an `IComparer<T>` and, where needed, a matching equality comparer.

**Priority: P1 generic contract consistency.**

### 23. MIME stream serialization uses the caller encoding as a whole-document transport encoding

`MimeWriter.Write(document, stream, encoding)` wraps the complete MIME document in a `StreamWriter` using the requested encoding. MIME headers and multipart framing require protocol-specific ASCII-compatible serialization, while individual bodies declare and encode their own charsets. A caller can select UTF-16 and produce NUL-separated headers and boundaries that are not valid MIME wire format.

**Fix:** serialize MIME framing and headers with a protocol-defined encoding and let each MIME part control its content transfer encoding and charset. If this helper is intended only for an internal textual representation, rename and document it accordingly.

**Priority: P1 interoperability.**

### ✅ 24. Interpolated-string alignment parsing can expose incidental numeric exceptions

Alignment is parsed with `int.Parse`. Oversized values throw `OverflowException` rather than the parser's documented `FormatException`, and the error lacks the source index and interpolation context.

**Fix:** use `int.TryParse` with invariant culture and throw a contextual `FormatException` for malformed or out-of-range alignment values.

**Priority: P1 parsing contract.**

### ✅ 25. The interpolation grammar does not prove full input coverage

The constructor iterates regex matches but never verifies that matches are contiguous and cover the entire input. The current alternatives appear intended to consume every character, but future regex changes or unsupported constructs can create unmatched gaps that are silently discarded.

**Fix:** track the expected next index, reject any gap, and verify the final match ends at `interpolatedString.Length`. This turns a fragile regex assumption into an explicit parser invariant.

**Priority: P1 parser safety.**

## Medium-priority findings

### 26. Parsed interpolation parts are only shallowly read-only

The list is read-only, but `LiteralPart` remains publicly mutable through `Append`. Consumers receiving the parsed result can therefore change the parser's stored representation after construction.

**Fix:** make parsed part implementations immutable, keep mutation internal to construction, or return immutable snapshots/records.

**Priority: P2 state integrity.**

### 27. The core package enables unsafe and preview features globally

`Utils.csproj` enables both `AllowUnsafeBlocks` and preview features for the entire foundational package. This broadens the trust and compatibility surface even for consumers using only ordinary helpers. The project also targets `net8.0` while referencing `System.Text.Encoding.CodePages` version `9.0.6`, which should be verified as an intentional and supported dependency combination.

**Fix:** isolate unsafe code into the narrowest project or file scope, disable preview features unless a documented public API requires them, and align package dependency versions with the target framework policy.

**Priority: P2 packaging and adoption stability.**

### 28. The core package has an inverted dependency on `Utils.Parser`

The shared foundation package references the much larger parser project while its package description presents it as the base layer for the ecosystem. This can pull parser dependencies into unrelated users and makes dependency cycles or version coupling more likely.

**Fix:** identify the exact parser-dependent feature and move it into a small integration package or reverse the dependency through abstractions. Keep the foundational package dependency-light.

**Priority: P2 architecture.**

## Required regression tests

- Reject zero and negative LRU capacities; prove `Count <= capacity` after every operation.
- Verify the LRU indexer throws for a missing key while `TryGetValue` remains non-throwing.
- Verify pair removal requires both key and value equality.
- Exercise every `CopyTo` invalid-argument case and assert no partial destination mutation.
- Reject constant, non-finite, and unsupported decreasing distribution functions.
- Stress integer generation across `int.MinValue`/`int.MaxValue` ranges and prove the exclusive upper bound.
- Reject cyclic boundaries outside the declared domain and test the chosen singleton/full-cycle semantics.
- Test cyclic domains whose ordering equality differs from object equality.
- Serialize MIME to a byte stream and verify protocol-valid header/boundary bytes independently of body charset.
- Reject oversized interpolation alignment with `FormatException` and a source index.
- Fuzz interpolation text and assert complete input coverage or deterministic rejection.
- Verify parsed parts cannot mutate the parser result after construction.

## Deployment warning

Until these items are addressed, do not treat `LRUCache<K,V>` as a drop-in `IDictionary<K,V>` implementation in generic code, do not use untrusted or dynamically supplied distribution functions, and do not use `MimeWriter.Write(..., Stream, encoding)` for interoperable Internet MIME when the selected encoding can alter header and boundary bytes. Avoid representing an all-cycle interval with equal cyclic endpoints unless the singleton behavior is explicitly intended.
