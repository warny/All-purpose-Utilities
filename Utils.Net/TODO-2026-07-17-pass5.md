# Utils.Net — Fifth quality and security audit pass (2026-07-17)

Follow-up static audit of the current `Utils.Net` source after the four passes consolidated in `TODO.md`. This pass avoids repeating the already documented protocol-security defects and focuses on lifecycle correctness, asynchronous event semantics, host network discovery, ARP invariants, DNS API contracts, packaging, and missing regression coverage. No production code is changed.

## High-priority findings

### 30. `CommandResponseServer.StartAsync` ignores the caller cancellation token

`StartAsync` accepts a `CancellationToken`, but creates an unrelated `CancellationTokenSource` and passes only that private token to the listener and processing loop. Cancellation requested by the caller after startup therefore has no effect.

**Risk:** hosting shutdown, request abortion, test cancellation, and application lifetime tokens cannot stop a running server. The method signature promises cancellation support that is not implemented.

**Fix:** create a linked token source from the caller token and an internal stop token, or register the caller token explicitly. Dispose the linked source deterministically.

**Priority: P1 lifecycle correctness.**

### 31. Multiple calls to `StartAsync` can orphan active listener threads and processing tasks

There is no state guard around startup. A second call overwrites `_stream`, `_reader`, `_writer`, `_listenTokenSource`, `_listenThread`, and `_processTask` while the previous listener and queue processor may still be running.

**Risk:** leaked threads/tasks, commands read from one stream and answered on another, disposal of the wrong resources, and nondeterministic cross-session state corruption.

**Fix:** make the instance explicitly single-start/single-use, or implement a complete stopped-to-started transition protected by a lifecycle lock. Throw `InvalidOperationException` on concurrent or repeated startup unless reuse has been fully reset and tested.

**Priority: P1 structural lifecycle safety.**

### 32. Multicast asynchronous `CommandReceived` handlers are not all awaited

`CommandReceived` is declared as `event Func<string, Task<IEnumerable<ServerResponse>>>`. Invoking a multicast delegate with a return value executes every subscriber but returns only the value from the last subscriber. The server therefore awaits only the last returned task; tasks from earlier subscribers can continue unobserved, fault silently, and have their responses discarded.

**Risk:** lost responses, unobserved exceptions, concurrent side effects, and behavior that changes with subscription order.

**Fix:** either allow exactly one fallback handler, or enumerate `GetInvocationList()`, await each handler according to an explicit sequential/parallel policy, and define how response sequences are combined.

**Priority: P1 asynchronous correctness.**

### 33. Command registration and context state are mutable without synchronization

`_handlers` is a `Dictionary` and `_contexts` is a `HashSet`. Public `RegisterCommand`, `AddContext`, `RemoveContext`, and `HasContext` can run concurrently with the processing loop and with handler access through `CommandContext`.

**Risk:** standard collection races, inconsistent required-context checks, and runtime exceptions if application code changes registrations or contexts while a session is active.

**Fix:** define configuration as immutable after startup, enforce that state, and keep session contexts owned by the processing loop. If runtime mutation is required, serialize it through the same command pipeline or protect it with a clear synchronization strategy.

**Priority: P1 concurrency contract.**

### 34. Handler and formatter exceptions still fault the entire processing task without a protocol-level close path

The current loop catches only `OperationCanceledException`. Exceptions from command handlers, the fallback event, response enumeration, the formatter, logging providers, or stream writes escape and fault `_processTask`. The listener thread can remain blocked until another component notices and disposes the instance.

This is related to the existing finding about handler isolation, but the current source also lacks one coordinated fault path that cancels the listener, completes pending work, and records the terminal exception.

**Fix:** wrap the complete dispatch/write transaction, cancel the session on an unexpected exception, close or complete the transport deterministically, and expose the terminal fault through `Completion` without silently swallowing it during normal observation.

**Priority: P1 failure containment.**

### 35. ARP serialization assumes a fixed Ethernet/IPv4 layout while exposing mutable length and type fields

`ArpPacket.ToBytes()` always allocates 28 bytes and copies addresses at Ethernet/IPv4 offsets. However, `HardwareAddressLength`, `ProtocolAddressLength`, `HardwareType`, `ProtocolType`, and all address properties are publicly mutable. IPv6 addresses, non-six-byte hardware addresses, or custom length fields can overflow the buffer or produce a packet whose header disagrees with its body.

**Fix:** either specialize the type as Ethernet/IPv4 ARP and validate exactly six-byte hardware plus four-byte protocol addresses, or allocate and lay out the packet from the declared lengths while validating that the supplied addresses match them.

**Priority: P1 packet correctness.**

### 36. ARP parsing validates a constant 28-byte minimum instead of the packet-declared size

`ArpPacket.Read` accepts any input of at least 28 bytes, then trusts the packet's hardware/protocol length fields when slicing. A crafted packet can declare lengths whose calculated body exceeds the available span, causing incidental range exceptions. Conversely, a valid non-Ethernet/non-IPv4 ARP packet can be rejected or misinterpreted.

**Fix:** after reading the first eight bytes, calculate `8 + 2 * hardwareLength + 2 * protocolLength` with checked arithmetic, require that many bytes, reject unreasonable or unsupported combinations explicitly, and parse only within that bound.

**Priority: P1 untrusted packet parsing.**

### 37. `DNSLookup` suppresses every per-server failure and loses all diagnostic context

The resolver catches all exceptions for each configured name server and finally throws a generic `Exception("Unable to execute the request")`. Cancellation, socket errors, malformed DNS responses, unsupported records, programming errors, and validation failures are indistinguishable. The original exceptions and failed endpoints are discarded.

**Risk:** operational diagnosis is difficult, security-validation failures can be silently retried as if they were transient transport errors, and callers cannot distinguish timeout from malformed or hostile responses.

**Fix:** catch only expected transport/parse exceptions, preserve endpoint and inner exceptions, aggregate failures when all servers fail, and provide cancellation-aware asynchronous APIs.

**Priority: P1 error semantics.**

### 38. `DNSLookup.NameServers` exposes mutable aliased arrays and accepts invalid resolver configurations

Constructors retain the caller-provided array directly, and the public property is freely replaceable. Null arrays, empty arrays, null entries, or concurrent mutation during `Request` are not rejected.

**Risk:** races and nondeterministic resolver selection; an empty configuration produces only the generic final exception; external code can change the active trust boundary after construction.

**Fix:** validate at least one non-null unicast address, defensively copy to an immutable/read-only collection, and require an explicit method or new resolver instance to change name servers.

**Priority: P1 resolver configuration integrity.**

## Medium-priority findings

### 39. Host DNS discovery incorrectly requires a default gateway

`NetworkParameters` collects DNS servers only from interfaces that are `Up` and have at least one gateway. Valid DNS configuration can exist on VPN, point-to-point, container, loopback, manually routed, or split-DNS interfaces without a default gateway.

**Risk:** `DNSLookup()` may omit the resolver that actually owns a private/split namespace or return no resolvers despite a usable system DNS configuration.

**Fix:** separate interface usability, route/gateway selection, and DNS discovery. Collect DNS addresses from suitable active interfaces, preserve interface/metric/scope metadata, and define an ordering policy rather than using gateway presence as a proxy.

**Priority: P2 host integration correctness.**

### 40. `PrimaryDns` throws an incidental index exception when no DNS server was discovered

`PrimaryDns` returns `DnsServers[0]` with no empty-state contract. Hosts with no eligible interface, restricted containers, transient network changes, or the gateway-filter issue above receive `IndexOutOfRangeException`.

**Fix:** expose `IPAddress? PrimaryDns`, `TryGetPrimaryDns`, or throw a documented domain-specific `InvalidOperationException`. Deduplicate discovered servers and return immutable snapshots.

**Priority: P2 API predictability.**

### 41. `DNSHeader.Append` has a contradictory identity contract and does not merge header semantics

The documentation says records are appended when IDs do not match, and the method throws when IDs are equal. For fragments or sections belonging to one DNS transaction, matching IDs would normally be the expected correlation condition. The exception text says "Mismatch headers" in the equal-ID case. The method also merges only record lists, leaving flags, response code, truncation state, and counts unspecified.

**Fix:** define the actual use case. If combining parts of one response, require matching transaction/question identity and reconcile flags deterministically. If combining unrelated result sets, rename the operation to avoid implying DNS packet reassembly and remove transaction-header semantics from the result.

**Priority: P2 semantic clarity.**

### 42. Server configuration arguments are not validated or defensively copied

`RegisterCommand` accepts null/blank commands, a null handler at runtime, and caller-owned `requiredContexts` arrays. `MaxConsecutiveErrors` accepts negative values, which silently behave like disabled limiting. Context methods likewise accept null/blank names.

**Fix:** validate command/context tokens, reject null handlers and negative limits, normalize command keys consistently, and copy required-context arrays into immutable storage.

**Priority: P2 defensive API correctness.**

### 43. Preview features are enabled package-wide without a demonstrated requirement

`Utils.Net.csproj` targets only `net9.0` and enables preview features for the complete networking package. The inspected public APIs use ordinary supported runtime/language features.

**Risk:** consumers inherit unnecessary preview constraints, compiler/runtime compatibility becomes less predictable, and a broad network utility package has a narrower adoption surface than required.

**Fix:** disable preview features unless a documented API requires them; pin the language version where necessary and consider an additional supported target if the implementation permits it.

**Priority: P2 packaging stability.**

## Duplications of intent to reduce

- Session lifetime is spread across stream fields, a thread, a task, a semaphore, and a cancellation source. Replace this with one explicit state machine/session object.
- Command/context validation is repeated implicitly across protocol subclasses. Centralize line-token and state-token validation in the command/response base layer.
- Resolver configuration, host DNS discovery, transport retries, and failure aggregation need one coherent resolver-options model rather than mutable arrays plus catch-all retry.
- ARP address lengths are represented both by mutable header fields and by the actual address objects. Establish one source of truth.

## Required regression tests

- Cancel the token supplied to `StartAsync` and verify listener, processor, and transport shutdown.
- Call `StartAsync` twice concurrently and sequentially; verify the chosen single-use/reuse contract.
- Register multiple asynchronous `CommandReceived` subscribers and prove every task is awaited or explicitly rejected.
- Mutate handlers and contexts during processing and verify the documented immutable/serialized policy.
- Force handler, formatter, logger, and writer exceptions and assert deterministic session termination plus observable `Completion` failure.
- Serialize Ethernet/IPv4 ARP and reject IPv6, non-six-byte MACs, inconsistent declared lengths, and buffer-overflow cases.
- Parse ARP packets with zero, oversized, and truncated declared address lengths using checked size calculations.
- Configure empty/null/mutated DNS server collections and verify explicit validation.
- Preserve endpoint-specific failures when all DNS servers fail; distinguish timeout, cancellation, parse failure, and protocol rejection.
- Discover DNS on an active VPN/point-to-point interface without a default gateway and handle a host with no DNS servers.
- Define and test `DNSHeader.Append` transaction-ID, question, flags, and error-code behavior.
- Verify all public configuration methods reject null, blank, control-character, and invalid range inputs.

## Deployment warning

The deployment warning from `TODO.md` remains applicable. In addition, treat each `CommandResponseServer` instance as single-start and externally synchronized, do not rely on the `StartAsync` cancellation token, and do not attach multiple asynchronous `CommandReceived` subscribers. Do not pass unvalidated or non-Ethernet/IPv4 data to `ArpPacket`, and do not rely on `NetworkParameters.PrimaryDns` on hosts with VPN, split-DNS, container, or gateway-less interfaces.
