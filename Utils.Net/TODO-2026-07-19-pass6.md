# Utils.Net — Sixth quality and security audit pass (2026-07-19)

This pass reviews the current source after the fixes already recorded in `DONE-2026-07-17.md`, the consolidated audit in `TODO.md`, and the fifth pass in `TODO-2026-07-17-pass5.md`. It intentionally avoids repeating findings already documented there and focuses on remaining client lifecycle, response routing, timer, disposal, framing, server enumeration, and NTP endpoint-selection defects. No production code is changed.

## High-priority findings

### 44. `UnsolicitedResponseReceived` is raised for every response line, including command replies

`ListenLoop` enqueues each parsed response and immediately invokes `UnsolicitedResponseReceived`. The same response is then consumed by `SendCommandAsync` or `ReadAsync`. `DrainPendingResponses` can invoke the event again for leftover queued responses.

**Risk:** expected command replies are misclassified as unsolicited, consumers can process the same response twice, and stateful protocol clients may observe replies before the command method returns.

**Fix:** route each incoming line exactly once. While a command owns the response channel, deliver its replies only to that command. Raise the unsolicited event only when no command/read waiter owns the response, or expose a separate observable stream with explicitly documented duplicate-delivery semantics.

**Priority: P1 protocol correctness.**

### 45. Exceptions from unsolicited-response subscribers terminate the listener thread

`UnsolicitedResponseReceived?.Invoke(response)` executes on the dedicated listener thread inside a block that catches only `IOException` and `ObjectDisposedException`. Any subscriber exception escapes, runs the `finally`, marks the client disconnected, and abandons the connection.

**Risk:** application callback code can accidentally kill transport processing; pending commands then fail as if the peer disconnected, and the original callback exception is not exposed through a completion task.

**Fix:** isolate subscriber exceptions from the transport loop, log or aggregate them through a dedicated callback-error channel, and define whether subscribers execute sequentially, through the thread pool, or through a bounded event dispatcher.

**Priority: P1 failure isolation.**

### 46. The keep-alive timer uses an `async void` callback and can overlap with itself

The `Timer` callback is an `async` lambda converted to `async void`. Timer disposal cannot await an in-flight callback. Changes to `NoOpInterval`, reconnects, slow commands, or callbacks already queued by the timer can produce overlapping keep-alive attempts. Exceptions are swallowed inside `SendNoOpAsync`, so repeated failures are invisible.

**Risk:** keep-alive commands can race with disconnect/disposal, retain the client after disposal, contend indefinitely on `_sendLock`, and hide a dead connection.

**Fix:** replace `Timer` with one cancellation-aware periodic task (`PeriodicTimer` or an explicit delay loop), serialize one keep-alive at a time, await shutdown, and expose/log terminal failures according to policy.

**Priority: P1 asynchronous lifecycle.**

### 47. `ConnectAsync(Stream, ...)` does not bind the listener lifetime to the caller token

The supplied cancellation token is passed only to `OnConnect`. The listener uses a new unrelated `CancellationTokenSource`, so cancelling the token after connection does not stop reads or close the client.

This is distinct from the already documented server-side token issue: the client has the same misleading lifetime contract.

**Fix:** link the caller token to the internal connection token for the entire session, or rename/document the parameter as applying only to connection initialization. A linked lifetime token should also stop the keep-alive loop and wake pending command waiters.

**Priority: P1 cancellation correctness.**

### 48. Repeated or concurrent `ConnectAsync` calls overwrite a live connection

There is no lifecycle guard before assigning `_client`, `_stream`, `_reader`, `_writer`, `_listenTokenSource`, `_listenThread`, and `_keepAliveTimer`. A second connect can leave the first listener and resources alive while all fields now refer to the second connection.

**Risk:** responses can be read from one stream and queued for commands sent on another; old timers and threads leak; disposal can close only the latest resources.

**Fix:** make `CommandResponseClient` explicitly single-use, or implement a complete state machine with atomic transitions and full teardown before reconnect. Reject concurrent connection attempts.

**Priority: P1 structural lifecycle safety.**

### 49. Disposal is not idempotent and can race with active operations

`Dispose` unconditionally disposes `_responseSignal` and `_sendLock`, does not set a disposed flag, and can be called again by `DisconnectAsync`, user code, or concurrent paths. Listener/timer callbacks and pending commands may still access these semaphores after they are disposed.

**Risk:** secondary `ObjectDisposedException`, release-after-dispose races, and loss of the original network failure during shutdown.

**Fix:** use an interlocked disposed state, cancel first, await or join all background work, fail pending waiters, then dispose synchronization primitives exactly once. Public operations should throw `ObjectDisposedException` consistently.

**Priority: P1 resource lifecycle.**

### 50. The finalizer performs managed-resource cleanup from the finalizer thread

The finalizer calls `Dispose(false)`, but that path still cancels a managed `CancellationTokenSource`, disposes readers, writers, streams, timers, clients and semaphores, and joins the listener thread whenever `_writer` is non-null.

**Risk:** blocking the finalizer thread, invoking arbitrary managed disposal logic after object graphs may already be finalized, and deadlock or long GC pauses while joining a thread.

**Fix:** remove the finalizer because the class owns only managed disposable resources. If a finalizer is ever required for a native handle, restrict the finalizer path to a `SafeHandle` and never join threads or dispose managed synchronization objects there.

**Priority: P1 runtime stability.**

### 51. A failed TCP connection attempt can retain partially initialized resources

`ConnectAsync(string, int, ...)` assigns a new `TcpClient`, awaits connection, then delegates to stream setup. If DNS/connect cancellation, socket connection, stream acquisition, or `OnConnect` fails, no catch disposes the newly allocated client and any partially created stream/listener resources.

**Fix:** build the connection in local variables, transfer ownership to fields only after successful initialization, and roll back every partially acquired resource on failure.

**Priority: P1 resource leak prevention.**

### 52. Line limits are documented as bytes but enforced as UTF-16 character counts

Both client and server use `StreamReader.Read()` and compare `StringBuilder.Length` with `MaxLineLength`, while the public documentation says the limits are measured in bytes. ASCII is configured today, but replacement/fallback behavior and future encoding changes make the contract inaccurate.

**Risk:** limit policy is ambiguous and can drift silently when encoding changes; callers configuring a byte-level protocol limit are not getting a byte-level guarantee.

**Fix:** either rename/document the setting as characters, or implement a bounded byte-oriented line reader that validates the protocol encoding and counts raw octets before decoding.

**Priority: P1 framing contract.**

### 53. `SendLinesAsync` bypasses command-line validation

Protocol subclasses can pass arbitrary lines to `SendLinesAsync`; unlike public argument-specific helpers, this method does not reject CR, LF, NUL, or embedded additional commands. A caller-controlled enumerable can therefore inject extra lines through one logical payload.

**Fix:** distinguish trusted raw protocol payload streaming from command-line transmission. Validate ordinary lines centrally, and expose a narrowly scoped raw-data method only for protocol states such as SMTP DATA/NNTP POST, with dot-stuffing and terminator handling owned by that protocol.

**Priority: P1 command framing.**

### 54. Lazy response enumerables can fault outside the handler exception boundary

Server handler invocation is wrapped in `try/catch`, but `responses.ToList()` occurs afterward. A handler returning a lazy enumerable can throw while enumerated, faulting `ProcessQueueAsync` outside the intended conversion to a `500` reply.

**Fix:** materialize and validate the enumerable inside the protected handler transaction. Reject null response entries and impose a response-count/byte limit before writing.

**Priority: P1 failure containment.**

## Medium-priority findings

### 55. `SendResponseAsync` has no cancellation token or session-state check after lock acquisition

An unsolicited server response waits indefinitely for `_writeLock`, cannot be cancelled by the caller, and may proceed after session cancellation or disposal once it acquires the semaphore.

**Fix:** add a cancellation token linked to the session lifetime, recheck running/disposed state after acquiring the lock, and define how pending unsolicited sends complete during shutdown.

**Priority: P2 asynchronous API contract.**

### 56. NTP uses only the first DNS address with no fallback policy

`GetTimeAsync` resolves all addresses but selects `addresses[0]`. If that address family or endpoint is unreachable, usable later addresses are never attempted. The selected ordering is controlled by the system resolver and may vary between hosts.

**Fix:** attempt resolved endpoints under one overall timeout/cancellation budget, preserve per-endpoint failures, and optionally support an address-family preference. Validate the port and host argument before DNS resolution.

**Priority: P2 network resilience.**

### 57. NTP accepts broadcast-mode responses on a connected unicast request

The method accepts mode 5 (broadcast) even though it explicitly connects a UDP socket to one server and sends a client-mode request. A normal response to this transaction should be server mode 4; accepting broadcast mode weakens the request/response state contract.

**Fix:** require mode 4 for this request API. If broadcast listening is desired, expose it as a separate API with different socket, correlation, and trust semantics.

**Priority: P2 protocol validation.**

### 58. Configuration properties can overflow or throw incidental timer/socket exceptions

`ListenTimeout` casts `TotalMilliseconds` directly to `int`; very large, negative (other than infinite), `NaN`-derived, or otherwise invalid values can overflow or trigger incidental stream exceptions. `NoOpInterval`, `MaxLineLength`, and `MaxResponseCount` likewise accept unsupported negative values without a coherent validation contract.

**Fix:** validate every setting at assignment, define accepted ranges, use checked conversion, and reject changes that cannot safely be applied while connected.

**Priority: P2 defensive API correctness.**

## Duplicated intent to consolidate

- Client and server both implement character-by-character line framing, logging sanitation, background listener threads, queues and semaphores. Extract one bounded line transport with explicit byte/character semantics and lifecycle ownership.
- Connection lifetime is spread across booleans, nullable fields, threads, timers and semaphores. Replace it with an explicit state machine and one session cancellation source.
- Command validation exists in protocol-specific call sites but not at the final write boundary. Centralize line safety immediately before transport writes.
- Background callback failures currently have no consistent observation mechanism. Provide one completion/fault task for listener, keep-alive and unsolicited-callback pipelines.

## Required regression tests

- Verify one solicited response is delivered only to its command waiter and is not raised as unsolicited.
- Throw from an unsolicited-response subscriber and verify the transport remains alive while the callback error is observable.
- Use a slow NOOP response and prove only one keep-alive runs at a time; dispose while it is active and verify shutdown completes.
- Cancel the token supplied to `ConnectAsync(Stream, ...)` after connection and verify listener, timer and waiters stop.
- Call `ConnectAsync` twice concurrently and sequentially and verify the chosen single-use/reconnect contract.
- Dispose concurrently from user code, `DisconnectAsync`, listener termination and keep-alive; verify exactly-once cleanup.
- Force failure at TCP connect, stream setup and `OnConnect`; verify every partially acquired resource is disposed.
- Test line limits with raw octets, non-ASCII replacement input and exact boundary lengths.
- Attempt CR/LF/NUL injection through `SendLinesAsync` and every protocol-specific multiline API.
- Return a lazy response enumerable that throws during enumeration and verify a controlled `500` or deterministic session fault.
- Cancel `SendResponseAsync` while it waits for another writer and during session shutdown.
- Resolve an NTP host to multiple endpoints where the first fails and a later one succeeds.
- Reject broadcast-mode NTP packets in the unicast request API.
- Validate negative, infinite, excessively large and boundary timeout/limit configuration values.

## Deployment warning

Until findings 44–54 are fixed, treat `CommandResponseClient` as single-use, do not subscribe callbacks that may throw, disable keep-alive unless externally supervised, and do not call `ConnectAsync` or `Dispose` concurrently. Do not use `SendLinesAsync` with caller-controlled strings. Continue to apply all deployment warnings from the earlier `Utils.Net` audits.
