# Utils.Net — Seventh quality and protocol audit pass (2026-07-19)

This pass complements the earlier `Utils.Net` audit files. It focuses on SMTP, POP3, NNTP, protocol-state recovery, multi-line parsing, argument grammar, time semantics, and malformed-server behavior. No production code is changed.

## High-priority findings

### 59. SMTP `EHLO` drops the final advertised extension

`EhloAsync` collects messages only for indexes `1` through `responses.Count - 2`. The final `250 ...` line is excluded even though SMTP servers commonly advertise a capability on that final line rather than using it only as a terminator.

**Risk:** capabilities such as `STARTTLS`, `AUTH`, `SIZE`, `SMTPUTF8`, or `CHUNKING` can be silently omitted. A caller may make an incorrect security or feature decision from an incomplete capability set.

**Fix:** parse every `250-...` and final `250 ...` line according to SMTP multiline semantics. Treat the first line separately only when it is demonstrably the server identity text, not by positional assumption. Return a structured capability model preserving keyword and arguments.

**Priority: P1 protocol correctness.**

### 60. SMTP envelope arguments can escape the generated angle-bracket syntax

`SendMailAsync` validates only CR, LF, and NUL before interpolating sender and recipient strings into `MAIL FROM:<...>` and `RCPT TO:<...>`. An argument containing `>` can terminate the mailbox path and append SMTP parameters or other malformed syntax on the same line.

**Risk:** command semantics can be altered despite the generic line-injection guard. Even when the peer rejects the line, the API contract does not guarantee that the supplied value is used only as one mailbox path.

**Fix:** model SMTP paths explicitly. Validate or encode mailbox/path syntax, reject unmatched `<`/`>`, control characters, whitespace and unsupported source routes, and expose ESMTP parameters as separate typed arguments rather than accepting them inside the address string.

**Priority: P1 command construction safety.**

### 61. SMTP mail transactions are not recovered after a partial recipient or DATA failure

`SendMailAsync` sends `MAIL FROM`, then each `RCPT TO`, then `DATA`. If one recipient is rejected, cancellation occurs, message generation fails, or the final DATA response is negative, no `RSET` is issued and no transaction-state object records the uncertain server state.

**Risk:** the next operation can inherit a partially open transaction, accepted recipients can remain associated with later commands, and retry behavior becomes server-dependent.

**Fix:** implement an explicit SMTP transaction state machine. On any failure after `MAIL FROM`, attempt `RSET` under a bounded cleanup token, preserve both the original and cleanup failures, and prevent another transaction until recovery or reconnect succeeds.

**Priority: P1 state integrity.**

### 62. SMTP accepts an empty recipient sequence and begins DATA without a valid envelope recipient

The recipient enumerable is not validated for nullness or emptiness and is consumed only once inside the send flow.

**Risk:** a null enumerable fails after `MAIL FROM`; an empty enumerable sends `DATA` without any `RCPT TO`; a lazy enumerable can throw or change while the transaction is already active.

**Fix:** materialize and validate a defensive recipient snapshot before sending `MAIL FROM`. Require at least one valid recipient and reject null entries before mutating remote state.

**Priority: P1 transaction validation.**

### 63. SMTP AUTH PLAIN and LOGIN force credentials through ASCII

User names and passwords are converted with `Encoding.ASCII`, replacing non-ASCII characters with `?`. SASL PLAIN uses UTF-8 representations under its modern specification, subject to mechanism/profile rules.

**Risk:** valid international credentials are corrupted locally, while logs and exception context do not explain that the transmitted credential differs from the supplied value.

**Fix:** define the supported SASL profile explicitly, use strict UTF-8 when appropriate, reject unrepresentable input rather than replacing characters, and validate prohibited NUL placement separately for authorization identity, authentication identity, and password.

**Priority: P1 authentication correctness.**

### 64. POP3 and NNTP multi-line terminators are accepted after trimming trailing whitespace

Both multi-line readers use `line.TrimEnd() == "."`. The protocol terminator is the exact octet sequence `".\r\n"`; a line such as `". "` is data, not a terminator.

**Risk:** malformed or hostile responses can truncate message/article data prematurely.

**Fix:** recognize only an exact single-dot logical line after CRLF framing. Do not normalize whitespace before terminator detection.

**Priority: P1 untrusted-response parsing.**

### 65. POP3 message retrieval changes line endings to the host platform convention

`RetrieveAsync` rebuilds message text with `StringBuilder.AppendLine`, which uses `Environment.NewLine` rather than POP3/Internet message CRLF.

**Risk:** retrieved MIME data is not byte- or text-faithful across platforms; signatures, hashes, canonicalization, and downstream parsers can observe modified content.

**Fix:** append `"\r\n"` explicitly, or expose a streaming/raw-byte API that preserves protocol octets. Document whether the final CRLF before the terminator is retained.

**Priority: P1 data fidelity.**

### 66. POP3 numeric response parsing silently manufactures zero values or throws incidental format exceptions

`GetStatAsync` returns zero for missing fields but uses `int.Parse` for present malformed or oversized values. Similar methods silently ignore malformed LIST/UIDL rows.

**Risk:** a malformed response can look like an empty mailbox, fail with an unhelpful `FormatException`, or produce an incomplete list without indicating protocol corruption.

**Fix:** use invariant, bounded `TryParse`; require the exact response shape for single-line commands; decide whether malformed rows invalidate the complete multi-line response or are returned as structured parse errors.

**Priority: P1 response integrity.**

### 67. NNTP UTC parameters are formatted without enforcing UTC semantics

`NewGroupsAsync` and `NewNewsAsync` accept parameters named `sinceUtc` but format the supplied `DateTime` directly. Local or unspecified values are sent unchanged with no `GMT` token and no conversion policy.

**Risk:** queries can target the wrong time window depending on caller `DateTime.Kind`, local timezone, DST, and server interpretation.

**Fix:** prefer `DateTimeOffset`, convert to UTC explicitly, and send the protocol's UTC/GMT indicator where required by the selected NNTP command form. Reject ambiguous `DateTimeKind.Unspecified` if retaining the existing signature.

**Priority: P1 temporal correctness.**

### 68. NNTP dot-unstuffing accepts malformed single-dot-prefixed data

Article, header, and body methods remove the first dot from every non-terminator line beginning with a dot. Correct dot-stuffed data begins with two dots; a malformed line such as `.abc` should not silently become `abc`.

**Risk:** corrupt responses are normalized into plausible article content and evidence of protocol invalidity is lost.

**Fix:** unstuff only lines beginning with `".."`; reject or preserve other single-dot-prefixed lines according to a documented strict/permissive policy.

**Priority: P1 data validation.**

### 69. Multi-line limit failures do not recover or retire the connection

When POP3 or NNTP line/character limits are exceeded, the reader throws immediately but unread lines through the terminating dot remain queued or continue arriving on the connection.

**Risk:** the next command can consume the remainder of the previous response as if it were its own status or payload, permanently desynchronizing the protocol session.

**Fix:** exceeding a framing or aggregate limit must poison and close the connection, or drain to the exact terminator under a separate hard bound before allowing reuse. Expose the connection as unusable until one of those paths completes.

**Priority: P1 session synchronization.**

### 70. Generic response-count limits can split legitimate protocol multi-line replies from their payload phase

`SendCommandAsync` limits only status responses collected before a completion/unknown severity line. POP3 and NNTP then read their dot-terminated payload through a separate API with separate limits. The two-stage contract is implicit and shared-queue based.

**Risk:** unsolicited events, cancellation, keep-alive activity, or another consumer can interleave between status and payload phases. The connection has no ownership token proving that the same command still owns subsequent lines.

**Fix:** represent a command exchange as one exclusive protocol operation that includes both initial status and optional dot-terminated body. Keep the send lock for the complete exchange or provide a lower-level transaction lease consumed by protocol subclasses.

**Priority: P1 concurrency and framing.**

## Medium-priority findings

### 71. NNTP `NextAsync` converts every negative response into “no next article”

Authentication failures, missing group selection, temporary server errors, and protocol failures all return `null`, identical to the normal no-next-article case.

**Fix:** inspect response codes and map only the documented “no next article” status to `null`; throw a typed protocol exception for other negative responses.

**Priority: P2 error semantics.**

### 72. Protocol methods expose generic `IOException` without preserving response codes

SMTP, POP3, and NNTP success guards throw `IOException` containing only the final message. The numeric/symbolic response code, severity, full response sequence, and command context are not retained structurally.

**Fix:** introduce a `ProtocolResponseException` carrying protocol name, command verb, final code, severity, response lines, and optional enhanced status code. Avoid including secret command arguments.

**Priority: P2 diagnostics and caller control.**

### 73. SMTP/POP3/NNTP payloads are fully materialized as strings and line lists

Mail DATA, retrieved messages, articles, headers, bodies, and large LIST/CAPA results are accumulated in memory. Existing line/character limits reduce but do not remove large allocation and duplication costs.

**Fix:** add streaming APIs over `TextReader`, `TextWriter`, `Stream`, or `IAsyncEnumerable<string>`, with byte-based limits and cancellation. Keep convenience materializing overloads as bounded wrappers.

**Priority: P2 scalability.**

## Duplicated intent to consolidate

- Centralize exact dot-terminated framing, strict terminator recognition, dot unstuffing, byte/line limits, and connection-poison behavior for POP3 and NNTP.
- Replace repeated `EnsureCompletionAsync` helpers with typed protocol response validation that retains codes and context.
- Model SMTP envelope paths, recipients, ESMTP parameters, and transaction recovery in one transaction object.
- Make a complete command plus optional payload phase own the connection exclusively rather than coordinating through the shared response queue manually.

## Required regression tests

- Parse EHLO responses where the only `STARTTLS` or `AUTH` advertisement is on the final `250` line.
- Reject SMTP paths containing closing brackets, whitespace/parameter injection, null recipients, empty recipient sets, and throwing lazy enumerables before any command is sent.
- Fail after one accepted recipient and verify bounded `RSET` recovery before the next transaction.
- Round-trip non-ASCII SASL credentials according to the chosen profile and reject forbidden NULs.
- Treat `. ` as POP3/NNTP data rather than a terminator; unstuff `..abc` but reject or preserve `.abc`.
- Verify POP3 retrieval always produces CRLF independent of operating system.
- Reject malformed/overflowing STAT, GROUP, LIST, UIDL, and STAT article fields without returning partial or zero-filled semantic results.
- Pass Local, UTC, and Unspecified times to NEWGROUPS/NEWNEWS and verify the documented UTC policy.
- Exceed multiline limits and prove the connection is either drained safely or marked unusable.
- Cancel between initial success status and dot-terminated payload and ensure no later command can consume the abandoned body.
- Distinguish NNTP “no next article” from authentication, sequence, and temporary server errors.
- Stream payloads larger than the convenience materialization limit without buffering the complete body.

## Deployment warning

Until items 59–70 are addressed, do not use the SMTP capability list as an authoritative security-feature inventory, do not pass untrusted mailbox strings to envelope methods, and do not reuse a POP3/NNTP connection after cancellation or a multiline-limit failure. Retrieved POP3 message text is not currently platform-independent, and protocol failures are not reliably distinguishable from ordinary empty/no-result conditions.
