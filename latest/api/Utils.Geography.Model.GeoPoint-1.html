EqualityPrecision

Number of decimal places latitude/longitude (and bearing, for GeoVector<T>) are rounded to before being compared in Equals(GeoPoint<T>?) and hashed in GetHashCode(). Rounding both sides the same way keeps the two members consistent with each other by construction, at the cost of treating two values that are extremely close but land on opposite sides of a rounding boundary as unequal. See the remarks on Equals(GeoPoint<T>?) for the full rationale and a worked example.

public bool Equals(GeoPoint<T>? other)

Parameters

Returns

bool

true if both points round to the same latitude and longitude (or are both at the same pole, regardless of longitude); otherwise false.

Remarks

Why rounding, and not a tolerance window: latitude/longitude are floating-point values produced by trigonometric computations (Travel(T), Intersections(GeoVector<T>), ...), which never land on exact values. An earlier implementation compared raw values with a tolerance window (|a - b| <= 1e-5) via FloatingPointComparer<T>, but that comparer is intentionally non-transitive (see its own documentation) and cannot be paired with a correct GetHashCode(): two values considered equal by the tolerance window could still hash differently, breaking Dictionary<TKey, TValue>/HashSet<T> usage. Rounding both operands to the same precision before comparing removes that risk entirely, because Equals(GeoPoint<T>?) and GetHashCode() then always operate on the exact same rounded values.

Known limitation — rounding-boundary straddling: this trades away the "equal within a tolerance window" behavior. Two values that are extremely close to each other, but fall on opposite sides of a rounding boundary, are now treated as unequal, even though a raw tolerance comparison would have called them equal. For example, with EqualityPrecision = 5, 1.0000449999 rounds down to 1.00004 while 1.0000450001 rounds up to 1.00005 — a difference of about 2e-7, far below the old 1e-5 tolerance, yet the two values are no longer equal. This is a deliberate, accepted trade-off (values this close in practice only arise from floating-point noise around a rounding boundary, and rounding-consistency was judged more valuable than tolerance-window equality for this type). See GeoPointTests.PointsOnOppositeSidesOfARoundingBoundaryAreNotEqual for a regression test that pins down this exact behavior.

Longitude wraps around, latitude doesn't: longitude is compared via AreEqualRounded(T, T, int) rather than a plain rounded comparison, so that values on opposite sides of the antimeridian (e.g. 179.9999951° and -179.9999999°, which both refer to almost the same point near 180°) round to the same normalized value instead of comparing as ~360° apart. Latitude never wraps (it is clamped to [-90°, 90°] and the poles are handled separately above), so it only needs a plain rounded comparison.

GetHashCode()

Returns a hash code consistent with Equals(GeoPoint<T>?): latitude is rounded, and longitude is normalized (to handle antimeridian wraparound) and rounded, to EqualityPrecision decimal places before hashing — the exact same values that Equals(GeoPoint<T>?) compares on, so equal points always hash equally.

Remarks

Equals(GeoPoint<T>?) intentionally does not offer this behavior anymore (see its remarks): a tolerance window cannot be paired with a correct GetHashCode() because it is non-transitive, so points must not be considered equal, hashed, or stored in a Dictionary<TKey, TValue>/HashSet<T> based on this method. Use it only for one-off comparisons (e.g. "is this GPS fix close enough to the target waypoint?"), never as a substitute for Equals(GeoPoint<T>?) in code that also relies on hashing.

Longitude comparison correctly handles the antimeridian wraparound via AreEqual(T, T, T) (e.g. 179.9999999° and -179.9999999° are about 2e-7° apart, not ~360° apart).
