Properties

Bounds

Gets the rectangular bounds, expressed in this projection's own native output units, that GeoPointToMapPoint(GeoPoint<T>) stays within for every valid GeoPoint<T>. Used by Normalize(ProjectedPoint<T>) (and, in turn, by MapPoint<T> and RepresentationConverter<T>) to convert a projected point into map-fraction coordinates before scaling to a pixel/tile grid.

(T MinX, T MaxX, T MinY, T MaxY) Bounds { get; }

Property Value

(T MinX, T MaxX, T MinY, T MaxY)

Remarks

For a projection whose output is genuinely unbounded at some point of the sphere (e.g. Mercator at the poles), Bounds must report a practical, finite envelope rather than the true (infinite) mathematical range — see each projection's own documentation for how that envelope is chosen and what it excludes.

This member has a default implementation (throwing NotSupportedException) rather than being required, so that adding it to this public interface does not break existing third-party IProjectionTransformation<T> implementations: they keep compiling and working exactly as before, and only fail — with a clear message — if they (or code that consumes them, like Normalize(ProjectedPoint<T>), MapPoint<T>'s ProjectedPoint<T> constructor, or MappointToTile(ProjectedPoint<T>, byte)) actually try to use it without overriding it. All 7 projections built into this package override it.

Normalize(ProjectedPoint<T>)

Normalizes projectedPoint into map-fraction coordinates using Bounds: (MinX, MinY) maps to (0, 0) and (MaxX, MaxY) maps to (1, 1), independently on each axis.

(T X, T Y) Normalize(ProjectedPoint<T> projectedPoint)

Parameters

projectedPoint ProjectedPoint<T>

A point produced by GeoPointToMapPoint(GeoPoint<T>) using this projection.

Returns

(T X, T Y)

The normalized (X, Y) coordinates. For points within Bounds these are in [0, 1]; a point outside Bounds (e.g. near an excluded singularity, see its remarks) normalizes outside that range rather than being clamped.

Remarks

This is a plain independent linear rescale of each axis: it does not assume that either axis corresponds to a cardinal direction (that assumption only holds for cylindrical/pseudo-cylindrical projections such as MercatorProjection<T>/EquirectangularProjection<T>, not for azimuthal ones such as LambertAzimuthalEqualArea<T>/StereographicProjection<T> where a single axis does not track latitude alone). Callers that need a specific tile-numbering orientation (e.g. row 0 = north, matching common slippy-map conventions) must apply that convention themselves on top of this normalized value.
