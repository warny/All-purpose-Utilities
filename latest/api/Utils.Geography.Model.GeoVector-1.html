Equals(GeoVector<T>?)

Determines whether the specified GeoVector<T> has the same position and bearing as this instance: latitude/longitude are compared via Equals(GeoPoint<T>?) (rounded to EqualityPrecision decimal places), and bearing is compared the same way longitude is: normalized and rounded to the same precision via AreEqualRounded(T, T, int), since bearing wraps around at 0°/360° just like longitude wraps around at the antimeridian.

public bool Equals(GeoVector<T>? other)

Parameters

other GeoVector<T>

The vector to compare with this instance.

Returns

bool

true if both vectors round to the same latitude, longitude, and bearing; otherwise false.

Remarks

This has the exact same known limitation as Equals(GeoPoint<T>?) (see its remarks for the full rationale and a worked example): rounding each coordinate before comparing keeps Equals(GeoVector<T>?) and GetHashCode() always consistent with each other, but two bearings that are extremely close yet fall on opposite sides of a rounding boundary (e.g. 1.0000449999 vs 1.0000450001, with EqualityPrecision = 5) are treated as unequal. See GeoVectorTests.VectorsOnOppositeSidesOfARoundingBoundaryAreNotEqual for a regression test pinning down this behavior.

GetHashCode()

Returns a hash code consistent with Equals(GeoVector<T>?): latitude is rounded, and longitude/bearing are normalized (to handle antimeridian/0-360° wraparound) and rounded, to EqualityPrecision decimal places before hashing — the exact same values that Equals(GeoVector<T>?) compares on.

Recenter(GeoVector<T>)

Returns a new GeoVector such that:

  • The current vector ("this") is mapped to (0°, 0°, 0°).
  • The new latitude = spherical distance (central angle) from "this" to "other".
  • The new longitude = difference in initial bearings, so that "this.Bearing" becomes 0°.
  • The new bearing = difference in heading from "this.Bearing".

Change the reference so that the current vector is effectively the new origin with heading=0.
