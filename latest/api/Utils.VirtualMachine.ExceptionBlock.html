Remarks

Typical VM usage follows conventional structured-exception semantics (catch executes before finally):

  • TRY instruction: push an ExceptionBlock with the handler addresses.
  • THROW instruction: call Throw(Context, object?); unwinds to this block, sets ThrownValue, and redirects execution to CatchAddress when present (storing FinallyAddress in PendingFinallyAddress for later), or directly to FinallyAddress when there is no catch clause. The block stays on the stack so that the handler body can inspect ThrownValue.
  • ENDCATCH instruction (catch body completed normally): call EndCatch(Context); if PendingFinallyAddress is set, jumps there for cleanup; otherwise pops the block.
  • ENDFINALLY instruction: call EndFinally(Context); if an exception is still in flight (finally-only path), propagates to the next outer handler; otherwise pops the block (normal cleanup after catch, or normal try-exit).
  • ENDTRY instruction: call Pop() to close the block on the normal (non-exception) path through a try body.

At least one of CatchAddress or FinallyAddress must be provided.

PendingFinallyAddress

When Throw(Context, object?) routes execution into a catch handler and both CatchAddress and FinallyAddress are present, this property stores the finally address so that the ENDCATCH instruction handler can jump there after the catch body completes normally. null when there is no pending finally (finally-only or catch-only scenarios, or when the block has not been thrown into yet).

public int? PendingFinallyAddress { get; }

int?

Phase

Current handler phase. Starts at Try; transitions to Catch when Throw(Context, object?) routes execution into the catch clause, and to Finally when execution enters the finally clause (either via EndCatch(Context) or directly from Throw(Context, object?) in the finally-only case). Throw(Context, object?) skips blocks whose Phase is not Try so that a throw from inside a handler propagates to the next outer handler instead of re-entering the same block.
