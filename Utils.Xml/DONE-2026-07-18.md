# Utils.Xml — Quality and security audit (2026-07-11)

> **Completed 2026-07-18.** All 26 items addressed across PRs #472 (items 4–26) and #474 (items 1–3, 7 dispatch-order fix, 19 cycle-detection). 44/44 XML tests pass.

Static review of `XmlUtils` and `XmlDataProcessor`, focused on parser safety, XPath dispatch correctness, reader state, namespace handling, resource limits, API contracts, and duplicated intent. No production code is changed by this commit.

## Critical priority

### ✅ 1. `ReadChildElements` does not track XML depth correctly

Both overloads increment/decrement their local `depth` only when an encountered start/end element has the same qualified name as the original parent. Nested elements with different names leave `depth` unchanged and are yielded as if they were immediate children.

**Risk:** grandchildren and deeper descendants are incorrectly exposed as direct children. Processing logic can silently attach data to the wrong parent.

**Fix:** capture `reader.Depth` at entry and yield only elements whose `Depth == parentDepth + 1`. Stop when the reader reaches the matching end element at `parentDepth`, independently of element names.

**Priority: P0 functional correctness.**

### ✅ 2. `ReadChildElements` mishandles empty parent elements

For `<parent />`, the iterator still calls `Read()` and has no matching end element to terminate on. It may scan and yield unrelated elements from the remainder of the document.

**Risk:** cross-scope data leakage and unpredictable reader advancement.

**Fix:** return immediately when `reader.IsEmptyElement` is true. Add an explicit reader-position contract for both normal completion and early iterator disposal.

**Priority: P0 functional correctness.**

## High priority

### ✅ 3. Yielding the same mutable `XmlReader` makes enumeration behavior consumer-dependent

Each iteration yields the original reader instance. If the consumer reads the element body, skips a subtree, calls another reader helper, or only partially consumes the iterator, it mutates the state used by the outer iterator.

**Risk:** siblings may be skipped, descendants may be reclassified, and behavior changes depending on what the loop body does.

**Fix:** use `ReadSubtree()` + `subtree.Read()` (Initial→Interactive transition) to yield an isolated sub-reader per child. Added a post-`Close()` guard for empty elements: `XmlSubtreeReader.Close()` does not advance the outer reader when in Interactive state at an empty element; `reader.Read()` is called manually to prevent re-yielding.

**Priority: P1 API contract.**

### ✅ 4. `XmlDataProcessor.Current` is not restored when a handler throws

`InvokeSingleNode` stores the previous context, assigns `Current`, invokes the handler, then restores the old value only on normal completion.

**Risk:** after an exception, the processor remains bound to the failed node. If the caller catches the exception and reuses the processor, `ValueOf`, `Apply`, and `GetNodes` operate on stale state.

**Fix:** restore `Current` in a `finally` block. Invocation exceptions are unwrapped from `TargetInvocationException` via `ExceptionDispatchInfo` to preserve the original stack trace.

**Priority: P1 correctness and lifecycle.**

### ✅ 5. Optional handler parameters are matched but not supplied

`ParametersMatch` accepts fewer arguments when the missing parameters are optional, but `MethodInfo.Invoke` receives the original shorter array. Reflection does not automatically append optional default values in this usage.

**Risk:** a handler can pass validation and then fail with `TargetParameterCountException`.

**Fix:** build the final invocation argument array in `TryBuildArguments`, filling omitted optional entries with `ParameterInfo.DefaultValue`.

**Priority: P1 functional correctness.**

### ✅ 6. Null arguments crash method selection before matching

Argument types are computed with `parameters.Select(o => o.GetType())`. A legitimate `null` argument causes `NullReferenceException`, even when the target parameter is nullable or a reference type.

**Risk:** supported handler signatures cannot receive null values, and failure diagnostics do not identify the argument or handler.

**Fix:** `TryBuildArguments` treats null as compatible with reference types and nullable value types, and incompatible with non-nullable value types.

**Priority: P1 functional correctness.**

### ✅ 7. Handler selection is order-dependent and only the first match runs

Triggers are populated from reflection into a `Dictionary<XPathExpression, Method>`. `InvokeSingleNode` iterates that dictionary and stops after the first matching XPath/signature pair. Reflection order is not a stable public contract.

**Risk:** adding or recompiling a handler can change which method executes.

**Fix:** triggers stored in `List<TriggerDescriptor>` ordered by `.OrderBy(m => m.MetadataToken)` within each class (declaration order), with derived-to-base class traversal so derived handlers take priority. Documentation corrected from "base-to-derived" to "derived-to-base".

**Priority: P1 deterministic behavior.**

### ✅ 8. Handler signatures are not validated when the processor is constructed

Methods decorated with `[Match]` may have non-void return types, async signatures, or unsupported parameter kinds. Errors occur only when a matching node is reached.

**Risk:** invalid processors appear successfully constructed and fail late on specific documents.

**Fix:** validate every handler once in the constructor: reject non-`void` return types (including `Task`/`ValueTask`) with a descriptive exception naming the method and XPath.

**Priority: P1 robustness.**

### ✅ 9. `Read(Stream)` materializes untrusted XML without hardened settings or a size limit

`Read(Stream)` constructs `XPathDocument` directly from the stream without DTD/entity/size hardening.

**Risk:** callers naturally treat a stream overload as safe for uploaded or network data.

**Fix:** added `ReadSecure(Stream)` with `DtdProcessing.Prohibit`, null `XmlResolver`, `MaxCharactersFromEntities`, and `MaxCharactersInDocument`. `Read(Stream)` documented as trusted/caller-policy overload.

**Priority: P1 security and resource limits.**

### ✅ 10. `ReadSecure(string)` accepts unrestricted URI schemes and performs I/O internally

The method is described as safe for untrusted XML, but the input is a URI passed directly to `XmlReader.Create`.

**Risk:** SSRF/local-file access when the URI itself is user-controlled.

**Fix:** documented explicitly in XML doc remarks that `ReadSecure(string)` hardens the XML parser only — URI scheme, redirect, and host policy remain the caller's responsibility. Callers handling untrusted URIs should acquire the stream themselves and use `ReadSecure(Stream)`.

**Priority: P1 security.**

### ✅ 11. Configured XPath expressions can consume unbounded CPU

`Apply(string)` and `GetNodes(string)` compile/evaluate arbitrary expressions repeatedly. Nested handlers can produce quadratic work.

**Risk:** derived processors that compose XPath from external input can suffer XPath injection or high CPU usage.

**Fix:** added `_dispatchDepth` / `MaxDispatchDepth = 20` cycle-detection guard in `InvokeSingleNode`; throws `InvalidOperationException` on runaway redispatch. Dynamic XPath injection discouraged in docs; `Apply(string)` validates non-whitespace input.

**Priority: P1 performance and security.**

## Medium priority

### ✅ 12. `GetXPath` does not always identify one unique node

An index is added only when preceding siblings with the same name exist. For the first of several same-name siblings, the generated path selects all matching siblings.

**Fix:** include `[n]` whenever any same-name sibling exists either before or after the element, so the first sibling also gets `[1]`.

**Priority: P2 correctness.**

### ✅ 13. `GetXPath` produces namespace-dependent paths without returning a namespace context

Paths use `XmlNode.Name`, which may contain document-specific prefixes.

**Fix:** documented explicitly in XML doc remarks that returned paths use document-specific prefixes and require the same `XmlNamespaceManager` mappings for re-evaluation.

**Priority: P2 API correctness.**

### ✅ 14. `GetXPath(XmlNode)` silently maps unsupported node kinds to their parent element

For text, CDATA, comment, and other non-element nodes, the method returned the parent element path.

**Fix:** unsupported node types now throw `NotSupportedException` with a descriptive message.

**Priority: P2 contract clarity.**

### ✅ 15. Namespace attribute validation does not match XML namespace rules

The prefix regex rejected valid `_`-prefixed NCNames; reserved prefixes and empty URIs were not validated.

**Fix:** use `XmlConvert.VerifyNCName`, reject reserved prefixes (`xml`, `xmlns`), validate non-empty URI, and detect conflicting duplicate prefix declarations with a clear exception.

**Priority: P2 validation.**

### ✅ 16. The alleged cached invoker still performs reflection for every call

`Method` stored an `Action` that merely called `MethodInfo.Invoke`, adding a wrapping layer without delivering compiled performance.

**Fix:** removed the redundant wrapper; `InvokeSingleNode` invokes `MethodInfo` directly through one centralized path with `TryBuildArguments`.

**Priority: P2 maintainability and performance.**

### ✅ 17. Nullability contracts are inconsistent with the implementation

`Current`, `ValueOf`, optional parameter arrays, and several method arguments were declared non-nullable while being null before processing or returning null on no match.

**Fix:** `Current` declared `XPathNavigator?`, `ValueOf` returns `string?`, parameter arrays use `object?[]`, public API boundaries have `ArgumentNullException.ThrowIfNull`.

**Priority: P2 API quality.**

## Second-pass findings

### ✅ 18. A processor instance is not safe for concurrent or overlapping `Read` operations

`Current` is a single mutable instance property. Two concurrent `Read` calls can overwrite each other's node.

**Fix:** `_activeOperations` Interlocked counter in `BeginRead`/`EndRead`; second concurrent call throws `InvalidOperationException` immediately.

**Priority: P1 correctness and thread safety.**

### ✅ 19. Recursive redispatch can create unbounded cycles

A handler may call `Apply(".")` and dispatch a node that matches the same handler again, causing stack overflow.

**Fix:** `_dispatchDepth` Interlocked counter, `MaxDispatchDepth = 20`; exceeding the limit throws `InvalidOperationException` before the recursive call.

**Priority: P1 robustness and resource limits.**

### ✅ 20. Asynchronous handlers are invoked but never awaited

A method returning `Task` or `ValueTask` was accepted and invoked through reflection, but processing continued immediately.

**Risk:** `Read` may return before processing is complete.

**Fix:** constructor rejects every non-`void` handler (including async) with a descriptive `InvalidOperationException`.

**Priority: P1 lifecycle correctness.**

### ✅ 21. Caller-provided `XmlReader` security settings are opaque and cannot be enforced

`Read(XmlReader)` wraps whatever reader it receives without inspecting its settings.

**Fix:** documented in XML doc remarks as a trusted/caller-policy overload; not equivalent to `ReadSecure(Stream)` unless the caller applied equivalent settings.

**Priority: P1 security contract.**

### ✅ 22. URI reads have no acquisition timeout, cancellation, or response-size policy before parsing

`ReadSecure(string)` delegates URI opening to `XmlReader.Create` with no connection timeout or cancellation.

**Fix:** documented explicitly in XML doc remarks as a known limitation; callers requiring timeout/redirect control should acquire the stream with an `HttpClient` and use `ReadSecure(Stream)`.

**Priority: P1 availability and SSRF hardening.**

### ✅ 23. Inherited private handlers are silently omitted

Private methods declared on base classes were not returned by `GetMethods(Public | NonPublic | Instance)`.

**Fix:** trigger discovery walks the type hierarchy with `BindingFlags.DeclaredOnly`, collecting private base-class handlers intentionally.

**Priority: P2 deterministic configuration.**

### ✅ 24. Duplicate or conflicting namespace declarations have order-dependent diagnostics

Namespace attributes were immediately passed to `XmlNamespaceManager.AddNamespace` without normalization.

**Fix:** declarations collected first, exact duplicates silently accepted, conflicting URIs for the same prefix throw `InvalidOperationException` naming both URIs.

**Priority: P2 configuration quality.**

### ✅ 25. `MatchAttribute` accepts null, empty, and whitespace XPath expressions until processor construction

**Fix:** `MatchAttribute` constructor validates null/empty/whitespace and throws `ArgumentException` immediately.

**Priority: P2 diagnostics.**

### ✅ 26. Public read and selection methods do not consistently validate null arguments or processor state

**Fix:** `ArgumentNullException.ThrowIfNull` at all public API boundaries; `ArgumentException.ThrowIfNullOrWhiteSpace` on XPath string inputs.

**Priority: P2 API contract.**

## Duplicated intent — resolved

- ✅ Two `ReadChildElements` overloads consolidated into one with an optional name filter.
- ✅ Five `Apply`/`GetNodes` variants share centralized `SelectNodes`/`InvokeTriggersOnNodes`/`TryBuildArguments`.
- ✅ Parser security centralized in `CreateSecureReaderSettings()`; all raw-source overloads documented on trust boundary.
- ✅ Handler matching, argument construction, invocation, exception unwrapping, context restoration, and recursion control consolidated in `InvokeSingleNode`.
- ✅ XPath path generation centralized in `GetElementXPath` / `GetAttributeXPath`.
- ✅ `ReadSecure(string)` limitations documented; stream-based path is the canonical secure entry point.
