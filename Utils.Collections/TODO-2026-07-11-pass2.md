# Utils.Collections — Second audit pass (2026-07-11)

Follow-up review of the deterministic adaptive skip structure and `SkipListDictionary<K,V>`. This pass treats lookup-driven promotion and deterministic level construction as intentional design choices. It focuses on the remaining correctness, concurrency, observability, performance, and API-contract issues. No production code is changed.

## High-priority findings

### 15. Concurrent readers can race while promoting the same upper-level links

Lookup operations intentionally optimize the structure by calling `CreateNewSkipNode`. Two readers traversing the same sparse region can both observe `currentElement.Up == null` and attempt to create/link an upper node concurrently.

`CreateUp` performs several independent writes (`Sub.Up`, `Previous.Next`, `Next.Previous`, and the new node links) without locking or compare-and-swap coordination. Interleaving these writes can lose a promotion, create inconsistent back-links, or detach part of an upper level.

**Fix:** explicitly document the type as not safe even for concurrent reads, or synchronize adaptive maintenance. A useful design is a pure lock-free/read-only traversal plus an optional best-effort promotion guarded by a narrow lock or atomic publication protocol.

**Priority: P1 structural integrity under concurrency.**

### 16. Structural and content mutations need different enumeration-version semantics

Lookup-driven promotion changes only upper index levels; bottom-level enumeration remains logically unchanged. It therefore does not necessarily need to invalidate an enumerator. In contrast, `Add`, `Remove`, and `Clear` rewrite the bottom chain and can silently alter an active enumeration.

The current implementation has no version counter, so a bottom-level mutation may cause an enumerator to skip nodes, include new nodes, or continue through detached links without any diagnostic.

**Fix:** maintain a content version incremented only for changes that affect the enumerated sequence. Promotions may use a separate structural/diagnostic version if needed. A dedicated enumerator should fail fast when the captured content version changes.

**Priority: P1 collection correctness.**

### 17. Adaptive promotion is not transactionally isolated from comparer failures

Within `FindElementPositionAtLevel`, promotion may occur before `comparer.Compare(value, currentElement.Value)` is called for the current node. If a user comparer throws, the lookup fails but may already have modified upper levels.

The resulting structure should remain valid, but the public operation has a hidden partial side effect and repeated failing lookups can continue changing memory/layout.

**Fix:** either document best-effort maintenance despite comparer exceptions, or defer promotion until after all comparison work needed for that traversal segment has succeeded. Add invariant tests with comparers that throw at controlled calls.

**Priority: P1 exception safety and predictability.**

### 18. No invariant checker exists for the custom adaptive algorithm

The data structure depends on coupled invariants across four links per node and two boundary towers:

- every `Next`/`Previous` pair must be reciprocal;
- every `Up`/`Sub` pair must be reciprocal;
- each level must remain sorted;
- upper nodes must reference live lower nodes with the same value;
- `_firstElement` and `_lastElement` must be the boundaries of the same top level;
- bottom-level node count must equal `Count`.

The current tests validate externally visible order but cannot detect orphaned, duplicated, cyclic, or one-way upper links until a later operation fails.

**Fix:** add an internal invariant validator exposed to tests through `InternalsVisibleTo`, and run it after every operation in model/property tests.

**Priority: P1 maintainability of a custom data structure.**

### 19. Removal correctness is tested only through final values, not through tower repair

Removing a first or last item rebuilds boundary towers with `CreateUp`, recursively removes the old tower, and may collapse top levels. Existing tests confirm the resulting bottom sequence, but not that every recreated boundary link and collapsed level remains reciprocal and reachable.

**Risk:** stale upper nodes can remain undetected while basic enumeration still succeeds.

**Fix:** combine boundary-removal scenarios with the invariant checker. Cover repeated alternating first/last removals, removals immediately after lookup-driven promotions, and reinsertions across multiple levels.

**Priority: P1 structural correctness.**

### 20. Duplicate comparer-equal elements make adaptive-index identity unstable

Duplicates are valid under the current multiset-like behavior, but upper levels contain copied nodes corresponding to particular bottom instances. As promotions and removals evolve the towers, `TryGet` may return different comparer-equal stored instances over time.

This is not inherently wrong, but it matters because `TryGet` is explicitly advertised as a way to retrieve the stored full object when the comparer uses only a subset of fields.

**Fix:** define whether `TryGet` returns the first, last, earliest inserted, or any comparer-equal instance. If stable identity matters, store duplicates in a bucket per key or add APIs such as `TryGetFirst`/`GetEqualRange`.

**Priority: P1 semantic correctness.**

## Medium-priority findings

### 21. Threshold behavior is off by one relative to its “maximum distance” wording

The promotion condition is `counter > _threshold`. Since the counter starts at zero and increments after each processed node, the traversal exceeds the configured threshold before promotion occurs. The effective spacing is therefore larger than a reader would infer from “maximum number of nodes to traverse”.

**Fix:** formally define whether threshold means maximum gap, number of unindexed interior nodes, or trigger-after count. Rename it if necessary and write exact shape tests for thresholds 2, 3, and 10.

**Priority: P2 configuration contract.**

### 22. Dictionary insertion performs redundant adaptive searches

`SkipListDictionary.Add` first calls `_skipList.Contains(probe)` and then `_skipList.Add(...)`, which searches again. Both traversals may perform adaptive maintenance.

**Risk:** duplicate work, avoidable comparer calls, and a larger mutation surface for one logical insertion.

**Fix:** expose a single skip-list insertion primitive returning whether an equal item already exists, or an insertion-position token valid for the current content version.

**Priority: P2 performance and intent duplication.**

### 23. The configured comparer is not exposed by either public collection

Unlike `SortedSet<T>` and `SortedDictionary<TKey,TValue>`, callers cannot retrieve the comparer that defines ordering and key identity.

**Risk:** wrapper/copy code cannot preserve semantics reliably, and duplicate/key behavior is harder to inspect or document at runtime.

**Fix:** expose read-only `Comparer`/`KeyComparer` properties.

**Priority: P2 API completeness.**

### 24. Nullable-reference analysis is not enabled for the package

The public dictionary uses `where K : notnull` and `[MaybeNullWhen(false)]`, but the project does not enable nullable-reference analysis. Internally, boundary and link fields are declared non-nullable even though null is a core sentinel state.

**Risk:** compiler assistance is lost precisely in code whose correctness depends heavily on nullable links and boundaries.

**Fix:** enable nullable analysis, annotate links and empty-state fields as nullable, and remove null-forgiving operators where possible. Treat the resulting warnings as an invariant-review aid.

**Priority: P2 static correctness.**

### 25. Performance claims are not backed by repeatable benchmarks

The deterministic adaptive design was chosen to reduce performance variance and improve real workloads, but the package contains tests rather than benchmarks demonstrating lookup warm-up, steady-state cost, memory overhead, insertion-order sensitivity, or comparison with probabilistic alternatives and framework collections.

**Fix:** add BenchmarkDotNet scenarios for cold/hot lookups, successful/failed searches, ascending/descending/random insertion, duplicate-heavy data, multiple thresholds, and read/write mixes. Record variance as well as mean throughput.

**Priority: P2 performance evidence.**

## Clarifications from the first pass

- Deterministic construction is an intentional performance design, not a defect.
- Lookup-driven promotion is an intentional adaptive optimization, not inherently a violation of collection semantics.
- Mutating comparer-relevant state is the normal documented restriction of sorted/indexed collections, not a defect specific to this implementation.
- The remaining concerns are truthful documentation, structural invariants, concurrent access, enumeration during content mutation, and measurable guarantees.

## Additional duplicated intent to reduce

- Search, duplicate detection, and insertion-position discovery should share one traversal result rather than repeating adaptive searches.
- Bottom-content versioning and upper-index maintenance tracking should be modeled separately.
- Boundary-tower reconstruction and generic node removal need one verified repair primitive.
- Comparer semantics should be exposed consistently by list and dictionary wrappers.
- Structural diagnostics and benchmarks should use the same measurable notions of level count, gap size, comparison count, and promotion count.

## Required tests

- Run many concurrent successful and failed readers over promotion-triggering regions; verify all structural invariants afterward.
- Use barriers to force two readers to attempt promotion of the same node simultaneously.
- Enumerate while performing lookup-only promotions and verify enumeration remains logically stable under the documented single-threaded policy.
- Enumerate while adding/removing/clearing and verify the chosen fail-fast behavior.
- Use a comparer that throws on a specified comparison count and validate structure after every failure point.
- Validate reciprocal horizontal/vertical links, sorted levels, boundary reachability, absence of cycles, and bottom count after every operation.
- Alternate first/last removals after creating several levels, then reinsert outside both boundaries.
- Define and test stable or explicitly arbitrary `TryGet` identity for comparer-equal duplicates.
- Verify exact promotion positions for small threshold values.
- Measure comparer-call counts for dictionary add/update/lookup paths.
- Benchmark cold versus warmed adaptive structures and report variance.

## Deployment warning

Treat `SkipList<T>` and `SkipListDictionary<K,V>` as strictly single-threaded until adaptive promotion is synchronized or a stronger concurrent-read protocol is implemented. Lookup-driven optimization is intentional and can coexist with bottom-level enumeration, but concurrent promotions and content mutation during enumeration currently have no safety contract.
