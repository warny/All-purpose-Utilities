# Utils.Reflection — Additional audit, passes 3 and 4 (2026-07-11)

This file complements `TODO.md` and `TODO-2026-07-11.md`. Pass 3 covers Windows/Linux/macOS isolation guarantees, sandbox executable discovery, worker environment filtering, Authenticode policy, and native-resource lifecycle. Pass 4 covers the general reflection helpers and delegate dispatch APIs that share the `Utils.Reflection` namespace. No finding below is fixed by this document update.

## Critical priority

### 52. Windows Job Object guarantees are claimed without checking native configuration results

`AppContainerSandbox.CreateConfiguredJobObject` calls `SetInformationJobObject` for `KillOnJobClose` and UI restrictions but ignores both return values.

**Risk:** the worker may be assigned to a Job Object whose lifecycle or UI restrictions were never configured, while comments and documentation present them as guaranteed.

**Fix:** verify every native return value. On failure, capture the Win32 error, close the Job Object, and fail sandbox creation. Report lifecycle-limit and UI-restriction failures separately.

**Priority: P0.**

## High priority

### 53. `AppContainerSandbox` remains callable after its native handles are released

`Dispose` frees the AppContainer SID and closes the Job Object, but `StartProcess`, `GetContainerSid`, `TryGetSecurityIdentifier`, and `GrantDirectoryReadAccess` do not check the disposed state.

**Risk:** use-after-free of a SID or closed/reused native handle.

**Fix:** throw `ObjectDisposedException` from every resource-dependent public member and replace raw `IntPtr` ownership with suitable `SafeHandle` types.

**Priority: P1 — native safety.**

### 54. The minimal environment allows every `DOTNET_` and `CORECLR_` variable

The prefix allowlist includes runtime variables that can enable startup hooks, profilers, diagnostics, additional dependency paths, and other hosting behavior.

**Risk:** the worker may load additional code before `RunWorkerIfRequested`, expose diagnostics, or inherit sensitive/non-reproducible host configuration.

**Fix:** replace prefix matching with a minimal named allowlist. Explicitly reject profiling, startup-hook, diagnostic-suspend, additional-deps, and similar injection-capable settings. Allow extensions only through an explicit option.

**Priority: P1.**

### 55. The Linux worker receives read/write access to the host's complete `/tmp`

The Emit worker needs access to one Unix-domain socket, but `AllowDiskWrite` becomes `--bind /tmp /tmp` in bubblewrap.

**Risk:** hostile worker code can inspect, modify, delete, or replace all user-accessible temporary files and interfere with other processes.

**Fix:** create a private per-worker IPC directory with restrictive permissions and bind only that path. Prefer structured path grants over global disk booleans.

**Priority: P1.**

### 56. `bwrap` and `sandbox-exec` are checked and then executed through two separate `PATH` resolutions

`CommandAvailability.Exists` confirms that a candidate file exists, but container launch later uses the bare command name again. Executability, ownership, and permissions are not validated.

**Risk:** time-of-check/time-of-use replacement, `PATH` manipulation, or execution of a counterfeit sandbox tool.

**Fix:** resolve and retain one canonical absolute executable path, validate it according to platform policy, and execute exactly that path.

**Priority: P1.**

### 57. Authenticode validation explicitly disables revocation checks

`WINTRUST_DATA.fdwRevocationChecks` is set to `WTD_REVOKE_NONE`, and no equivalent revocation policy is requested through provider flags.

**Risk:** a binary signed with a compromised or revoked certificate may still pass an API described as “valid and trusted.”

**Fix:** define explicit online and cache-only revocation policies. Return a structured result separating cryptographic validity, chain trust, and revocation status instead of one ambiguous boolean.

**Priority: P1 — trust validation.**

### 59. `ReflectionEx.LoadAssemblies` uses `Assembly.Load` with file paths and probes every file

The method enumerates all files and passes each full path to `Assembly.Load(string)`, whose string overload expects an assembly display name rather than a filesystem path. All exceptions are optionally swallowed.

**Risk:** valid assemblies may fail to load, unrelated files are needlessly probed, and failures disappear without diagnostics. The method name suggests deterministic directory loading that the implementation does not provide.

**Fix:** restrict candidates to managed assembly extensions, use an explicit `AssemblyLoadContext`/`LoadFromAssemblyPath` policy with canonical paths, and return or log per-file failures. Define duplicate identity and unload behavior.

**Priority: P1 — functional and loading-context issue.**

### 60. `IsAssignableFromEx` treats numeric storage size as assignment compatibility

For numeric types, the method allows any assignment to a floating-point destination and otherwise compares only `Marshal.SizeOf`.

**Risk:** equal width does not imply safe conversion (`uint` to `int`, `long` to `ulong`), floating-point conversion can lose range/precision, and decimal/native-layout behavior does not model C# implicit conversion rules. Callers may accept values that later overflow or change meaning.

**Fix:** rename the method to reflect a deliberately lossy conversion policy, or implement a table based on C# implicit numeric conversions plus nullable/enum handling. Keep storage-size compatibility as a separate API.

**Priority: P1 — semantic correctness.**

## Medium priority

### 58. `GrantDirectoryReadAccess` creates directories and continues after ACL failure

The ACL operation creates a missing path and converts several failures into a `TraceWarning`.

**Risk:** unintended filesystem side effects and no reliable way for a security-sensitive caller to know whether access was actually granted.

**Fix:** separate directory creation from ACL assignment and expose a required/best-effort policy or structured result.

**Priority: P2.**

### 61. `GetDirectInterfaces` returns no inherited interfaces for an interface type

The implementation returns an empty sequence whenever `BaseType` is `null`. Interfaces always have a null base type, even when they inherit other interfaces.

**Risk:** interface hierarchy inspection silently omits all direct parent interfaces and makes `GetTypeHierarchy` inconsistent between classes and interfaces.

**Fix:** for interface types, subtract interfaces inherited through each candidate parent from `type.GetInterfaces()` to retain only direct parents. Define deterministic ordering.

**Priority: P2 — functional bug.**

### 62. Assembly-wide type enumeration is all-or-nothing

`ReflectionEx.GetTypes` calls `Assembly.GetTypes()` directly for every loaded assembly. One `ReflectionTypeLoadException` aborts the complete enumeration.

**Risk:** optional or partially loadable assemblies prevent discovery in every other valid assembly.

**Fix:** provide strict and tolerant variants. The tolerant path should return non-null `ReflectionTypeLoadException.Types` entries and expose loader exceptions to diagnostics instead of discarding them.

**Priority: P2.**

### 63. Property/field lookup silently selects the first ambiguous member

`GetPropertyOrField` calls `GetMember(...).FirstOrDefault()` after filtering. Hidden members, different binding scopes, or unusual metadata can yield more than one candidate.

**Risk:** result depends on reflection enumeration order rather than an explicit resolution policy.

**Fix:** return a nullable type accurately, reject ambiguity by default, and offer an explicit declaring-type/inheritance preference when needed.

**Priority: P2.**

### 64. Generic static-method helpers validate the wrong type state

`GetStaticMethods` and `GetStaticMethod` check `IsGenericType`, then call `MakeGenericType`. A closed constructed generic type passes the check but cannot be constructed again; arity and null arguments are not validated explicitly.

**Fix:** require `IsGenericTypeDefinition`, validate generic argument count/constraints before construction, and distinguish “method not found” from ambiguous overloads.

**Priority: P2.**

### 65. `DelegateInvoker` ignores interface registrations and is not concurrency-safe

Dispatch walks only the runtime class `BaseType` chain. A delegate registered for an implemented interface is never selected. Concurrent mutation and invocation use a regular `Dictionary` without synchronization.

**Fix:** define precedence across exact type, base classes, and interfaces; detect ambiguous interface matches; use immutable snapshots or synchronization for concurrent access.

**Priority: P2.**

### 66. `ExtendedInvoker` has unstable null and ambiguity semantics

Null arguments are represented as `typeof(object)`, so overloads accepting nullable reference types may not match as expected. Equal-distance candidates are resolved by registration order, and `DynamicInvoke` wraps target exceptions in `TargetInvocationException`.

**Fix:** model null as an unknown/reference-compatible argument, detect and report equal-best ambiguity, document or unwrap invocation exceptions consistently, and cache compiled invokers for repeated calls.

**Priority: P2.**

### 67. The `Utils.Reflection` namespace is split across two assemblies/packages

General helpers such as `ReflectionEx`, `TypeEx`, `DelegateInvoker`, and `ExtendedInvoker` live under `Utils/Reflection` and are compiled into the main `Utils` project, while the NuGet package `omy.Utils.Reflection` contains the native mapping and process-isolation implementation. Both expose the same namespace.

**Risk:** consumers can import `Utils.Reflection` yet still miss expected types depending on which package they referenced. Package description and namespace ownership are misleading, and future type-name collisions are possible.

**Fix:** choose one package boundary. Move/link the general reflection helpers into `Utils.Reflection`, or rename the native-mapping namespace/package to a narrower domain. Add package-level API tests ensuring documented types are actually shipped.

**Priority: P2 — architecture and packaging.**

## Duplicated intent to consolidate

- Resolve sandbox executables once and carry the validated absolute path through launch.
- Use `SafeHandle` consistently for native Windows resources.
- Replace broad disk booleans with explicit path grants.
- Return structured signature-verification results.
- Centralize type discovery with strict/tolerant loading policies and diagnostics.
- Define one reusable member-resolution policy for properties, fields, and methods.
- Consolidate delegate dispatch around a documented specificity/ambiguity algorithm.
- Align namespace ownership with the actual NuGet package boundary.

## Required tests

- Simulate failure of both `SetInformationJobObject` calls and verify immediate handle cleanup.
- Invoke every resource-dependent `AppContainerSandbox` member after disposal.
- Test startup-hook, profiler, diagnostic, and explicitly allowed environment variables.
- Verify that a Linux worker sees only its private IPC directory.
- Change `PATH` or replace a sandbox executable after resolution and verify the validated path remains fixed.
- Test valid, expired, untrusted, and revoked Authenticode chains under online and cache-only policies.
- Verify directory ACL grants have no implicit creation side effect.
- Load valid DLLs, native DLLs, text files, duplicate identities, and partially loadable assemblies from a test directory.
- Exhaustively compare `IsAssignableFromEx` decisions with the intended C# numeric-conversion policy.
- Test direct interface inheritance, `ReflectionTypeLoadException`, hidden members, closed/open generic types, and generic-arity mismatches.
- Test exact/base/interface delegate dispatch, ambiguous interfaces, null overloads, equal-distance candidates, target exceptions, and concurrent registration/invocation.
- Build/package `omy.Utils.Reflection` and assert every documented public type is present in the produced assembly.

## Deployment warning

Until items 52–57 are fixed, process containers do not provide homogeneous guarantees across platforms. Until items 59–60 are fixed, do not rely on directory assembly loading or `IsAssignableFromEx` as authoritative correctness/security gates. The shared `Utils.Reflection` namespace also currently spans two separate packages.
