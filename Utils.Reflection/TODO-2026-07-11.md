# Utils.Reflection — Additional audit (2026-07-11)

This file contains the third and fourth static-review findings for `Utils.Reflection`. It complements the completed historical log in `TODO.md`. These findings are not fixed by this document update.

## Critical priority

### 37. The safe-by-default path can silently fall back to an unsandboxed worker

`EmitWorkerProcess.StartWorkerProcess` catches any sandbox launch failure, disposes the container, and starts the same worker as a regular child process. The caller receives the same proxy without any indication that isolation is no longer active.

**Risk:** a sandbox configuration failure silently turns a security boundary into full-user execution.

**Fix:** fail closed by default. Introduce an explicit fallback policy and expose the effective isolation level in diagnostics. Unsandboxed fallback must require explicit acknowledgement.

**Priority: P0.**

### 38. Worker method tokens are not restricted to the loaded interface

The worker resolves `MethodMetadataToken` directly through `InterfaceType.Module.ResolveMethod` and invokes the result without proving that it belongs to the loaded interface contract.

**Risk:** a client able to write to the pipe may invoke another method from the same module, including a static method. This is especially serious on Unix where pipe-peer identity is not verified.

**Fix:** build an immutable logical method-ID table from the permitted interface methods at load time. Never accept arbitrary metadata tokens from the transport.

**Priority: P0.**

## High priority

### 39. Protocol JSON lines and payloads have no size limits

Requests and responses are framed with `ReadLine`/`WriteLine` and fully materialized in memory. No limit exists for line length, arrays, arguments, return values, exception text, or effective JSON depth.

**Risk:** unbounded memory consumption and denial of service in either process.

**Fix:** use length-prefixed bounded framing or a bounded reader. Add configurable request, response, argument, return-value, and JSON-depth limits.

**Priority: P1.**

### 40. The worker creates one `Task.Run` per request without bounded concurrency

Every valid non-shutdown request is immediately dispatched and retained in a `ConcurrentBag<Task>`.

**Risk:** a fast producer can exhaust the thread pool, memory, and native resources. Timed-out calls remain active and can accumulate.

**Fix:** use a bounded queue and configurable worker/per-handle concurrency limits with deterministic backpressure or rejection.

**Priority: P1.**

### 41. A timeout does not cancel remote work and the worker remains reusable

After timeout, the host removes the pending request, but the native call continues. Its eventual response is discarded.

**Risk:** late side effects, unknown execution outcome, and progressive worker exhaustion after repeated timeouts.

**Fix:** document the result as indeterminate after timeout and provide a configurable policy: keep, terminate, recycle, or retire the worker after an abandoned-call threshold.

**Priority: P1.**

### 42. Struct validation does not match the actual `System.Text.Json` contract

Validation inspects public and non-public fields, while `IncludeFields` serializes public fields only; public properties may be serialized without being validated.

**Risk:** accepted structs can silently lose state or expose an unvalidated serialized member type.

**Fix:** define an explicit transport DTO contract and validate the exact members selected by `JsonTypeInfo`. Add real round-trip tests for readonly, explicit-layout, private-field, public-field, and public-property structs.

**Priority: P1.**

### 45. `EmitRoundRobin` does not guarantee non-concurrent access to one native-library instance

The proxy selects the next worker without checking whether it is already busy. Once concurrent calls exceed `workerCount`, a worker can receive another call while its previous call is still active.

**Risk:** a library selected specifically because it is not thread-safe can still be invoked concurrently inside the same process and native instance.

**Fix:** serialize calls per worker or allocate only from a queue of idle workers. Otherwise correct the API documentation and stop promising this guarantee.

**Priority: P1 — functional and native-safety issue.**

### 46. Mapped delegates remain callable after the native library is unloaded

`LibraryMapper.Dispose` calls `NativeLibrary.Free`, but mapped delegate fields and properties remain populated. External copies can also outlive the mapper.

**Risk:** calling a stale function pointer may crash the process with no reliable managed exception.

**Fix:** introduce explicit disposed-state guards, clear mapped members, and avoid exposing raw delegates that can outlive the library lease.

**Priority: P1.**

### 47. Native library loading and member mapping are not transactional

The DLL is loaded first, then members are populated one by one. A missing export, invalid member, or failed assignment can leave a partially initialized object and a handle released only by finalization.

**Risk:** leaked native handles and partially mapped instances.

**Fix:** resolve and construct every delegate in a preparation phase, commit assignments only after complete validation, and free the library immediately on any failure.

**Priority: P1.**

### 48. `EmitWorkerPool.Emit` is not atomic with respect to `Dispose`

The pool lock protects only obtaining the worker. Interface loading and proxy creation occur after the lock is released.

**Risk:** a proxy can be returned for a dead worker, or a loaded handle can be orphaned during concurrent pool disposal.

**Fix:** use an operation lease/state machine or retain synchronization until the handle has a fully attached owner. Define whether disposal waits for or rejects in-progress emissions.

**Priority: P1 — concurrency.**

### 49. Proxies can be disposed while an invocation is active

The exclusive, pooled, and round-robin proxies do not coordinate invocation leases with disposal.

**Risk:** a handle can be unloaded or a process killed while another thread is still executing through it.

**Fix:** block new calls once disposal starts, wait for active leases before unload/kill, and expose a separate forced-close policy when required.

**Priority: P1 — concurrency.**

## Medium priority

### 43. Malformed JSON messages are ignored instead of terminating the protocol

Both ends swallow `JsonException` and continue. The originating request then waits until timeout while the connection remains nominally usable.

**Fix:** treat framing/deserialization corruption as fatal, fail all pending requests, and terminate the worker. A tolerant mode would need an explicit correlated error and a very small error quota.

**Priority: P2.**

### 50. Late proxy-creation failures can orphan handles or workers

`EmitWorkerPool.Emit` loads a handle before `DispatchProxy.Create`; `EmitRoundRobin` creates all workers before proxy construction. Neither path compensates for a failure during proxy creation/attachment.

**Fix:** wrap creation and attachment in `try/catch`; unload the pool handle or dispose all created workers on failure.

**Priority: P2.**

### 51. Public timeout documentation contradicts current behavior

`LibraryMapper.Emit` still states that a timed-out worker is killed and the proxy becomes unusable. The implementation now abandons only the request and keeps the worker alive.

**Risk:** callers make incorrect idempotency and recovery decisions.

**Fix:** align XML documentation, README text, exceptions, and tests with the real “remote execution may still be running” semantics.

**Priority: P2 — contractual documentation.**

## Low priority

### 44. Request IDs can be reused after integer overflow

`nextId` is an `int` incremented with `Interlocked.Increment`.

**Fix:** use `long`, or allocate IDs with collision checks against `pending`.

**Priority: P3.**

## Duplicated intent to consolidate

- Centralize sandbox selection, fallback policy, and effective isolation reporting.
- Build one immutable interface contract used by validation, generated code, transport, and invocation.
- Group time, volume, and concurrency limits in a shared `EmitWorkerLimits` configuration.
- Use one lease/closure abstraction for calls, handles, proxies, pools, and round-robin workers.
- Split native export discovery, delegate preparation, validation, and transactional commit.

## Required tests

- Force sandbox launch failure and verify fail-closed default behavior.
- Attempt to call a same-module method outside the loaded interface.
- Fuzz oversized/deep JSON and large arrays/responses.
- Exceed concurrency limits and verify bounded behavior.
- Chain timed-out calls and verify retirement/recycling policy.
- Round-trip all supported and rejected struct shapes.
- Verify malformed JSON immediately terminates the connection.
- Exercise request IDs near overflow.
- Run more simultaneous calls than round-robin workers and prove one-call-per-worker when requested.
- Dispose mapped libraries and proxies while calls are active.
- Force every post-load and post-proxy-creation failure and verify full cleanup.

## Deployment warning

Until items 37, 38, 45, and 46 are fixed, do not treat `LibraryMapper.Emit<TInterface>` or `EmitRoundRobin<TInterface>` as an absolute security or concurrency boundary. On Linux and macOS, this warning is stronger because pipe-client identity is not currently verified.
