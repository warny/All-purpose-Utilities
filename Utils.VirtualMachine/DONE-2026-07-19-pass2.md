## High priority

### ✅ 17. LEB128 readers accept overlong and overflowing encodings

`ReadULEB128` and `ReadSLEB128` keep consuming bytes until a byte without the continuation bit appears. They do not enforce the maximum encoded length for a 64-bit value, reject non-canonical encodings, or check whether high payload bits exceed the target width.

After the shift reaches 64, C# masks integral shift counts, so further `<< shift` operations wrap modulo 64 instead of failing. An overlong encoding can therefore overwrite low-order bits and return an incorrect value rather than being rejected.

**Risk:** malformed bytecode can decode to attacker-controlled incorrect values, consume an arbitrarily long prefix of the instruction stream, or defer failure until end-of-input.

**Fix:** limit 64-bit LEB128 to at most ten bytes, validate the tenth-byte payload/sign-extension constraints, reject overlong/non-canonical forms according to an explicit policy, and throw a dedicated malformed-operand exception.

**Priority: P1 numeric correctness and resource bounds.**

### ✅ 18. Failed byte reads advance the instruction pointer

Both endian readers implement `ReadByte` and `ReadSByte` using `context.Data.Span[context.InstructionPointer++]`. The post-increment side effect occurs while evaluating the index. If the index is outside the span, the read throws after the instruction pointer has already advanced.

Fixed-width readers increment the pointer only after `MemoryMarshal.Read` succeeds, so failure semantics differ by operand type. LEB128 truncation inherits the byte-reader behavior and may leave the pointer one byte beyond the actual end.

**Risk:** retry, diagnostics, exception handlers, and inspectors observe a mutated pointer after a failed read. Malformed instructions are not transactional and may be impossible to re-decode from the original address.

**Fix:** validate availability first, then advance the pointer only after a successful read. Prefer one central checked read primitive and make every fixed-width/LEB128 reader use it.

**Priority: P1 execution-state integrity.**

### ✅ 19. Operand decoding is not transactional across multi-operand instructions

Expression-compiled handlers read operands sequentially from the live context before invoking the instruction method. If a later operand is truncated or malformed, earlier reads have already advanced `InstructionPointer`.

**Risk:** a failed instruction leaves the context after a partially consumed operand sequence. Recovery code cannot reliably identify or retry the original instruction, and diagnostics may report an address different from the true failure point.

**Fix:** decode against a local cursor or temporary reader state, validate all operands, then commit the final instruction pointer immediately before invoking the handler. Alternatively, restore the original pointer in the dispatch exception path and document that failed instructions are atomic.

**Priority: P1 malformed-bytecode handling.**

### ✅ 20. Cancellation cannot interrupt a blocking or non-terminating instruction handler

`VirtualProcessor.Execute`, `Scheduler.Run`, and `RunAsync` check cancellation only between instructions or scheduling rounds. Instruction delegates are synchronous and receive no cancellation token. A handler that blocks, loops, waits indefinitely, or performs long-running external work prevents cancellation from being observed.

**Risk:** the apparent cancellation contract does not bound execution time for untrusted or defective handlers. `RunAsync` can also remain stuck inside a synchronous instruction despite its asynchronous API.

**Fix:** state explicitly that cancellation is cooperative between instructions, or introduce an execution context carrying a token/deadline that handlers must observe. For hostile code, run handlers in an isolation boundary with enforceable timeout/termination rather than relying on tokens.

**Priority: P1 availability contract.**

### ✅ 21. Unbounded programs can run forever without an instruction budget

`Execute` stops only when the instruction pointer leaves the data range, becomes negative, or cancellation is requested. A valid jump loop can run indefinitely. The scheduler quantum limits one pass but `Run` and `RunAsync` immediately continue scheduling ready processes without any total-instruction or wall-clock budget.

**Risk:** a tiny valid bytecode program can consume CPU forever unless every caller supplies and triggers cancellation. This prevents using the VM as a bounded evaluator for untrusted programs.

**Fix:** add an optional total instruction budget/deadline to execution policy, count dispatched instructions centrally, and throw a dedicated quota exception when exhausted.

**Priority: P1 resource limits.**

### ✅ 22. Memory fault reporting can itself overflow for unsigned address types

When an access fails, `VirtualProcess` converts the generic address with `long.CreateChecked(currentAddress)` to build `MemoryAccessException`. For valid `ulong` addresses above `long.MaxValue`, or another wide unsigned implementation, this conversion throws `OverflowException` before the intended VM memory fault is created.

**Risk:** the exception type and diagnostic path depend on the address type/range. The original access violation is hidden by an error-reporting conversion.

**Fix:** store the address in a lossless generic/string/`BigInteger` representation, or make `MemoryAccessException` generic/accept an invariant textual address. Error construction must never require a narrowing conversion.

**Priority: P1 diagnostics.**

### ✅ 23. Cross-page reads partially modify the destination before failure

Like writes, `VirtualProcess.Read` copies one page at a time. If a later page is unmapped, earlier destination bytes have already been overwritten before `MemoryAccessException` is thrown.

**Risk:** callers may use partially updated buffers after catching the exception, especially when reading structured values that cross a page boundary. Read and write atomicity are both currently implicit.

**Fix:** validate the complete page range before copying, or expose/document partial-progress semantics explicitly. Keep read and write policies symmetric.

**Priority: P1 memory consistency.**

## Medium priority

### ✅ 24. `ReadOnlyMemory<byte>` does not provide a security boundary for physical pages

`VirtualPage.AsReadOnlyMemory` returns a view over the actual backing array. Although normal consumers cannot write through `ReadOnlyMemory<byte>`, advanced APIs such as `MemoryMarshal.TryGetArray` can recover the underlying `ArraySegment<byte>` and mutate the page without going through `VirtualProcess.Write` or access checks.

**Risk:** code holding a `VirtualPage` can bypass per-process access rights and mutate executable/data pages directly. The method is safe as an API convention, not as isolation.

**Fix:** document that in-process page permissions are cooperative only, or return a defensive copy for untrusted consumers. Do not present `ReadOnlyMemory` as an enforcement mechanism.

**Priority: P2 isolation contract.**

### ✅ 25. Freed virtual processes remain usable objects without a terminal ownership state

`FreeProcess` removes mappings and unregisters the process, but `VirtualProcess` itself has no disposed/freed state. Existing references can continue calling `Read`, `Write`, `IsAccessible`, and `GetAccess`; future internal/public mapping paths can accidentally revive the object unless every caller repeats ownership checks.

**Risk:** lifecycle validity is external to the process object and easy to bypass during future changes. Failures appear as ordinary unmapped-memory faults instead of use-after-free diagnostics.

**Fix:** mark a process as freed, reject every subsequent operation with `ObjectDisposedException` or a VM-specific lifecycle exception, and make ownership validation part of the process itself.

**Priority: P2 lifecycle clarity.**

### ✅ 26. Empty call-stack return and a legitimate `-1` return address are indistinguishable

`CallStack.Return` and `SimpleCallStack.Return` return `-1` when empty, while `Call(int)` accepts any integer including `-1`. Since `Context` also uses `-1` as the termination signal, an explicitly pushed negative return address is indistinguishable from stack underflow and may terminate execution silently.

**Risk:** malformed call bytecode can hide underflow or create ambiguous control flow instead of producing a deterministic stack fault.

**Fix:** reject negative return addresses, or replace the sentinel API with `bool TryReturn(out int address)` plus a throwing `Return` overload. Use a dedicated underflow exception rather than overloading a valid integer domain.

**Priority: P2 control-flow contract.**

### ✅ 27. Operand stacks have no depth limit or underflow abstraction

`DefaultContext.Stack` and `TypedStackContext<TValue>.Stack` expose raw `Stack<T>` instances. Push growth is unbounded and pop/peek underflow surfaces as generic `InvalidOperationException` from the collection.

**Risk:** bytecode can exhaust memory through repeated pushes, while stack errors lack instruction address, process identity, and VM-specific diagnostics.

**Fix:** wrap operand stacks behind a bounded VM stack abstraction that emits dedicated overflow/underflow faults and integrates with the shared VM limits policy proposed in the first pass.

**Priority: P2 resource limits and diagnostics.**

## Additional duplicated intent to reduce

- `NormalReader` and `InvertedReader` duplicate pointer advancement and bounds behavior. Centralize checked slicing/cursor advancement, then apply endianness conversion separately.
- Fixed-width and LEB128 decoding should share one transactional cursor abstraction rather than mutate `Context.InstructionPointer` independently.
- Cross-page read/write range discovery should be one validation routine followed by direction-specific copying.
- Call-stack and operand-stack underflow/overflow should use common VM fault types instead of collection exceptions and integer sentinels.

## Required tests

- Reject ULEB128/SLEB128 encodings longer than ten bytes and invalid tenth-byte payloads.
- Test canonical and non-canonical encodings at `ulong.MaxValue`, `long.MaxValue`, and `long.MinValue`.
- Verify every failed byte/fixed-width/LEB128 read leaves `InstructionPointer` at the documented position.
- Truncate the second or third operand of an instruction and verify transactional pointer behavior.
- Run a handler that blocks or loops and verify the documented cancellation limitation or enforceable timeout policy.
- Execute an intentional bytecode loop with a finite instruction budget and verify quota termination.
- Trigger memory faults with `ulong` addresses above `long.MaxValue` and preserve `MemoryAccessException`.
- Verify cross-page read failure either leaves the destination unchanged or reports partial progress by contract.
- Attempt backing-array extraction from `AsReadOnlyMemory` and document/test the selected isolation policy.
- Use a process after `FreeProcess` and verify a dedicated lifecycle failure.
- Push `-1` as a return address and exercise empty returns under the new non-ambiguous API.
- Enforce operand-stack depth and VM-specific underflow diagnostics.

## Deployment warning

Until items 17–23 are addressed, malformed operands can corrupt the instruction pointer, overlong LEB128 values can decode incorrectly, cancellation does not bound individual handler execution, and valid bytecode loops have no intrinsic execution quota. Virtual-memory reads also have partial-commit behavior, and memory-fault reporting is not reliable for the full unsigned address domain.
