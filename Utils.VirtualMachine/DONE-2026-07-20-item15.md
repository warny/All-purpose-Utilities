## What was changed

### New types (4 files)

- **`VmLimitKind.cs`** — Enum with 7 values identifying which structural limit was hit:
  `CallStackDepth`, `ControlFlowDepth`, `OperandStackDepth`, `InstructionCount`,
  `PhysicalPageCount`, `MemoryProcessCount`, `ScheduledProcessCount`.

- **`VmLimitExceededException.cs`** — New base exception (derives from `Exception`) carrying
  `LimitKind`, `Limit`, and `AttemptedValue`. Replaces `InvalidOperationException` for soft
  capacity overflows.

- **`VirtualMachineLimits.cs`** — Immutable sealed class encapsulating all structural capacity
  limits: `MaxCallStackDepth` (512), `MaxControlFlowDepth` (1024), `MaxOperandStackDepth` (1024),
  `MaxPhysicalPages` (int.MaxValue), `MaxMemoryProcesses` (int.MaxValue),
  `MaxScheduledProcesses` (int.MaxValue), `SchedulerQuantumSteps` (100). Validates all params ≥ 1.
  Exposes `VirtualMachineLimits.Default`.

- **`ExecutionLimits.cs`** — Immutable sealed class for instruction-count budgets.
  `MaxInstructions` is `null` (unlimited) or a positive `long`. Exposes `ExecutionLimits.Unlimited`.

## Design decisions

- **Soft vs hard limits:** `VmLimitExceededException` for configurable capacity overflows (the
  ones embeds applications care about). `InvalidOperationException` for hard identifier-counter
  exhaustion (int.MaxValue) which is an unreachable runtime condition, not a policy.
- **Exact-boundary rule:** budget check fires *before* dispatch, so a program that terminates on
  its Nth instruction (N == budget) completes normally. The check `dispatched >= maxInstructions`
  ensures this.
- **Backwards compatibility:** all existing scalar constructors and `Execute`/`Run`/`RunAsync`
  signatures are preserved unchanged. New overloads accepting `VirtualMachineLimits` or
  `ExecutionLimits` are additive.
- **Master process counts toward memory-process limit:** `CreateProcess` checks
  `_processes.Count >= _maxMemoryProcesses` where the master is already in `_processes`. With
  `maxMemoryProcesses=1`, only the master fits.
