## High priority

### ✅ 28. A throw inside a catch or finally can be caught again by the same exception block

`ControlFlowStack.Throw` finds the nearest `ExceptionBlock`, stores the thrown value, pushes that same block back onto the stack, and redirects to its `finally` or `catch`. The block remains on the stack while its handler executes. A new throw from inside that catch/finally therefore finds the same block again instead of propagating to the next outer handler.

**Risk:** handler code can loop back into its own catch/finally, repeatedly execute side effects, or recurse indefinitely. This differs from normal structured exception semantics, where an exception thrown by a handler is not caught by the same try statement.

**Fix:** track the phase of each exception block (`Try`, `Catch`, `Finally`, completed) and make handler lookup skip blocks whose protected try region is no longer active. Alternatively, remove the current block before entering its handler and carry handler state in a separate continuation record.

**Priority: P1 exception correctness.**

### 29. Control-flow targets are never validated against the instruction stream

Conditional, loop, exception, break, continue, catch, finally, and return addresses are stored as arbitrary integers. No helper verifies ordering, non-negativity, bytecode bounds, or that a target points to an instruction boundary. Assigning a negative or past-end address often makes the processor stop normally because the execution loop treats such values as termination.

**Risk:** malformed branch metadata can silently terminate instead of faulting, jump into operand bytes, or create invalid block ranges. Diagnostics cannot distinguish deliberate `Terminate()` from an accidental invalid target.

**Fix:** validate branch targets when blocks/frames are created and, ideally, validate the complete bytecode control-flow graph before execution. At runtime, use a dedicated checked jump API that distinguishes `Terminate` from invalid addresses and can enforce known instruction boundaries.

**Priority: P1 malformed-bytecode handling.**

### ✅ 30. `VirtualMemoryContext(VirtualPage, process)` does not validate page/process relationship or execute access

The page-based constructor accepts any `VirtualPage` and any `VirtualProcess<TAddress>`. It does not verify that both belong to the same `VirtualMemory`, that the page is mapped into the process, that the mapping is still active, or that the process has an execute-capable permission. `PageAccess` currently distinguishes only read-only and read-write, so executable memory has no explicit policy.

**Risk:** a process can execute bytes from a foreign, unmapped, or logically inaccessible page. Freeing/unmapping the process does not invalidate the already captured instruction-memory view.

**Fix:** make context creation an operation of `VirtualMemory`, validate ownership/mapping, and introduce an explicit execute permission or clearly document that code access is outside the page-permission model. Define whether unmapping/freeing invalidates an active context.

**Priority: P1 isolation model.**

### ✅ 31. Breakpoint redirection still invokes `BeforeInstruction` for the old instruction

`NotifyInspector` calls `OnBreakpoint` and then unconditionally calls `BeforeInstruction` using the original address and instruction name. If `OnBreakpoint` changes `InstructionPointer` to redirect or halt, `BeforeInstruction` receives a context pointing elsewhere while its metadata still identifies the old instruction. Only after both callbacks does dispatch notice the redirection.

**Risk:** debugger callbacks observe internally contradictory state and may trace, profile, mutate, or redirect twice based on stale instruction metadata.

**Fix:** after `OnBreakpoint`, immediately detect pointer changes and stop notification/dispatch for the old instruction, or pass an immutable inspection event containing both the original address and an explicit mutable action/result rather than using `Context.InstructionPointer` as control signaling.

**Priority: P1 debugger correctness.**

### 32. Inspector exceptions are indistinguishable from instruction execution faults and leave arbitrary context mutations

`OnBreakpoint` and `BeforeInstruction` run outside the instruction-handler exception translation and without a lifecycle policy. If an inspector throws after modifying the context, the exception escapes directly and the pointer/stack state remains whatever the inspector left behind.

**Risk:** attaching diagnostics changes processor fault behavior and may leave execution non-resumable. Scheduler processes can also remain `Running` through the first-pass scheduler defect.

**Fix:** define whether inspector failures are fatal debugger faults, isolated/logged errors, or ordinary execution faults. Snapshot/restore execution state when required and include callback phase/address in a dedicated exception type.

**Priority: P1 lifecycle and diagnostics.**

### ✅ 33. Physical pages cannot be freed or reclaimed

`VirtualMemory` supports allocating pages and freeing processes, but there is no page-release operation. Every allocated page remains in `_pages` and permanently mapped into the master process. Long-lived VMs that load/unload programs or allocate temporary memory can only grow.

**Risk:** unbounded retained memory and page identifiers, even after all non-master mappings are removed. Applications cannot implement deterministic memory reclamation.

**Fix:** add page lifecycle management with checks for active mappings, optional forced unmapping, master mapping removal, zeroing policy, and use-after-free protection. Integrate page quotas with the shared VM limits policy.

**Priority: P1 resource lifecycle.**

### ✅ 34. Shared physical pages have no synchronization or consistency contract

The same `VirtualPage` may be mapped into multiple processes. Reads and writes copy directly to the shared backing array with no synchronization, versioning, atomic primitive, or statement that VM execution must be single-threaded across those processes.

**Risk:** concurrent schedulers/processors can observe torn multi-byte values, race on code/data, and invalidate assumptions made by range prevalidation or transactional read/write fixes.

**Fix:** explicitly declare the entire memory model single-threaded, or provide synchronization/versioning at page or memory-manager level. Define atomicity guarantees for scalar and cross-page operations.

**Priority: P1 memory consistency.**

## Medium priority

### ✅ 35. Stored null locals are indistinguishable from missing or incompatible locals

`CallFrame.TryGetLocal<T>` succeeds only when `raw is T`. A stored null value never satisfies this pattern, even when `T` is a nullable or reference type. `GetLocal<T>` consequently throws the same `KeyNotFoundException` used for absence and type mismatch.

**Risk:** frames cannot faithfully store nullable locals, and diagnostics conflate three distinct states: missing name, present null, and wrong type.

**Fix:** first check key presence, then handle null compatibility separately. Consider distinct `ContainsLocal`, untyped retrieval, and typed conversion APIs, with dedicated exceptions for absence versus type mismatch.

**Priority: P2 frame contract.**

### ✅ 36. `Locals` is a live dictionary view rather than a stable diagnostic snapshot

`CallFrame.Locals` exposes the internal dictionary through `IReadOnlyDictionary`. The interface prevents direct mutation but does not make enumeration stable if the frame is modified concurrently or re-entrantly.

**Fix:** document live-view semantics or return an immutable snapshot for debugger/diagnostic use. Align this with the snapshot policy proposed for instructions, mappings, processes, and pages.

**Priority: P2 API clarity.**

### ✅ 37. Public execution APIs do not validate null contexts consistently

`Execute`, `ExecuteStep`, control-flow navigation methods, and inspector notification paths assume non-null contexts and fail through `NullReferenceException`. Other APIs in the module use explicit null guards.

**Risk:** inconsistent caller diagnostics and avoidable internal-looking failures at public boundaries.

**Fix:** add `ArgumentNullException.ThrowIfNull` consistently and centralize context/jump validation in shared helpers.

**Priority: P2 API quality.**

### ✅ 38. `VirtualProcessorException` exposes mutable opcode arrays

Constructors defensively copy opcode bytes, but `OpcodeBytes` returns the mutable `byte[]` directly. Consumers can alter exception diagnostics after construction, including data used by logging or later inspection.

**Fix:** expose `ReadOnlyMemory<byte>`, `ImmutableArray<byte>`, or a defensive copy. Use the same immutable opcode value type proposed for instruction registration.

**Priority: P2 diagnostic integrity.**

### ✅ 39. `[Serializable]` on `VirtualProcessorException` does not preserve custom diagnostic fields

The exception is marked `[Serializable]`, but no serialization constructor or custom serialization logic stores `InstructionPointer`, `OpcodeBytes`, and `InstructionName`. Depending on the serialization mechanism/runtime policy, the marker either suggests unsupported legacy serialization or loses the VM-specific fields.

**Fix:** remove the marker when binary exception serialization is not supported, or implement/document the required serialization contract. Prefer explicit diagnostic DTOs for cross-process reporting.

**Priority: P2 contract accuracy.**

### ✅ 40. Termination is encoded as a magic instruction-pointer value throughout the model

`Context.Terminate()` writes `-1`, call-stack underflow also returns `-1`, and arbitrary branch/return targets may set negative values. The processor interprets every negative pointer as successful termination.

**Risk:** underflow, invalid jumps, and explicit halt collapse into the same state and are difficult to diagnose or validate.

**Fix:** add an explicit execution state/termination reason separate from the numeric instruction pointer. Keep the pointer in a valid address domain and make invalid control transfers fault deterministically.

**Priority: P2 architecture.**

## Additional duplicated intent to reduce

- Branch, return, break, continue, catch, finally, inspector redirect, and termination all assign `InstructionPointer` directly. Route every transfer through one validated control-transfer API.
- Exception-phase state is spread across `ExceptionBlock` flags and `ControlFlowStack` methods. Replace it with an explicit state machine/continuation model.
- Context ownership, process ownership, mapping validation, and page lifetime should be established by `VirtualMemory` factories rather than independently trusted constructors.
- Debugger inspection should use immutable event snapshots rather than live collections plus direct pointer mutation.

## Required tests

- Throw from inside catch and finally blocks and verify propagation skips the current exception block.
- Test nested try/catch/finally with rethrow and a second exception raised by cleanup code.
- Reject negative, past-end, reversed, and mid-operand control-flow targets.
- Distinguish explicit termination from invalid jump and call-stack underflow.
- Reject `VirtualMemoryContext` creation with foreign, unmapped, read-incompatible, freed, or mismatched page/process pairs.
- Redirect from `OnBreakpoint` and verify `BeforeInstruction` is not called with stale metadata.
- Throw from each inspector callback and verify the documented state/fault policy.
- Allocate, map, unmap, free, and reuse pages under the selected lifecycle policy.
- Exercise concurrent access to a shared page and verify the documented synchronization/atomicity model.
- Store null locals for reference and nullable value types and distinguish them from missing/wrong-type locals.
- Verify exception opcode diagnostics cannot be mutated externally and survive the supported reporting/serialization path.

## Deployment warning

Until items 28–34 are addressed, structured exceptions can re-enter their own handlers, invalid branch targets may masquerade as normal termination, page-backed execution bypasses the virtual-memory mapping model, debugger callbacks can observe stale metadata, and physical memory cannot be reclaimed. Shared pages should be treated as single-threaded cooperative memory only.
