# Utils.VirtualMachine — Item 15 audit: coherent limits and fault policy (2026-07-20)

## Scope and baseline

This audit prepares item 15 from `TODO.md`: consolidate the virtual-machine resource limits and fault semantics without making the low-level components unusable on their own.

The analysis uses the intended post-round-3 state from PR #483 as the functional baseline. That branch already introduces or completes:

- bounded operand stacks;
- bounded control-flow stacks;
- processor and scheduler instruction budgets;
- deterministic identifier exhaustion checks;
- page reclamation;
- completion validation.

The remaining work is therefore not merely to add missing counters. It is to define ownership, defaults, propagation, exception semantics, and composition rules for those limits.

## Executive conclusion

Do **not** introduce one large `VirtualMachineLimits` object containing every operational concern, including cancellation and mutable run state.

Use three separate concepts:

1. **`VirtualMachineLimits`** — immutable structural capacity defaults shared while constructing contexts, stacks, memory managers, and schedulers.
2. **`ExecutionLimits`** — immutable limits for one `Execute`, `Run`, or `RunAsync` call, especially the instruction budget.
3. **Existing `CancellationToken` and scheduler lifecycle rules** — remain runtime inputs and execution behavior, not static limits.

A shared limits object should be optional. Every low-level class should retain its scalar constructor so it can still be used independently.

## Findings

### 15.1 — Defaults are inconsistent and cannot be inspected as one policy

Call depth defaults to 512 while control-flow and operand stacks default to 1024. Scheduler quantum defaults to 100, instruction execution is unlimited, and memory has no configurable count ceiling.

An embedding application cannot inspect one object and establish what an untrusted program may consume.

**Recommendation:** define named defaults once in `VirtualMachineLimits.Default`, then have convenience constructors delegate to those values. Existing public constants may remain as compatibility aliases during migration.

### 15.2 — Construction limits and per-run budgets have different lifetimes

Stack depths, maximum process counts, and maximum page counts are fixed for the lifetime of their owning object. An instruction budget belongs to one execution call and may differ between invocations.

Putting both directly into the same object encourages ambiguous behavior:

- Is `MaxInstructions` cumulative over the processor lifetime?
- Per scheduler lifetime?
- Per process?
- Per `Run` call?

**Recommendation:** keep instruction budgets in a separate `ExecutionLimits` value passed to `Execute`, `Run`, and `RunAsync`.

### 15.3 — Cancellation is not a limit

A `CancellationToken` is mutable runtime capability and carries registrations and external ownership. Storing it in an immutable shared policy would create lifetime leaks and surprising reuse between runs.

**Recommendation:** retain `CancellationToken` as a direct method argument. Do not add it to `VirtualMachineLimits` or `ExecutionLimits`.

### 15.4 — Context construction prevents clean policy propagation

`DefaultContext`, `TypedStackContext<TValue>`, and `CallStackContext` independently construct their stacks. `CallStackContext` can receive a custom `ICallStack`, but there is no simple constructor that applies one shared policy to both its operand stack and call stack.

The same issue will recur for any context combining call, operand, and control-flow stacks.

**Recommendation:** add policy-aware overloads while preserving explicit dependency injection:

```csharp
public DefaultContext(ReadOnlyMemory<byte> data, VirtualMachineLimits limits);
public TypedStackContext(ReadOnlyMemory<byte> data, VirtualMachineLimits limits);
public CallStackContext(ReadOnlyMemory<byte> data, VirtualMachineLimits limits);
```

The overloads should create their internal stacks from the relevant fields. Existing constructors taking scalar values or an `ICallStack` remain valid and authoritative.

### 15.5 — `VirtualMemory` needs both count and byte semantics clarified

A maximum page count is easy to enforce atomically before allocation. A maximum total byte count is derived from `MaxPages * PageSize`, but multiplication may overflow and page size is fixed per manager.

Providing both `MaxPages` and `MaxMemoryBytes` risks contradictory configurations and rounding questions.

**Recommendation for the first implementation:** expose `MaxPages` only. Document the effective byte ceiling as `MaxPages * PageSize`. Add a byte-based factory later only if a real caller requires one.

### 15.6 — Process-count ownership is ambiguous

There are two unrelated process registries:

- `VirtualMemory<TAddress>.Processes`, including the master process;
- `Scheduler<T>.Processes`, containing scheduled execution contexts.

A single `MaxProcesses` property would be ambiguous and could accidentally couple unrelated objects.

**Recommendation:** use distinct properties:

- `MaxMemoryProcesses` — explicitly state whether the master counts. Recommended: **yes**, because it occupies an identifier and list entry.
- `MaxScheduledProcesses`.

### 15.7 — Limits must be checked before mutation

Allocation and registration methods must fail without partial changes. Existing identifier checks follow this direction, but the shared policy needs an explicit rule.

Examples:

- page count checked before constructing, adding, or mapping the page;
- memory-process count checked before incrementing the id or adding the process;
- scheduled-process count checked before assigning the id or mutating the list;
- stack depth checked before push.

**Recommendation:** document precondition-first, no-partial-registration semantics for every capacity limit.

### 15.8 — Current exception types do not expose machine-readable limit data

Stack limits generally throw `InvalidOperationException` with component-specific text. Instruction budgets use a dedicated exception carrying the configured budget. Identifier exhaustion also uses `InvalidOperationException`.

Consumers cannot reliably distinguish:

- stack overflow;
- configured page limit;
- identifier-space exhaustion;
- malformed bytecode;
- memory access fault.

**Recommendation:** add a common base exception for configured resource exhaustion, without reclassifying semantic faults such as invalid memory access.

```csharp
public enum VmLimitKind
{
    CallStackDepth,
    ControlFlowDepth,
    OperandStackDepth,
    InstructionCount,
    PhysicalPageCount,
    MemoryProcessCount,
    ScheduledProcessCount
}

public class VmLimitExceededException : VirtualProcessorException
{
    public VmLimitKind LimitKind { get; }
    public long Limit { get; }
    public long AttemptedValue { get; }
}
```

`InstructionBudgetExceededException` may derive from this class for compatibility. Stack implementations can initially keep `InvalidOperationException` if changing their public exception contract is considered too disruptive, but the final target should be consistent.

Identifier-space exhaustion should remain a separate configuration/capacity failure: it is not the same as reaching a caller-configured limit.

### 15.9 — Fault behavior should not be generalized prematurely

`VirtualProcessor.Execute` propagates exceptions. `Scheduler` converts execution exceptions into `Faulted` process state. These are deliberately different APIs.

A global enum such as `FaultBehavior.Throw/Terminate/Ignore` would be dangerous:

- ignoring memory or malformed-bytecode faults can continue from corrupted state;
- converting every exception into termination loses diagnostics;
- low-level stack classes do not own process lifecycle.

**Recommendation:** item 15 should unify limit configuration and limit exception metadata, but preserve the existing ownership of fault transitions:

- processor methods throw;
- scheduler catches execution faults and marks the process `Faulted`;
- cancellation remains `OperationCanceledException` and must not fault a process unless explicitly documented by the scheduler API;
- budget exhaustion remains resumable and restores a running scheduled process to `Ready`.

A broader fault-policy abstraction should be a separate item backed by concrete use cases.

### 15.10 — Zero and unlimited semantics need one convention

Processor and scheduler budgets currently use `0` as unlimited and reject negative values inconsistently across revisions. Structural limits reject values below one.

**Recommendation:**

- structural capacities: strictly positive;
- per-run instruction budget: `null` means unlimited, or retain `0` for source compatibility;
- negative values: always `ArgumentOutOfRangeException`;
- never use `int.MaxValue` as a synonym for unlimited in public documentation.

A nullable budget is clearer in a new `ExecutionLimits` API:

```csharp
public sealed record ExecutionLimits
{
    public long? MaxInstructions { get; init; }
}
```

The existing `long maxInstructions = 0` overloads can delegate to it.

### 15.11 — Shared policy propagation must be observable

If a context or memory manager receives a `VirtualMachineLimits`, callers need to know which values were actually applied.

**Recommendation:** expose immutable effective limits:

```csharp
public VirtualMachineLimits Limits { get; }
```

on high-level owners such as `VirtualMemory` and optionally `Scheduler`. Low-level stacks already expose `MaxDepth`; that remains sufficient.

Do not retain a caller-mutable options object. `VirtualMachineLimits` must be immutable and validated at construction.

## Proposed API

```csharp
public sealed record VirtualMachineLimits
{
    public static VirtualMachineLimits Default { get; } = new();

    public int MaxCallStackDepth { get; init; } = 512;
    public int MaxControlFlowDepth { get; init; } = 1024;
    public int MaxOperandStackDepth { get; init; } = 1024;

    public int MaxPhysicalPages { get; init; } = int.MaxValue;
    public int MaxMemoryProcesses { get; init; } = int.MaxValue;
    public int MaxScheduledProcesses { get; init; } = int.MaxValue;

    public int SchedulerQuantumSteps { get; init; } = 100;
}

public sealed record ExecutionLimits
{
    public static ExecutionLimits Unlimited { get; } = new();

    public long? MaxInstructions { get; init; }
}
```

### Important design constraints

- The records must validate all values. Because `init` properties alone cannot validate the final combination reliably, prefer an explicit constructor or a validated factory/builder.
- `SchedulerQuantumSteps` could instead remain scheduler configuration. Including it is convenient for a complete host profile, but it is not a safety ceiling. If included, document this distinction.
- Page size should remain an explicit `VirtualMemory` constructor argument; it is layout configuration, not a global limit.
- Address type capacity and identifier exhaustion remain hard implementation ceilings independent of configured limits.

## Constructor integration plan

### Low-level components

Keep current scalar constructors:

```csharp
new CallStack(maxDepth);
new ControlFlowStack(maxDepth);
new BoundedStack<T>(maxDepth);
```

Optionally add convenience overloads only where they improve composition:

```csharp
new CallStack(limits);
new ControlFlowStack(limits);
```

Avoid making generic `BoundedStack<T>` depend on the complete VM policy; it is a reusable primitive and should continue to receive a scalar depth.

### Contexts

Add `VirtualMachineLimits` overloads that select the relevant scalar values. Preserve the constructors that accept explicit stack implementations.

### Scheduler

```csharp
public Scheduler(VirtualMachineLimits limits);
public void Run(ExecutionLimits limits, CancellationToken cancellationToken = default);
public Task RunAsync(ExecutionLimits limits, CancellationToken cancellationToken = default);
```

Keep existing overloads and delegate to the new methods.

### Processor

```csharp
public void Execute(
    T context,
    ExecutionLimits limits,
    CancellationToken cancellationToken = default);
```

Keep the existing overload for source compatibility.

### Virtual memory

```csharp
public VirtualMemory(int pageSize, VirtualMachineLimits limits);
```

Enforce `MaxPhysicalPages` and `MaxMemoryProcesses` before any mutation. Decide and document that the master process counts toward `MaxMemoryProcesses`.

## Recommended implementation phases

### Phase 1 — Value objects and validation

1. Add `VirtualMachineLimits` and `ExecutionLimits`.
2. Add validation tests for zero, negative, overflow-prone, and contradictory values.
3. Define default values from the current behavior to avoid silent compatibility changes.

### Phase 2 — Stack and context propagation

1. Add policy-aware context constructors.
2. Route call/control/operand depth values into existing scalar constructors.
3. Verify custom stack injection still overrides convenience policy construction.

### Phase 3 — Scheduler and processor execution budgets

1. Centralize duplicated budget validation and accounting.
2. Add `ExecutionLimits` overloads.
3. Keep exact-boundary behavior: a program terminating on its final permitted instruction succeeds; an additional required instruction throws.
4. Preserve scheduler resumability after budget exhaustion.

### Phase 4 — Memory and process capacities

1. Add maximum physical page count.
2. Add maximum memory process count, including the master.
3. Add maximum scheduled process count.
4. Verify all failures are atomic and do not consume identifiers.

### Phase 5 — Exception vocabulary

1. Add `VmLimitKind` and `VmLimitExceededException`.
2. Derive `InstructionBudgetExceededException` from it without removing its existing `Budget` property.
3. Migrate stack and configured capacity failures deliberately, documenting any exception-contract change.
4. Keep identifier exhaustion and invalid access separate.

## Required tests

### Limits validation

- Defaults reproduce the prior effective values.
- Every structural limit rejects zero and negative values.
- Execution budget rejects zero only if nullable semantics replace the compatibility overload; negative values always fail.
- The limits object cannot be mutated after construction.

### Propagation

- `DefaultContext(limits)` applies `MaxOperandStackDepth`.
- `TypedStackContext<T>(limits)` applies `MaxOperandStackDepth`.
- `CallStackContext(limits)` applies both operand and call depth.
- `ControlFlowStack` created from the policy applies `MaxControlFlowDepth`.
- Explicit injected stacks remain usable and are not silently replaced.

### Execution budget

- Processor execution stops exactly at the budget without overshoot.
- Scheduler budget is shared across all processes.
- A program terminating exactly on the final permitted instruction succeeds.
- A program requiring one more instruction throws.
- A scheduled process interrupted by budget exhaustion returns to `Ready` and can resume.
- Cancellation and budget exhaustion remain distinguishable.

### Memory and process capacities

- `MaxPhysicalPages = 1` permits one allocation and rejects the second.
- Rejected allocation does not consume a page id or virtual address.
- Freeing a page permits a replacement only if the policy is a simultaneous-capacity limit rather than a lifetime-allocation limit. Recommended semantics: simultaneous live count.
- `MaxMemoryProcesses` includes the master process.
- Rejected process creation does not consume an id.
- `MaxScheduledProcesses` is independent from `MaxMemoryProcesses`.

### Exception semantics

- Every configured limit reports its `VmLimitKind`, configured limit, and attempted value.
- Identifier exhaustion is not reported as a configured-limit exception.
- Memory access violations remain `MemoryAccessException`.
- Scheduler execution faults still become `Faulted`; budget exhaustion remains resumable.

## Compatibility risks

1. Changing stack overflow from `InvalidOperationException` to a dedicated type can break exception-specific callers.
2. Adding policy overloads can create ambiguous constructor resolution when `null` is passed.
3. Counting the master process must be explicit; otherwise `MaxMemoryProcesses = 1` is ambiguous.
4. A default finite page/process cap would be a behavior change. Use effectively unlimited defaults initially.
5. A shared object must not imply that changing one instance later updates already-constructed components.
6. `SchedulerQuantumSteps` is scheduling behavior, not an execution budget; naming and documentation must prevent confusion.

## Suggested first PR scope

Keep the first item-15 PR narrow and low-risk:

1. introduce validated immutable `VirtualMachineLimits` and `ExecutionLimits`;
2. add policy-aware context, scheduler, processor, and memory overloads;
3. preserve all existing defaults and old overloads;
4. enforce configurable live page/process counts;
5. centralize budget conversion and exact-boundary tests;
6. postpone exception hierarchy migration to a follow-up unless it can be done without changing existing catch behavior.

This delivers a coherent host-facing sandbox profile while avoiding a broad fault-policy redesign in the same change.
