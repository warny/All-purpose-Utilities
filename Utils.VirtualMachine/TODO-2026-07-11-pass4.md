# Utils.VirtualMachine — Fourth audit pass (2026-07-11)

> **Partially completed 2026-07-18/19.** Item 44 addressed in PR fix/utils-vm-quality-audit-p0-p1. Items 46, 47, 48 addressed in PR fix/utils-vm-quality-audit-round2. Items 42, 43 addressed in PR fix/utils-vm-quality-audit-round3. Items 41, 45, 49 remain open.

Follow-up review focused on public documentation, scheduler registration invariants, instruction-stream immutability, loop/exception semantics, asynchronous scheduling responsiveness, and configuration diagnostics. This file complements the previous audit files; no production code is changed.

## High priority

### 41. Catch/finally execution order is reversed from conventional structured exception semantics

When an exception block has both a catch and a finally target, `ControlFlowStack.Throw` redirects to `FinallyAddress` first and stores the catch address for `EndFinally`, which then jumps to the catch. The README explicitly documents this order.

Conventional `try/catch/finally` semantics execute the matching catch first and the finally block afterward, regardless of whether the catch completes normally or throws. The current model instead runs cleanup before recovery and does not provide a corresponding guaranteed finally-after-catch phase.

**Risk:** VM programs described as `try/catch/finally` behave differently from mainstream language semantics. Cleanup may run before the catch repairs state, and changes made by the catch are not covered by the finally block.

**Fix:** either implement conventional phase ordering with an explicit exception-block state machine, or rename/document the construct as a different two-stage handler model. Tests must cover normal try exit, catch completion, catch rethrow, finally-only, catch-only, and nested combinations.

**Priority: P1 control-flow correctness.**

### ✅ 42. The documented loop pattern can push one block per iteration

`LoopBlock` documentation describes the LOOP instruction as pushing a block and `ENDLOOP` as jumping back to `StartAddress`, described as the loop header/LOOP instruction. Re-executing that instruction naturally pushes another `LoopBlock` unless each concrete VM adds an undocumented guard.

**Risk:** ordinary loops can grow `ControlFlowStack` on every iteration, corrupt nesting, change BREAK/CONTINUE behavior, and eventually exhaust memory.

**Fix:** define `StartAddress` unambiguously as either the first body/condition instruction after the push, or add a dedicated iteration transition that reuses the existing block. Provide complete reference handlers and tests proving stack depth remains constant across iterations.

**Priority: P1 control-flow correctness and resource safety.**

### ✅ 43. `Context.Data` is not immutable despite its read-only contract

`Context` stores caller-provided `ReadOnlyMemory<byte>` without copying it. When that memory originated from a `byte[]`, the caller retains a writable alias and can modify opcode or operand bytes during execution. The documentation states that the instruction stream is read-only and that handlers cannot modify it, but this is only enforced through the property type.

**Risk:** externally mutated or concurrently changed bytecode makes dispatch nondeterministic, invalidates instruction-boundary validation/caching assumptions, and can turn inspected bytes into different executed bytes.

**Fix:** distinguish trusted borrowed memory from immutable program images. Copy bytecode by default, accept `ImmutableArray<byte>`, or explicitly expose a borrowed/mutable mode with a strict single-threaded contract and version checks.

**Priority: P1 execution integrity.**

### ✅ 44. The scheduler allows the same context to be registered multiple times

`AddProcess` accepts any context/processor pair and does not check whether the context is already registered. Two scheduled processes can therefore share the same `InstructionPointer`, data, operand stack, call stack, and control-flow state. During one `Step` pass, each wrapper may advance the same context independently and then maintain separate lifecycle states for that shared execution.

**Risk:** duplicated registration produces state corruption and contradictory process states: one wrapper may be `Terminated` while another remains `Ready`, both referring to the same context.

**Fix:** reject duplicate context registration by reference unless an explicit shared-context mode exists. Clarify whether processors may be shared separately from contexts and validate the chosen ownership model.

**Priority: P1 scheduler correctness.**

### 45. Scheduler completion depends only on instruction-pointer range, not VM execution state

After a quantum, `Scheduler.Step` marks a process terminated when `InstructionPointer < 0` or `>= Data.Length`. It does not verify that call frames, control-flow blocks, pending exception handlers, or operand invariants are complete.

**Risk:** truncated programs can be reported as successfully terminated while still inside a call, loop, conditional, try/finally, or with unresolved exception state. Structural bytecode errors become indistinguishable from normal completion.

**Fix:** allow contexts/processors to provide a completion validator or final state. For structured contexts, require empty/valid stacks and no pending exception phase unless abrupt termination was explicitly requested.

**Priority: P1 malformed-program detection.**

## Medium priority

### ✅ 46. `RunAsync` can monopolize the calling thread for an entire scheduling round

`RunAsync` awaits only after `Step`, while one `Step` executes up to `QuantumSteps` instructions for every ready process. With many processes, a large quantum, or slow handlers, the method may execute substantial synchronous work before yielding.

**Risk:** the API is described as non-blocking, but UI/server responsiveness can still be poor and cancellation is delayed until the full round completes.

**Fix:** document the cooperative granularity accurately, or add an asynchronous step mode that yields/checks cancellation between processes or after a configurable instruction count/time slice.

**Priority: P2 responsiveness contract.**

### ✅ 47. The README scheduler example does not compile and contradicts implementation semantics

The README uses `new Scheduler<DefaultContext>(quantum: 10)`, but the constructor parameter is named `quantumSteps`. It later references `ProcessState.Finished`, while the enum exposes `Terminated`. It also says each `Step()` advances only the highest-priority process, whereas the implementation and tests execute every ready process in descending priority order.

**Risk:** package consumers copy invalid code and form an incorrect scheduling model. Documentation examples are currently not protected by compilation tests.

**Fix:** correct the names and scheduling description, then compile README snippets in CI using documentation tests or extracted sample projects.

**Priority: P2 documentation correctness.**

### ✅ 48. `InstructionAttribute` accepts empty or whitespace diagnostic names

The attribute rejects only a null name. Empty/whitespace values are accepted and later appear in instruction enumeration, inspector callbacks, and exceptions. Runtime registration similarly converts a null name to an empty string.

**Risk:** debugging and fault reports can lose the only human-readable instruction identity, especially when multiple opcodes are involved.

**Fix:** require non-empty names or define a deterministic fallback derived from the method/opcode. Apply one validation policy to reflected and runtime registrations.

**Priority: P2 diagnostics.**

### 49. Breakpoints have no validation or lifecycle policy

`Breakpoints` is a public mutable `HashSet<int>`. It accepts negative, past-end, and mid-operand addresses, and remains unchanged when instruction registrations/program images change. A breakpoint is checked only when dispatch happens to start at exactly that address.

**Risk:** invalid breakpoints fail silently, stale breakpoints may target different instructions, and consumers cannot distinguish unresolved breakpoints from valid-but-not-yet-hit ones.

**Fix:** provide breakpoint registration against a validated program/instruction map, return resolution status, and define behavior when code changes. Keep a low-level raw-address API only when explicitly needed.

**Priority: P2 debugger API quality.**

## Additional duplicated intent to reduce

- Program ownership/immutability should be represented by a `ProgramImage` abstraction shared by contexts, instruction-boundary validation, inspectors, and breakpoints.
- Scheduler registration should centralize ownership checks for context, processor, name, and lifecycle instead of treating each wrapper as independent metadata.
- Structured completion validation should be shared by call-stack, control-flow, and exception contexts rather than inferred solely from the instruction pointer.
- README examples should be compiled from the same source used by tests to prevent API/documentation drift.

## Required tests

- Verify catch/finally ordering for normal completion, throw, catch return, catch rethrow, and nested handlers.
- Execute multiple loop iterations using the documented handler pattern and assert constant control-flow depth.
- Mutate the original byte array during execution and verify the selected immutable/borrowed-memory policy.
- Register the same context twice and verify deterministic rejection or documented shared-state behavior.
- Reach end-of-data with non-empty call/control-flow stacks and verify malformed completion is detected.
- Measure `RunAsync` cancellation/yield granularity across many processes and large quanta.
- Compile every README sample in CI.
- Reject empty instruction names or verify deterministic fallback names.
- Add invalid, unresolved, and stale breakpoints and verify explicit resolution diagnostics.

## Deployment warning

Until items 41–45 are addressed, the advertised structured-control-flow model can execute finally/catch in a surprising order, documented loops may accumulate blocks, bytecode can be mutated through retained aliases, the same execution context can be scheduled twice, and reaching the end of the byte array may be mistaken for successful structured completion.
