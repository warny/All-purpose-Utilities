# Utils.NumberToString — Fourth audit pass (2026-07-11)

Follow-up review focused on XML schema enforcement, deserialization diagnostics, static initialization resilience, form-variant expansion, culture-key normalization, configuration inheritance semantics, and generated scale agreement. This file complements the previous audit files; no production code is changed.

## High priority

### 86. The published XSD is not used to validate configuration XML

`ReadConfiguration` constructs an `XmlSerializer` with the expected namespace and deserializes directly from a `StringReader`. It never creates a schema-validating `XmlReader`, loads `NumberConvertionConfiguration.xsd`, or handles validation events.

The XSD therefore acts only as documentation/code-generation input. XML that violates declared restrictions may still deserialize, while unknown elements/attributes can be ignored by `XmlSerializer` depending on their position.

**Risk:** misspelled elements, missing required semantic structures, invalid ranges, and unsupported combinations can pass deserialization and fail much later with unrelated exceptions or silently use defaults.

**Fix:** validate XML against the packaged schema before deserialization, with secure reader settings and contextual line/position diagnostics. Keep semantic validation as a second phase because XSD alone cannot express all cross-reference and ambiguity rules.

**Priority: P1 configuration correctness.**

### 87. One invalid built-in configuration can make the entire type unusable

The static constructor initializes every embedded language configuration in one call. Any deserialization, inheritance, regex, scale, variant, or constructor failure escapes the static constructor and is wrapped in `TypeInitializationException` for every subsequent access to `NumberToStringConverter`.

**Risk:** one defective locale bricks all locales, including otherwise valid English fallback, for the lifetime of the process/AppDomain.

**Fix:** validate embedded resources in build/CI and load them into an isolated temporary registry before committing. At runtime, either fail fast with a complete aggregate diagnostic during an explicit initialization step or isolate optional locale failures while guaranteeing a known-safe core converter.

**Priority: P1 availability and deployment safety.**

### 88. `forms=` cardinal expansion silently truncates mismatched form counts

`ExpandFormVariants` splits `forms` and iterates only to `Math.Min(entries.Length, dimValues.Count)`. Extra forms are ignored; missing forms leave declared dimension values without a rule; empty entries are skipped silently.

**Risk:** a typo or reordered dimension list changes grammatical output without configuration failure. The XSD documentation presents positional forms as mapping to the dimension's ordered values, but the implementation accepts incomplete or surplus mappings.

**Fix:** require the number of entries to equal the number of declared dimension values, unless an explicit sparse syntax is introduced. Report the dimension, expected values, supplied forms, and XML location.

**Priority: P1 linguistic configuration correctness.**

### 89. Implicit ordinal fallback depends on traversal/declaration order

When an ordinal exception/rule omits its base `string`/`to` and supplies form variants, the loader captures the first expanded form as the no-variant fallback. “First” depends on XML traversal order and nested expansion order, not explicitly on each dimension's declared default value.

**Risk:** reordering equivalent XML nodes can change the default ordinal output, and the selected fallback may correspond to a non-default gender/case.

**Fix:** resolve fallback by matching the full declared default variant query. If no form matches defaults, require an explicit base form or reject the configuration.

**Priority: P1 deterministic linguistic behavior.**

### 90. Dimension aliases can collide during XML loading before contextual validation

The loader builds `nameNormalizer` with `ToDictionary` over canonical names and local aliases. Two dimensions sharing a canonical name/alias, or one alias equal to another canonical name case-insensitively, throw a generic duplicate-key exception before `ValidateVariantReferences` can provide a language-specific diagnostic.

**Risk:** configuration failure lacks the conflicting dimension declarations and can be difficult to trace in a large locale file.

**Fix:** validate canonical names and aliases explicitly before dictionary construction, reject empty/whitespace names and case-insensitive collisions, and include the culture/language path in diagnostics.

**Priority: P1 configuration diagnostics.**

### 91. Culture identifiers are accepted without canonical normalization

Culture strings from XML and public registration are used directly as dictionary keys with case-insensitive comparison, but are not trimmed, normalized through `CultureInfo`, or validated as BCP-47 identifiers. Values such as `"fr-FR "`, underscore variants, or malformed subtags can become registered yet unreachable through normal lookup/fallback.

**Risk:** apparently registered cultures silently fall back to another language, while duplicate logical cultures may exist under non-canonical spellings.

**Fix:** trim and canonicalize identifiers at parse/registration boundaries, preserve explicitly supported neutral/custom identifiers through a documented policy, and reject malformed or colliding canonical keys.

**Priority: P1 lookup correctness.**

### 92. `baseOn` cannot distinguish “inherit” from intentional empty overrides for many fields

`MergeLanguageType` treats null, default numeric values, and often empty collections as “inherit from base.” Examples include `GroupSize == 0`, empty/missing trigger collections, and nullable strings. This prevents a derived language from intentionally clearing inherited triggers, separators, suffixes, hooks, or collections when empty is meaningful.

**Risk:** derived configurations retain behavior they attempted to remove, and XML cannot express a full override/clear operation consistently.

**Fix:** use presence-aware configuration DTOs (`Specified` flags, nullable wrappers, or explicit merge modes) and define per-field replace/merge/clear semantics. Avoid using value defaults as presence indicators.

**Priority: P1 inheritance correctness.**

## Medium priority

### 93. Static scale names are sorted but not checked for contiguous scale indices

`ReadConverter` orders static scale entries by their numeric `value` and then discards the values, passing only the strings to `NumberScale`. Missing, duplicate, or non-zero-based indices therefore shift names to different scale positions instead of preserving/rejecting the declared mapping.

**Risk:** a missing scale entry can make every subsequent large number use the wrong scale name.

**Fix:** require unique contiguous indices starting at the documented base, or store static names in an indexed dictionary that preserves explicit positions and rejects holes according to policy.

**Priority: P2 scale correctness.**

### 94. XML replacements without `newValue` are silently omitted outside form variants

`ParseReplacements` filters out entries whose `NewValue` is null. This is valid when child form variants provide all replacements, but an accidentally omitted `newValue` with no valid child forms disappears without error.

The same filtering occurs for replacements nested in variant rules.

**Risk:** intended transformations simply do not run and no configuration diagnostic identifies the incomplete rule.

**Fix:** require either a base replacement value or at least one valid expanded variant form. Reject empty rules after expansion and report their `oldValue` and owning culture.

**Priority: P2 configuration diagnostics.**

### 95. The schema permits structurally weak digit and fraction lists that runtime assumes are complete

The XSD allows zero digit entries in `DigitListType` and does not constrain fraction digit counts to positive supported values. Runtime code later indexes digit dictionaries directly and uses fraction digit counts as denominator metadata.

**Risk:** schema-valid XML can still produce missing-key failures or nonsensical fraction definitions, demonstrating that current XSD guarantees are materially weaker than runtime preconditions.

**Fix:** strengthen the schema where practical, but primarily implement one semantic validator that checks required group coverage, unique digits, valid fraction keys, placeholders, scale tables, and all runtime indexing assumptions.

**Priority: P2 schema/runtime alignment.**

## Additional duplicated intent to reduce

- XML schema validation and semantic model validation should produce one aggregated configuration report rather than failures spread across deserialization, loader helpers, and converter constructors.
- Culture normalization should be shared by lookup, registration, inheritance references, cache keys, and date-culture resolution.
- Variant-form expansion for cardinal replacements, ordinal exceptions, and ordinal word rules should use one strict dimension-product mapper with explicit fallback selection.
- Inheritance should operate on presence-aware immutable nodes rather than manually applying different null/default/empty heuristics per field.
- Scale static-name parsing should preserve explicit indices instead of sorting and dropping them.

## Required tests

- Feed XML with unknown/misspelled elements, invalid XSD values, and missing required structures and require line-aware validation failures.
- Corrupt one embedded locale and verify initialization reports it without obscuring all other validation results.
- Test `forms` lists that are shorter, longer, or contain empty entries relative to dimension values.
- Reorder nested ordinal forms and verify fallback remains tied to declared default variants.
- Configure canonical/local dimension-name collisions and require contextual diagnostics.
- Register culture identifiers with spaces, casing differences, underscores, malformed tags, and canonical collisions.
- Derive a locale that intentionally clears inherited triggers, separators, variants, and optional sections.
- Supply sparse/duplicate/non-zero-based static scale indices and verify deterministic rejection.
- Create replacement nodes with neither `newValue` nor usable form variants.
- Validate empty/sparse digit lists and non-positive/duplicate fraction digit definitions.

## Deployment warning

Until items 86–92 are addressed, configuration XML is not actually schema-validated, one defective embedded locale can disable the entire converter type, positional variant forms can be silently truncated, ordinal defaults depend on declaration order, and derived configurations cannot reliably clear inherited behavior.
