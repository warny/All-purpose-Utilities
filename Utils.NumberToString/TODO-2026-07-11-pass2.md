# Utils.NumberToString — Second audit pass (2026-07-11)

Follow-up review focused on linguistic pipeline consistency, composite conversions, ordinal/multiplicative special cases, decimal precision contracts, temporal formatting, configurable separators, and variant-aware output. This file complements `TODO.md`; no production code is changed.

## High priority

### 62. ✅ Cardinal zero bypasses variants, triggers, adjustments, and language finalization

`Convert(BigInteger, variants)` returns `Zero` immediately when the input is zero, before building the variant query or running raw adjustment, variant rules, end triggers, and `LanguageSpecifics.FinalizeWriting`.

**Risk:** zero behaves differently from every non-zero cardinal. Configurations cannot apply gender/case variants, orthographic finalization, capitalization, or end-trigger transformations to zero.

**Fix:** treat zero as a raw cardinal token and pass it through the same post-processing pipeline as all other values. Keep only numeric decomposition bypassed.

**Priority: P1 linguistic correctness.**

### 63. Ordinal plugin and exception results bypass most of the ordinal pipeline

`ConvertOrdinal` returns immediately for language-specific plugin results and both variant/base ordinal exceptions. Those branches do not apply `OrdinalPrefix`, `_rawAdjustFunction`, end triggers, or `LanguageSpecifics.FinalizeWriting`, while generated ordinal forms do.

**Risk:** identical ordinal text can be capitalized, prefixed, normalized, or transformed differently solely because it came from an exception/plugin rather than the generic ordinal algorithm.

**Fix:** make every ordinal source produce a raw ordinal body, then run one shared final pipeline for adjustment, triggers, language finalization, prefix, and sign handling.

**Priority: P1 pipeline correctness.**

### 64. Named multiplicatives bypass variants and language finalization

When `Multiplicatives` contains the requested value, `ConvertMultiplicative` returns the configured string directly. Unnamed forms instead call `Convert(multiplier, variants)` and therefore use cardinal variants and finalization before appending the suffix.

**Risk:** named and generated multiplicatives follow different casing, orthography, variant, trigger, and adjustment rules. A language-specific finalizer cannot normalize named forms.

**Fix:** represent named multiplicatives as raw forms and run a dedicated shared multiplicative final pipeline. Define whether variants apply to the named form, the cardinal component, or both.

**Priority: P1 linguistic consistency.**

### 65. Composite conversions apply finalization inconsistently and may finalize subparts twice

Decimal and rational conversion call public `Convert(...)` for integer/numerator/denominator subparts, receiving already finalized strings. They then run `AdjustFunction` on the assembled result; `AdjustFunction` itself composes the raw adjust function with `LanguageSpecifics.FinalizeWriting`. Temporal, currency, multiplicative, and date conversions use different combinations and often do not finalize the complete assembled phrase at all.

**Risk:** non-idempotent language finalizers can transform subparts twice in decimals/fractions, while connectors, unit names, month names, and complete phrases remain unprocessed in other APIs. Output correctness depends on composition path.

**Fix:** separate raw cardinal generation from final phrase rendering. Composite APIs should build from raw typed fragments and invoke one finalizer exactly once on the complete phrase, with explicit stages for morphology and punctuation.

**Priority: P1 architecture and linguistic correctness.**

### 66. ✅ Decimal suffix pluralization narrows potentially large digit strings to `long`

For named decimal fractions, the code parses the fractional digits with `long.Parse(digits)` to choose the suffix plural form. A `decimal` can expose up to 28 fractional digits, so many valid fractional numerators exceed `long.MaxValue` and throw even though the value itself is valid.

The decimal separator pluralization also casts `integerPart` to `long`, failing for decimal integral parts outside the `Int64` range.

**Risk:** valid `decimal` values fail during grammatical agreement, independently of the cardinal converter's `BigInteger` support.

**Fix:** use `BigInteger`-aware plural/agreement APIs or reduce agreement input to the language-relevant residue through an explicit plural-rule abstraction. Never narrow the numeric value solely for morphology.

**Priority: P1 numeric and linguistic correctness.**

### 67. ✅ `mandatoryDecimalDigits` is not validated against the `decimal` precision domain

Any non-negative value is passed directly to `decimal.Round(number, mandatoryDecimalDigits, ...)`. .NET supports only 0 through 28 decimal places. Values above 28 fail with `ArgumentOutOfRangeException`, while very large requested padding would also create an unreasonable output contract.

**Fix:** validate the supported range at the public boundary and document whether the maximum is 28 or a lower configured limit. Use a separate arbitrary-precision decimal model if more digits are required.

**Priority: P1 API correctness.**

### 68. ✅ Hyphens are removed from decimal and fraction components regardless of language rules

Named decimal fractions and both fraction numerator/denominator paths call `.Replace("-", " ")` on already generated number text. This globally rewrites every hyphen, even when hyphenation is linguistically meaningful or configured as the normal separator.

**Risk:** valid orthography is silently changed for languages that require or permit hyphenated compounds. The behavior is hidden inside fraction composition and cannot be configured.

**Fix:** remove this text-level normalization. Phrase assembly should use structured fragments or a language-specific composition hook, not global punctuation replacement.

**Priority: P1 linguistic correctness.**

### 69. ✅ Configurable separator trimming removes characters, not separator tokens

`ConvertRaw` calls `TrimEnd` with the union of every character appearing in `GroupSeparator` and `Separator`. `TrimEnd` removes any run of those characters, not exact separator strings.

For a configurable word connector such as `"and"`, every trailing `a`, `n`, or `d` can be stripped from the generated number even when it belongs to the final word.

**Risk:** valid output can be truncated by configuration values that contain letters or meaningful punctuation.

**Fix:** append separators conditionally so no trailing separator is produced, or remove exact suffix tokens with ordinal string comparison. Never convert multi-character separators into a character trim set.

**Priority: P1 text correctness.**

### 70. ✅ Missing configured time units silently discard parts of a duration or time

`SupportsTimeConversion` is true when at least one unit is configured. `Convert(TimeSpan)` and `Convert(TimeOnly)` emit a component only if its corresponding key (`hour`, `minute`, `second`) exists. Non-zero components whose unit is absent are silently omitted; if all non-zero components are unsupported, the result becomes `Zero`.

**Risk:** a one-hour duration can be rendered as zero when only minute/second units are configured. Partial configurations produce numerically false text without diagnostics.

**Fix:** validate that the configuration contains every unit required by the public temporal APIs, or throw when a non-zero component has no representation. Alternatively expose capability flags per unit and narrower conversion methods.

**Priority: P1 semantic correctness.**

### 71. ✅ `DateFirstDay` overrides both ordinal and cardinal day placeholders

For day 1, `_dateFirstDay` replaces both `{ordinal-day}` and `{cardinal-day}`. A special form such as French `premier` is appropriate for an ordinal placeholder but not for a deliberately cardinal pattern.

**Risk:** a configuration requesting cardinal day text still receives an ordinal-specific lexical form.

**Fix:** apply `DateFirstDay` only to `{ordinal-day}`, or split the configuration into explicit `FirstOrdinalDay` and `FirstCardinalDay` forms.

**Priority: P1 linguistic correctness.**

## Medium priority

### 72. Temporal conversion silently drops sub-second precision

`Convert(TimeSpan)` and `Convert(TimeOnly)` process only hours, minutes, and seconds. Milliseconds and ticks are ignored, including cases where the entire value is sub-second and therefore rendered as zero.

**Fix:** document truncation explicitly, offer a rounding policy, or support configurable millisecond/sub-second units. A non-zero value should not silently become zero without a declared precision contract.

**Priority: P2 API semantics.**

### 73. Date/time pattern replacement is sequential and not token-safe

`Convert(DateOnly)` performs chained string replacements for `{month}`, `{ordinal-day}`, `{cardinal-day}`, and `{year}`. Text inserted by an earlier replacement can itself contain a later token and be rewritten unintentionally.

**Fix:** parse the date pattern into immutable literal/token segments once during configuration validation and render tokens without rescanning inserted values. Reject unknown or malformed placeholders.

**Priority: P2 robustness.**

### 74. Composite unit and connector text does not participate in variants

Currency unit names, temporal unit names, fraction connectors, date-time connectors, and month names are concatenated after cardinal conversion. Variant rules operate primarily on the number fragments and cannot reliably express agreement or case changes for the surrounding noun/connector phrase.

**Risk:** languages with case, gender, or plural agreement beyond the numeral cannot produce grammatically complete composite phrases through the current generic variant model.

**Fix:** introduce typed phrase components and language-specific composition hooks, or allow variant-conditioned forms for units/connectors and run them through one final phrase pipeline.

**Priority: P2 extensibility.**

## Additional duplicated intent to reduce

- Zero, ordinal exceptions, ordinal plugins, named multiplicatives, and normal generated forms should converge into one result-finalization pipeline per conversion kind.
- Decimal, rational, currency, time, and date conversion should compose raw fragments rather than finalized strings.
- Plural/agreement selection should accept arbitrary-precision numeric context through one language-aware abstraction instead of repeated `long` casts.
- Separator and punctuation handling should use structured tokens rather than `Replace`, `TrimEnd(char[])`, and string concatenation.
- Temporal capability validation should be centralized during converter construction rather than silently checked component by component at runtime.

## Required tests

- Verify ordinal plugin, ordinal exception, and generated ordinal paths receive identical prefix/finalization stages.
- Verify named and generated multiplicatives use the same final phrase policy.
- Use a deliberately non-idempotent finalizer and assert every composite conversion invokes it exactly once.
- Convert decimals with integral/fractional magnitudes above `long.MaxValue` while named suffix agreement is enabled.
- Reject `mandatoryDecimalDigits` outside the supported range.
- Preserve meaningful hyphens in decimal and general fraction output.
- Configure alphabetic separators and verify final words are never character-trimmed.
- Convert durations/times with missing hour/minute/second definitions and require explicit failure or documented capability behavior.
- Verify `{cardinal-day}` and `{ordinal-day}` differ correctly for day 1.
- Test sub-second-only durations under the selected truncation/rounding policy.
- Render patterns whose inserted values contain brace-like/token text without secondary replacement.

## Deployment warning

Until items 62–71 are addressed, outputs can differ by internal conversion path: zero and special forms may skip finalization, decimal/fraction phrases may be finalized twice, large valid decimals can overflow during plural selection, configured separators can truncate words, and incomplete time-unit configurations can silently produce numerically false text.
