# Utils.NumberToString — Third audit pass (2026-07-11)

Follow-up review focused on effective immutability, arbitrary-precision rule filters, scale-name generation, public group APIs, maximum-value semantics, culture-sensitive casing, and validation of programmatic trigger/replacement models. This file complements the previous audit files; no production code is changed.

## High priority

### 75. Converter configuration is only shallowly immutable

The converter copies top-level rule collections with `ToImmutableArray`/`ToImmutableDictionary`, but several contained model objects retain caller-owned mutable collections:

- `VariantDimension.Values` stores the supplied `IReadOnlyList<string>` directly;
- `VariantRule.Constraints` and `Replacements` retain the supplied dictionary/list;
- `OrdinalVariantRule` retains all supplied dictionaries;
- `TriggerRule.GroupIndices` retains the supplied array and `Replaces` list;
- `TriggerReplace.Forms` retains supplied constraint dictionaries and lists.

The constructor then builds `_dimensionIndex`, `_sortedVariantRules`, and `_hasScaleSpecificVariantRules` from the initial state. Later mutation of nested collections changes runtime matching without rebuilding those indexes.

**Risk:** output can change after converter construction, while cached/sorted metadata remains stale and contradictory. Shared converters can become nondeterministic under concurrent mutation.

**Fix:** deep-copy every nested collection into immutable structures when model objects are constructed or when the converter snapshots its options. Expose immutable value objects rather than `IReadOnly*` wrappers over caller-owned collections.

**Priority: P1 determinism and thread safety.**

### 76. `onValue` rules silently stop working above `long.MaxValue`

Cardinal conversion supports `BigInteger`, but global variant/replacement filtering passes the numeric value only when the absolute value fits in `long`; otherwise it passes `null`. `ApplyVariantRules` treats a null numeric value as non-matching for `OnValue`, and global value-filtered replacements are skipped entirely.

**Risk:** the same rule works for smaller values and silently disappears for larger supported cardinals. Configuration authors cannot express arbitrary-precision conditions even though the main API advertises arbitrary-precision conversion.

**Fix:** use `IntRange<BigInteger>` or a numeric predicate abstraction throughout rule evaluation. If filters are intentionally limited to `long`, reject/configure a maximum compatible range instead of silently suppressing rules.

**Priority: P1 arbitrary-precision correctness.**

### 77. ✅ `NumberScale` does not validate the collections required by dynamic generation

`GetScaleName` performs `int.DivRem(scale, ScaleSuffixes.Count)` and indexes several prefix lists by decimal digits. The constructor allows:

- an empty `ScaleSuffixes` list, causing division by zero once static names are exhausted;
- prefix lists shorter than ten entries, causing index failures;
- null/empty elements that later produce empty or malformed scale names;
- arbitrary `StartIndex` values that may generate negative intermediate scale indexes.

**Risk:** valid large cardinal conversion fails only when a particular scale boundary is reached, far from configuration construction.

**Fix:** validate scale configuration eagerly: at least one suffix when dynamic generation is possible, exactly ten entries for each digit-prefix table, valid non-null strings, and a non-negative/index-safe start policy.

**Priority: P1 configuration safety.**

### 78. ✅ `MaxNumber` does not consistently constrain non-integer public APIs

`MaxNumber` is enforced in `Convert(BigInteger)`, but decimal conversion decomposes the input and validates only the integral component through the cardinal call. For example, with `MaxNumber = 1`, a value such as `1.9m` can still be converted. Currency, fractions, years, time values, and multiplicatives apply different or no maximum semantics.

A negative configured `MaxNumber` is also accepted, making every non-zero cardinal fail while zero still succeeds.

**Risk:** the documented “maximum supported number” has conversion-kind-dependent meaning and can be bypassed by fractional values.

**Fix:** define whether the limit applies to absolute input magnitude, integral components, or cardinal-only conversion. Validate `MaxNumber >= 0` and enforce the selected policy consistently at every relevant public API boundary.

**Priority: P1 API contract.**

### 79. ✅ Public `ConvertGroup` uses floating-point exponentiation for integer decomposition

`ConvertGroup` computes `10^(groupNumber - 1)` with `Math.Pow` and casts the result to `long`. The method is public and accepts arbitrary `groupNumber`; sufficiently large values produce an imprecise or overflowing conversion, while negative values lead to invalid fractions/casts and later dictionary failures.

**Risk:** a public integer conversion helper has undefined behavior outside the small group sizes used by bundled configurations, with exceptions unrelated to the actual precondition.

**Fix:** validate `groupNumber` against configured group keys and calculate powers exactly with checked integer/`BigInteger` arithmetic. Consider making this method private/internal unless direct group conversion is a supported public feature.

**Priority: P1 numeric correctness.**

### 80. ✅ Programmatic trigger models accept invalid and null internal state

`TriggerReplace` does not validate `from`, `forms`, form constraints, or replacement values before compiling a regex or storing references. `TriggerRule` likewise accepts a null replacement collection and arbitrary group indices. Empty literal `from` values can have surprising replacement semantics; invalid regexes throw low-level construction exceptions without language/rule context.

**Risk:** malformed programmatic options fail late during conversion or with diagnostics that do not identify the owning language and trigger path.

**Fix:** validate and deep-copy trigger models at construction. Require non-empty source patterns, non-null forms/replacements, non-negative group indexes, valid variant references, and wrap regex compilation failures with configuration context.

**Priority: P1 configuration robustness.**

### 81. ✅ Duplicate exact replacement keys fail through a generic dictionary-construction exception

The constructor builds `_replacementLookup` with `ToImmutableDictionary(r => r.OldValue, ...)` for global unfiltered replacements. Two exact rules with the same `OldValue` cause an `ArgumentException` from dictionary construction, without identifying the conflicting rules or offering an explicit precedence policy.

**Risk:** a configuration feature that otherwise relies heavily on ordered transformations has an undocumented duplicate behavior and poor diagnostics.

**Fix:** detect duplicate exact keys during validation and either reject them with a precise language/rule diagnostic or define deterministic first/last/priority semantics.

**Priority: P1 configuration determinism.**

## Medium priority

### 82. ✅ Scale-name capitalization depends on the process current culture

`NumberScale.GetScaleName` uses `char.ToUpper(value[0])` without specifying a culture. Generated scale names can therefore differ according to `CurrentCulture`, notably for casing-sensitive alphabets such as Turkish.

**Risk:** the same converter and number can produce different output on different threads or hosts.

**Fix:** use `ToUpperInvariant` when configuration strings are culture-neutral, or pass the converter's explicit culture/casing strategy into `NumberScale`.

**Priority: P2 globalization determinism.**

### 83. ✅ `GetScaleName` has no public argument validation

The method is public but accepts negative scale values. A negative value satisfies `scale < StaticValues.Count` and is then used as a negative list index. Very large values can also overflow when adding `StartIndex` or calculating prefix indexes.

**Fix:** reject negative values explicitly, use checked arithmetic, and document the maximum supported/generated scale range.

**Priority: P2 API quality.**

### 84. Replacement execution is cascading but the contract does not clearly define fixed-point versus single-pass behavior

Global substring replacements run sequentially over the text, so an earlier replacement can create input for a later rule. Exact replacement returns immediately and prevents any subsequent substring rules. Trigger and variant replacements have still different sequencing behavior.

**Risk:** equivalent-looking rules produce different results depending on their category and declaration placement. Reordering configuration can alter output through accidental cascades.

**Fix:** document a formal transformation pipeline and whether each stage is single-pass, cascading, or non-recursive. Detect cycles/ambiguous overlaps where appropriate and provide explicit priorities.

**Priority: P2 maintainability and determinism.**

### 85. Culture lookup and month rendering use different fallback models

Converter lookup recursively strips BCP-47 subtags and ultimately falls back to English. Month rendering later calls `CultureInfo.GetCultureInfo(LanguageIdentifier)` and, on failure, silently emits the numeric month. A custom converter can therefore produce words in one language but a numeric month without any capability warning.

**Risk:** date output silently becomes mixed-format and lookup behavior differs between number words and month names.

**Fix:** resolve and validate a concrete date culture during converter construction, or require explicit configured month names. Expose date support only when both the pattern and month-name source are valid.

**Priority: P2 date correctness.**

## Additional duplicated intent to reduce

- Every public configuration model should share one immutable validated representation rather than copying some levels while retaining mutable nested objects.
- Cardinal numeric predicates, replacement `onValue`, variant `onValue`, plural selection, and maximum checks should share one arbitrary-precision numeric context.
- Scale construction and public scale lookup should use one validated exact-integer generation component.
- Trigger, replacement, variant, and ordinal rule validation should report one consistent configuration path and collision policy.
- Culture fallback for converter lookup, finalization, casing, and month names should be resolved once into an explicit locale context.

## Required tests

- Mutate every source list/dictionary/array after converter construction and verify output remains unchanged.
- Run concurrent conversions while mutating retained option models and verify mutation is impossible or isolated.
- Apply `onValue` rules below, at, and above `long.MaxValue` using `BigInteger` cardinals.
- Reject empty scale suffixes, short prefix tables, null entries, invalid start indexes, and negative scale requests during validation.
- Define and test `MaxNumber` against decimals, currency, fractions, years, multiplicatives, and negative configured limits.
- Call `ConvertGroup` with zero, negative, missing, and oversized group numbers.
- Reject null/empty trigger sources, null form collections, invalid regexes, invalid group indexes, and unknown variant constraints with contextual diagnostics.
- Configure duplicate exact replacements and verify the documented collision policy.
- Generate capitalized scale names under Turkish and invariant current cultures and assert stable output.
- Test replacement cascades and exact-plus-substring combinations against the documented stage policy.
- Build a custom language identifier unsupported by `CultureInfo` and verify date capability fails explicitly or uses configured month names.

## Deployment warning

Until items 75–81 are addressed, converter instances are not reliably immutable, large `BigInteger` values silently bypass configured value rules, malformed scale tables fail only at high magnitudes, fractional APIs can evade `MaxNumber`, and programmatic trigger/replacement errors surface late with weak diagnostics.
